# Internationalisierung (i18n)

Shifty unterstützt drei Sprachen: **Englisch (En), Deutsch (De),
Tschechisch (Cs)**.

## Wo Übersetzungen leben

Im Frontend unter `shifty-dioxus/src/i18n/` — vermutlich ein
Loader-basiertes System mit einem Key-Value-Store pro Sprache.
**[Zu prüfen]** exakte Struktur.

Das Backend selbst gibt keine übersetzten Strings zurück — es liefert
Enum-Werte oder Codes, und das Frontend mappt sie zur User-Sprache.
Ausnahme: PDF-Templates und E-Mail-Templates (Text Templates, F10) —
die können mehrsprachig sein und werden im Backend gerendert.

## Regel: Drei Locales, keine Ausnahme

**Konvention (CLAUDE.md):** Jeder neue Text-Key muss in **allen drei
Sprachen** eingetragen werden — nicht nur En.

Wenn du das vergisst:

- Fallback zeigt entweder den Key-Namen oder einen Leerstring
  (**[Zu prüfen]** genaues Verhalten).
- Ein Cs-User sieht deutschen oder englischen Text — schlechter Style,
  wenn die anderen Strings der Seite in Cs sind.

## Muster für neuen Text

1. Neuen Key in `i18n/en/…` einführen.
2. Denselben Key mit deutscher Übersetzung in `i18n/de/…`.
3. Denselben Key mit tschechischer Übersetzung in `i18n/cs/…` — im
   Zweifel Machine-Translate mit Kennzeichnung, damit ein Native-Speaker
   drüber gehen kann.
4. Component nutzt einen Lookup wie `t("my.key")`.

## Randfälle

### Satzstruktur

Deutsch und Tschechisch haben andere Satzstellung als Englisch. Wenn
du im Frontend Satz-Fragmente konkatenierst (`t("foo") + " " + t("bar")`),
brichst du Deutsch und Tschechisch.

**Regel:** Immer Full-Sentence-Templates mit Placeholders benutzen:

```
"welcome_message": "Willkommen, {name}, du hast {count} neue Nachrichten"
```

Nie `"welcome" + user + "you_have" + count + "messages"`.

### Plural-Formen

- Englisch: 2 Formen (1, ≠1).
- Deutsch: 2 Formen (1, ≠1).
- Tschechisch: 3 Formen (1, 2-4, 5+ oder 0).

**[Zu prüfen]** ob das i18n-Framework Plural-Rules unterstützt (ICU
MessageFormat oder ähnlich). Wenn nicht, sind Pluralisierungen manuell
zu bauen — im Zweifel per Enum-Key mit expliziten Formen.

### Zahlen- und Datumsformate

Locale-abhängig:

- Deutsch: `1.234,56` (Dezimal-Komma).
- Englisch: `1,234.56` (Dezimal-Punkt).
- Tschechisch: `1 234,56` (schmaler Space als Tausender-Separator,
  Dezimal-Komma).

**[Zu prüfen]** Wie Shifty das aktuell handhabt — Frontend-Formatter
oder Backend-Serialisierung.

### Feature-spezifische i18n-Keys

Manche Features (v.a. Reporting, Absence-Kategorien) haben eine große
Zahl von Enum-Werten, die alle übersetzt werden müssen. Die Feature-Dokus
listen die relevanten Key-Präfixe.
