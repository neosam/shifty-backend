# Feature: Employee Management

> **Short form:** Creation and maintenance of employee master data (Sales
> Persons), their employment contracts (Employee Work Details), their
> recurring availability blocks (Unavailable), and their assignment to
> shift plans.

**Cluster ID:** F01
**Status:** production
**First introduced:** v0.1 (base tables 2024-05), successively extended up to v2.4
**Responsible crates:**
`service::{sales_person, sales_person_shiftplan, sales_person_unavailable, employee_work_details}`,
`service_impl::{sales_person, sales_person_shiftplan, sales_person_unavailable, employee_work_details}`,
`dao::{sales_person, sales_person_shiftplan, sales_person_unavailable, employee_work_details}`,
`dao_impl_sqlite::{sales_person, sales_person_shiftplan, sales_person_unavailable, employee_work_details}`,

## 1. What is it? (Business context)

Employee Management is the foundation of the entire Shifty data model:
nearly every other domain (Bookings, Absences, Extra Hours, Reporting,
Billing Periods) hangs off a `sales_person.id`. This cluster holds four
tightly interlocked aggregates:

1. **Sales Person** — the employee's master data (name, identifying
   color, `is_paid`, `inactive`, optional link to a user login).
2. **Employee Work Details** (the historically grown name for the
   **employment contract**) — weekly hours, working days, vacation
   entitlement, dynamic/cap flags, and a fixed "committed voluntary" per
   week, valid for a time range `[from_year/kw/dow, to_year/kw/dow]`.
3. **Sales Person Unavailable** — one entry per (Sales Person, year,
   calendar week, day of week) on which the employee does **not** want
   to be scheduled (recurring availability block at day-of-week
   granularity — not the Absence system for whole Vacation/sick
   periods, which lives in F02).
4. **Sales Person Shiftplan** — which shift plans the employee is allowed
   to be booked into, with permission level (`available` or
   `planner_only`).

From the user's perspective (see `docs/employee-management_de.md` and
`docs/employee-management.md` for the end-user manual with screenshots),
the onboarding flow is:

1. **Create sales person** via *User Management → Sales Persons → Create
   New Sales Person* (`shifty-dioxus/src/page/sales_person_details.rs`).
   The HR user enters the name, identifying color, `is_paid`/`inactive`,
   optionally the linked login user, and the shift-plan assignments.
2. **Set up an employment contract** via the employee detail page
   (`shifty-dioxus/src/page/employee_details.rs`) in the *Contracts*
   area → `ContractModal`. Without a contract the employee has no
   target hours and the balance remains empty.
3. **Optionally enter Unavailable weekdays** — the sales user does this
   themselves via the weekly view in the shift planner.
4. **Optionally enter Extra Hours** manually (Cluster F02 — Extra Hours /
   Absences, not here).

Deactivation ("This person is inactive") is the preferred alternative
to actual deletion; deletion is exposed as a soft-delete by the backend
service, but is deliberately not offered in the UI in order to preserve
the balance history and FK consistency with Bookings/Extra Hours.

## 2. Business rules

### 2.1 Sales Person

- **Visibility of `is_paid` is privacy-sensitive.** Non-HR users see
  `is_paid = None` (`SalesPersonServiceImpl::get_all`,
  `service_impl/src/sales_person.rs:62-70`). In `get()` the field is
  only returned if the calling user is HR **or** the caller is the
  linked user of the Sales Person record itself
  (`sales_person.rs:120-143`).
- **Delete = soft delete.** `delete()` sets `deleted` to the current
  timestamp and rotates `version` (`sales_person.rs:253-279`). The
  `deleted` field is **not modifiable via update()**
  (`ValidationFailureItem::ModificationNotAllowed("deleted")`, `sales_person.rs:228-235`).
- **Optimistic locking via `version`.** Update requires that the
  submitted `version` matches the persisted value; otherwise
  `ServiceError::EntityConflicts` (`sales_person.rs:220-226`).
- **Create guards.** `id != Uuid::nil()` → `IdSetOnCreate`; `version !=
  Uuid::nil()` → `VersionSetOnCreate` (`sales_person.rs:177-183`). Both
  `id` and `version` are generated by the service via `UuidService`.
- **User assignment is 1:1.** `sales_person_user` has
  `UNIQUE(sales_person_id)` **and** `UNIQUE(user_id)`
  (`migrations/sqlite/20240506114107_add_sales_person.sql:20-21`).
  Replacement runs in a TX via `discard_assigned_user` + `assign_to_user`
  (`sales_person.rs:316-342`), so an assignment move is atomic.

### 2.2 Employee Work Details (employment contract)

- **Time range in calendar-week granularity** via
  `(from_year, from_calendar_week, from_day_of_week)` up to
  `(to_year, to_calendar_week, to_day_of_week)`. The "active" contract
  for a week is determined by a tuple comparison:
  `(from_year, from_calendar_week) <= (year, calendar_week) <= (to_year, to_calendar_week)`
  (`service_impl/src/employee_work_details.rs:114-118`). **[To verify]**
  whether `from_day_of_week` / `to_day_of_week` are deliberately
  ignored in this comparison — the current implementation only relies
  on calendar-week granularity.
- **Weekly hours distribution.** `expected_hours` is the weekly total;
  `workdays_per_week` is the divisor for `hours_per_day()`; the checked
  weekday flags (`monday`..`sunday`) define which days are candidates at
  all (`potential_weekday_list`,
  `service/src/employee_work_details.rs:78-102`).
- **Holiday hours.** `holiday_hours = expected_hours / potential_days_per_week`
  — that is, a holiday falling on a potential working day relieves the
  employee by this share (`employee_work_details.rs:112-114`).
- **`is_dynamic` (v1.5)** — instead of fixed weekly hours, the actually
  worked hours count as the agreed hours (target = actual). Migration
  `20251029192107_add-column-is-dynamic-to-employee-work-details.sql`.
- **`cap_planned_hours_to_expected` (v2.x)** — caps the balance-relevant
  hours per week to `expected_hours`; anything above that counts as
  volunteer work (see `docs/employee-management_de.md` §6). Migration
  `20260426120000_add-cap-flag-to-employee-work-details.sql`.
- **`committed_voluntary` (v2.4)** — weekly firmly committed voluntary
  hours in addition to the paid target; takes effect in the annual
  overview and capacity planning. Migration
  `20260623120000_add-committed-voluntary-to-employee-work-details.sql`.
- **Vacation proration at contract edges (D-28-04, Phase 28).**
  `vacation_days_for_year(year)` prorates at year start and year end:
  start → `vac * (ordinal(from) - 1) / days_in_year(year)` (strictly
  before the start; a Jan 1 start subtracts 0),
  end → `vac * (1 - ordinal(to) / days_in_year(year))`
  (`employee_work_details.rs:158-194`). Regression suite starting at
  `employee_work_details.rs:286-408`.
- **Update is partial and does NOT change `from_*`.** `update()` only
  writes `to_*`, `expected_hours`, `vacation_days`, `workdays_per_week`,
  `is_dynamic`, `cap_planned_hours_to_expected`, `committed_voluntary`
  and `version`
  (`service_impl/src/employee_work_details.rs:240-253`). Weekday flags
  and the start point remain as they were on create — changing these
  fields requires a new contract.
- **Create guards.** `id != nil` → `IdSetOnCreate`; `version != nil` →
  `VersionSetOnCreate` (`employee_work_details.rs:197-208`). `created`
  is set by the service.
- **Delete = soft delete.** `delete()` sets `deleted` and keeps the
  record for history (`employee_work_details.rs:262-288`).

### 2.3 Sales Person Unavailable

- **One record per weekday per CW.** `create()` checks whether an
  undeleted row already exists for
  `(sales_person_id, year, calendar_week, day_of_week)` and otherwise
  rejects with `ServiceError::EntityAlreadyExists`
  (`service_impl/src/sales_person_unavailable.rs:155-168`).
- **Create guards.** `id`, `version`, `deleted` and `created` must be
  empty, otherwise `IdSetOnCreate` / `VersionSetOnCreate` /
  `DeletedSetOnCreate` / `CreatedSetOnCreate`
  (`sales_person_unavailable.rs:143-154`).
- **Delete = soft delete.** Sets `deleted` and rotates `version`
  (`sales_person_unavailable.rs:190-230`).
- **Auth gate.** All read/write ops accept **either** a Shiftplanner
  **or** the sales user themselves (via
  `SalesPersonService::verify_user_is_sales_person`,
  `sales_person_unavailable.rs:41-50`, `73-82`, `132-141`, `203-212`). The
  week-wide bulk query `get_by_week()` is **only** available to
  Shiftplanners (`sales_person_unavailable.rs:109-111`).

### 2.4 Sales Person Shiftplan

- **Permission level per assignment.** `permission_level` ∈
  {`available`, `planner_only`} — CHECK constraint in the migration
  `20260402000000_add-permission-level-to-sales-person-shiftplan.sql:1`.
- **"Leave empty = allowed everywhere".** If a Sales Person has **no**
  shift plan assignment at all, they count as bookable in **every**
  shift plan instance (`get_bookable_sales_persons` / `is_eligible`,
  `service_impl/src/sales_person_shiftplan.rs:100-118` and `137-140`).
- **`planner_only` blocks self-booking.** The person appears in
  `get_bookable_sales_persons` only if the calling user is a
  Shiftplanner (`sales_person_shiftplan.rs:114-115` and
  `147-154`).
- **Inactive Sales Persons are never listed.** Filter in
  `get_bookable_sales_persons` (`sales_person_shiftplan.rs:96-99`).
- **Full replace on set.** `set_shiftplans_for_sales_person`
  writes the assignment list completely fresh, in the same TX, after
  verifying the existence of the Sales Person
  (`sales_person_shiftplan.rs:52-75`).

### Relationships

All tables of the cluster hang off `sales_person(id)` via foreign key
(exception: `sales_person_shiftplan` additionally hangs off
`shiftplan(id)`). SQLite only checks FKs if `PRAGMA foreign_keys=ON` is
active — the soft-delete semantics (`WHERE deleted IS NULL`) make the FK
check only partially meaningful anyway (see
`docs/domain/edge-cases.md` §8).

```
user ────────────< sales_person_user >──── sales_person ──┬──< employee_work_details
                                                          ├──< sales_person_unavailable
                                                          └──< sales_person_shiftplan >── shiftplan
```

## 4. Service API

All services are classified according to the convention in
`shifty-backend/CLAUDE.md`:

| Service | Tier | Foreign domain services? |
| --- | --- | --- |
| `SalesPersonService` | **Basic** | no — only `PermissionService`, `ClockService`, `UuidService`, `TransactionDao` |
| `EmployeeWorkDetailsService` | edge case / **Business-Logic-lite** | consumes `SalesPersonService` as auth helper (`verify_user_is_sales_person`, `find_by_sales_person_id`) |

| `SalesPersonShiftplanService` | **Business-Logic** | consumes `SalesPersonService` (existence + aggregate listing for `get_bookable_sales_persons`) |

The three edge-case services use `SalesPersonService` **only** for
authorization ("is the caller themselves this Sales Person record?")
or existence checks — the cross-aggregate share is minimal.

## 7. Edge cases

For the central reference see
[`../domain/edge-cases.md`](../domain/edge-cases.md). Feature-specific
edges for this cluster:

- **Changing `from` on a running contract is not supported.** `update()`
  only writes back the `to_*` fields + core parameters; anyone wanting
  to shift a start point has to end the existing contract and create a
  new one. See §2.2. Cf. also §1.2 in edge-cases (contract switch).
- **Contract gap between two contracts.** `find_for_week` returns
  `EntityNotFoundGeneric` if the week lies between two contracts
  (`service_impl/src/employee_work_details.rs:119-126`). Reporting
  consumers must handle this case — see edge-cases §1.2.
- **Contract-end proration for non-Dec 31 endings.**
  `vacation_days_for_year` subtracts
  `vac * (1 - ordinal(to) / days_in_year)` at year-end; a Dec 31 ending
  subtracts 0. Regression guards from
  `service/src/employee_work_details.rs:286-408`.
- **Sales Person `is_paid` masking.** Anyone who is not HR and **not**
  themselves the linked user gets `is_paid = None` — not `false`.
  Wrong interpretation in the frontend as "unpaid" would be a
  downstream info leak. See §2.1 and `sales_person.rs:120-143`.
- **Delete = soft delete.** All delete endpoints only set `deleted`.
  Subsequent `create()` calls can create **the same name** again
  (no `UNIQUE(name)`), but the FK subgraph (Bookings, Extra
  Hours, Absences) still hangs off the original record. See
  edge-cases §8.
- **User assignment race.** `sales_person_user` has
  `UNIQUE(user_id)`; a replacement runs `discard_assigned_user` +
  `assign_to_user` in the same TX. Two parallel HR requests assigning
  the same login user to two different Sales Persons will fail on the
  second commit with a SQLite unique constraint error — the error
  message is currently **[To verify]** a generic DAO error, not
  business-readable.
- **`sales_person.background_color`.** Free hex field — no format
  check in the backend. Invalid values (`"red"`, empty string) land raw
  in the DB and render in the frontend as style `background-color:` —
  the style attribute silently ignores garbage. **[To verify]** whether
  validation is worthwhile.
- **Weekday blocks across the year boundary.** `sales_person_unavailable`
  stores `(year, calendar_week)`. Per ISO 8601, CW 1 can lie in
  December of the previous year — consumers must consistently handle
  the semantics of `(iso_week_year, iso_week)`. See edge-cases §4.
- **Shift plan assignment: "empty = everywhere".** This is by design
  (onboarding convenience), but it causes a freshly created Sales
  Person to appear as bookable in **every** shift plan until they get
  a first assignment. Anyone wanting more restrictive behavior must
  deliberately set a non-assignment on creation.
- **`planner_only` semantics.** Sales users **cannot** book themselves
  into such plans — the person doesn't appear in
  `get_bookable_sales_persons` for non-Shiftplanners in the first
  place. The booking service (F03) MUST check `is_eligible` so that
  direct API calls don't open a hole.
- **`get_all` also returns soft-deleted persons.** The DAO method
  `all` in `dao_impl_sqlite/src/sales_person.rs` **[To verify]**
  filters on `deleted IS NULL`; in the frontend the user management
  list also shows inactive but not deleted persons. Anyone needing
  a Sales Person list of "really all ever created" has to build a
  special read — no such endpoint exists at the moment.

Unit tests live under `service_impl/src/test/`:

- **`sales_person.rs`** (26 tests, `service_impl/src/test/sales_person.rs`).
  Covered are all auth combinations of `get_all`/`get`
  (Shiftplanner / Sales / HR / no permission), create validation
  (`IdSetOnCreate`, `VersionSetOnCreate`), update guards (no permission,
  not found, conflict, `deleted` not modifiable, name/color update
  roundtrip), soft-delete success and failure, as well as `exists`.

  Specifically covers the late field extensions: `update` propagates
  `committed_voluntary` and `cap_planned_hours_to_expected` to the DAO
  — these are regression guards for the v2.x migrations. **Known gap:**
  there are **no** unit tests for `create()` validation, `find_for_week`
  across contract boundaries, `all_for_week` filtering by sales user,
  or auth deny paths. Contract proration on the other hand has a dense
  regression suite directly in the service trait module
  (`service/src/employee_work_details.rs:286-408`, 5 tests, Phase 28).
- **`sales_person_unavailable.rs`** (16 tests). `get_all_*`, `get_by_week*`
  and create/delete each in the variants Shiftplanner / self /
  no-permission plus create validation (`IdSetOnCreate`,
  `VersionSetOnCreate`, `AlreadyExists`) and delete not-found.
- **`sales_person_shiftplan.rs`** (21 tests). Covers `get`/`set`
  (including clear = replace with empty list), bookable computation
  ("no assignments = everywhere", filter inactive persons, mixed
  `available`/`planner_only`, include/exclude `planner_only` based on
  the caller role), `is_eligible` in all combinations, and
  auth-deny paths.

Integration tests (in-memory SQLite) run in the same directory; they
share `service_impl/src/test/helpers` (mock auth, time fake).

**Known gaps:**

- Employee Work Details `create`/`find_for_week`/`all_for_week` /
  auth deny paths (see above).
- `SalesPersonService::get_all_paid` — no dedicated test.
- REST handlers currently have no `oneshot` tower tests for this
  cluster (in contrast to `extra_hours`/`feature_flag`, where it is
  the norm).
