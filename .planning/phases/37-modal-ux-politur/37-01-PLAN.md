<objective>
Fix MOD-01: a mouse-drag that begins inside a modal (e.g. text selection) and is released on the backdrop must NOT close the modal. Only a real outside click — mousedown AND mouseup on the backdrop — closes it.

The root cause (dialog.rs backdrop `onclick` fires unconditionally, with no tracking of where the gesture started) is fixed centrally in the shared `Dialog` component so ALL nine `Dialog` users benefit at once (D-02). The one component with its OWN custom backdrop that carries the identical bug — `absence_convert_modal.rs` — is fixed inline with the same pattern in this same plan (D-03), so no known duplicate of the bug is left behind.

Mechanism (D-01): a `use_signal` flag `pressed_on_backdrop`. The backdrop `onmousedown` sets it true; the panel content `stop_propagation`s its own mousedown (so a panel-originated press never reaches the backdrop and never sets the flag) and clears the flag; the backdrop close handler fires `on_close` only when the flag is set, then resets it. No `web_sys` target comparison — Dioxus `MouseData` exposes no `target`/`currentTarget`.

Purpose: eliminate the reported UX defect where selecting text inside a form and releasing outside destroys the user's in-progress edit.

Output / Artifacts this phase produces:
- A pure, unit-tested decision helper `BackdropPress` in dialog.rs (the `pressed_on_backdrop` state machine).
- Signal-flag wiring in `DialogContent` (backdrop + panel mouse handlers).
- The same pattern applied inline to the `absence_convert_modal.rs` custom backdrop, reusing the shared helper.

No new attack surface: this is pure client-side modal open/close UI state; no server trust decision depends on it. No new dependencies, no migration, no snapshot bump.
</objective>
