      ```
      async fn get_year(&self, year: u32, context: Authentication<Self::Context>, tx: Option<Self::Transaction>) -> Result<Arc<[(u8, Arc<[ShortEmployeeReport]>)]>, ServiceError> {
          let tx = self.transaction_dao.use_transaction(tx).await?;
          // Bulk-Loads: 3 DAO-Roundtrips für das ganze Jahr statt 55×3.
          // employee_work_details.all liefert alle Rows — `find_working_hours_for_calendar_week`
          // filtert im Helper pro Woche. Kein `_for_year`-Endpoint nötig (existiert eh nicht).
          let work_details = self
              .employee_work_details_service
              .all(Authentication::Full, tx.clone())
              .await?;
          let shiftplan_reports = self
              .shiftplan_report_service
              .extract_shiftplan_report_for_year(year, Authentication::Full, tx.clone())
              .await?;
          let extra_hours = self
              .extra_hours_service
              .find_by_year(year, Authentication::Full, tx.clone())
              .await?;

          let weeks_in_year = time::util::weeks_in_year(year as i32);
          let weeks: Vec<(u32, u8)> = (1..=weeks_in_year).map(|w| (year, w)).collect();

          let assembled = self
              .assemble_weeks(&weeks, &work_details, &shiftplan_reports, &extra_hours, context, tx.clone())
              .await?;

          self.transaction_dao.commit(tx).await?;
          Ok(assembled.into())
      }
      ```

    - **Vec-Contract (D-52-03):** `assemble_weeks` liefert bereits `Vec<(u8, Arc<[ShortEmployeeReport]>)>` in Eingabe-Reihenfolge; die Eingabe ist `(1..=weeks_in_year(year))` strikt aufsteigend → Ausgabe erfüllt D-52-03 automatisch. Für Wochen ohne Bookings/WorkingHours liefert der Helper ein leeres `Arc<[]>` (weil die per-Woche-Aggregations-Schleife über `working_hours`-HashMap leer ist → kein `push` → leerer Vec → `.into()` → leeres `Arc<[]>`). Das entspricht der Semantik von `get_week` für dieselbe Woche.

    - **Kein neuer Test in diesem Plan** — Byte-Identität wird durch die Wave-1-Fixtures gegen `get_weekly_summary` (in Wave 4 Plan 05 nach Umbau) verifiziert. Ein separater `get_year` vs. 55×`get_week`-Test ist redundant, weil beide auf denselben Helper mit denselben Slice-Referenzen delegieren würden. **Aber:** Ein rudimentärer Sanity-Test wird ergänzt (nächster Task), der beweist dass für ein Mini-Setup mit 2 Wochen + 1 MA das Ergebnis 2 Elemente hat und die Indizes stimmen.
