<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser (Cookie) → REST | `<a href download>` schickt Cookie automatisch; `forbid_unauthenticated`-Middleware bewacht — 401 bei fehlender Auth. |
| REST → Service | Handler reicht `context.into()` (typed `Authentication<User>`) an den Service durch. |
| Handler-Pre-Check → Service-Gate | Race-Window <1ms — Handler-Pre-Check ist Primaerpfad fuer 409; Service-Gate ist Defense-in-Depth (Wave 1). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-49-01 | Spoofing / Authentication | `download_week_pdf` handler | medium | mitigate | Bestehende `forbid_unauthenticated`-Middleware (rest/src/lib.rs) — 401 fuer nicht-authentifizierte Requests. Handler-Test `rest_unauthenticated_returns_401` verifiziert. |
| T-49-02 | Elevation of Privilege | Access-Control | low | accept | Kein Admin-Gate per PDF-05 (bewusst). Employee-Auth → 200 verifiziert via `rest_employee_auth_returns_200`. Alle authentifizierten User haben Read-Zugriff auf Shiftplan-Views (v2.3-Scope-Konsens). |
| T-49-03 | Tampering / Input Validation | Path params + WeekStatus gate | medium | mitigate | Axum-Typing (Uuid/u32/u8) filtert schlecht geformte Requests → 400. WeekStatus-Pre-Check im Handler fuer Race-Window; Service-Gate als Defense-in-Depth (Wave 1). Test-Matrix ueber alle 4 WeekStatus-Werte. |
