## Spacing Scale

Tailwind's numeric scale is used throughout the app. The effective step values (all multiples of 4) are:

| Token | Tailwind class | px equivalent | Usage in this phase |
|-------|---------------|---------------|---------------------|
| xs | `gap-1`, `p-1` | 4px | Icon gaps, tiny inline padding |
| sm | `p-2`, `gap-2`, `py-2`, `gap-3`, `py-3` | 8px / 12px | Dense warning items, button vertical padding, form-row gaps |
| md | `px-4`, `p-4`, `mx-4` | 16px | Section padding + outer margins (warning/conflict blocks) |
| lg | `px-3` | 12px | Toggle button + nav item horizontal padding |
| xl | `pb-4` | 16px | Heading bottom spacing |

Pattern from existing conflict section: `mx-4 my-3 px-4 py-3 bg-bad-soft border border-bad rounded-md`
Pattern from existing warning banner: `mx-4 mt-3` outer wrapper, inner `border border-warn bg-warn-soft rounded-md px-4 py-3`

New Phase-24 elements:
- Settings page form rows use `gap-3` (12px) between label and control, matching slot-edit form pattern.
- Toggle control uses `px-3 py-2` (12px/8px) for the button, consistent with `admin_panel_item_class`.

### Inherited legacy values — do NOT use for new Phase 24 elements

These non-multiple-of-4 arbitrary values exist in the surrounding code (form inputs, top bar). They are documented only so the executor recognizes them in context. **New Phase 24 surfaces must NOT adopt them** — the new toggle button stays at `px-3 py-2`.

| Legacy class | px | Where it appears (existing code) |
|--------------|-----|----------------------------------|
| `px-[10px]` | 10px | Existing form-input horizontal padding (`slot_edit.rs`, select fields) |
| `h-[34px]` | 34px | Existing form-input / select height |

## Typography

All roles are already defined in `tailwind.config.js` and used throughout the existing app.
Phase 24 uses only a subset of the existing scale, and constrains itself to exactly **two font weights**: `font-normal` (400) and `font-semibold` (600).

| Role | Tailwind class | Size | Weight | Line Height | Usage in this phase |
|------|---------------|------|--------|-------------|---------------------|
| Micro (eyebrow) | `text-micro font-semibold uppercase` | 11px | 600 | 14px | Section eyebrow header in warning sections and settings page sub-labels |
| Small | `text-small font-normal` | 12px | 400 | 16px | Helper text, descriptions below toggle, inline errors, success flash |
| Body | `text-body` | 14px | 400 | 20px | Warning list items, toggle state label, inline error text |
| Label (emphasized) | `text-body font-semibold` | 14px | 600 | 20px | Toggle row label, toggle-on button state |
| H2 | `text-h2 font-semibold` | 18px | 600 | 24px | Settings page heading, warning section heading |

**Two-weight contract:** only `font-normal` (400) and `font-semibold` (600) are used. `font-medium` (500) is intentionally NOT used anywhere in Phase 24 — any place that would otherwise be medium-weight (e.g. the toggle row label) uses `font-semibold` instead.

Phase 23 established: warning eyebrow = `text-micro text-warn font-semibold uppercase`.
Phase 24 overage section heading mirrors the conflict section: `text-h2 font-semibold pb-2 text-warn` (with ⚠️ prefix glyph, matching the conflict block pattern at `shiftplan.rs:871`).

## Color

All values are CSS variable aliases defined in `tailwind.config.js` and resolved via `input.css`.

**60/30/10 split:** inherited from the existing app's surface distribution (dominant `surface`/`bg`, secondary `surface-alt`/cards, accent reserved for active nav and key emphasis). Phase 24 does NOT introduce a new dominant/secondary palette — it only layers the existing `accent`, `warn`, and `bad` semantic tokens on top of the existing surface distribution for the three new surfaces.

| Role | Token | Tailwind classes | Usage in this phase |
|------|-------|-----------------|---------------------|
| Dominant surface | `surface` | `bg-surface` | Settings page background, form area |
| Secondary surface | `surface-alt` | `bg-surface-alt` | Hover state on toggle button, nav items |
| Border | `border` | `border-border` | Settings card border, admin panel container |
| Ink | `ink` | `text-ink` | Body text, toggle label |
| Ink soft | `ink-soft` | `text-ink-soft` | Descriptive text below toggle |
| Ink muted | `ink-muted` | `text-ink-muted` | Helper text, section sub-labels |

| Warn | `warn` | `text-warn`, `bg-warn-soft`, `border-warn`, `border-l-[3px] border-warn` | Overage warning section (both the heading and the WarningList border-left) |

**Accent reserved for:** active nav state (Settings nav entry when on `/settings/`), toggle "on" state indicator text.

- Settings page card/panel: `bg-surface border border-border rounded-md` — matches `ADMIN_PANEL_CONTAINER_CLASS`.

## Component Inventory

Three UI surfaces added in this phase. All reuse or mirror existing patterns.

### Surface 1: `/settings/` route — Paid-Limit toggle (D-24-06)

**Route:** `Route::Settings {}` at `/settings/`
**Component:** `SettingsPage` in `shifty-dioxus/src/page/settings.rs`
**Nav entry:** Added to `NavVisibility` as `settings: has("admin")` in `top_bar.rs`; classified as `is_admin_target(NavTarget::Settings) = true` (lives in the admin dropdown).
**Nav key:** Reuse existing `Key::Settings` → En: "Settings" / De: "Einstellungen" / Cs: "Nastavení"

Page layout mirrors `UserManagementPage`:
- Page header: `text-h2 font-semibold` title from `Key::Settings`
- Single card/section: `bg-surface border border-border rounded-md p-4` containing one toggle row
- Toggle row: label (`text-body text-ink font-semibold`) + description (`text-small text-ink-soft`) + toggle control

Toggle control: a `<button>` element (NOT `<input type="checkbox">`) with two visual states:
- **Off (weich / soft):** `px-3 py-2 rounded-md border border-border text-ink text-body bg-surface` — neutral, matches inactive `admin_panel_item_class`
- **On (hart / hard):** `px-3 py-2 rounded-md border border-bad text-bad text-body font-semibold bg-bad-soft` — uses `bad` semantic color because hard enforcement is a restrictive action
- Toggle state label: shown inline after the button text, `text-small text-ink-muted`

The toggle calls `PUT /toggle/paid_limit_hard_enforcement/enable` or `disable` via the existing Toggle REST API.

### Surface 2: Inline hard-block error at booking slot (D-24-05)

**Location:** `shifty-dioxus/src/page/shiftplan.rs` — the `AddUserToSlot` handling block (currently at `:403-441`)
**Trigger:** Backend returns new HTTP status (not 403) for `PaidLimitExceeded`; frontend detects and renders inline.
**Rendering:** A non-dismissible inline div directly below the slot interaction area (NOT a modal, NOT a dismissible banner).

Pattern: `div { class: "text-bad text-small font-normal mt-1", "..." }` — same pattern as the slot edit save error at `slot_edit.rs:265`.

This error clears on the next successful booking or when the user navigates away. It does NOT replace the existing `booking_warnings` dismissible banner — it is a separate, non-dismissible inline element scoped to the slot.

### Surface 3: Persistent overage warning section above the shiftplan (D-24-03)

**Location:** `shifty-dioxus/src/page/shiftplan.rs` — inserted below the existing `booking_warnings` banner, above the `ShiftplanTabBar` div.
**Trigger:** Computed client-side from `state::shiftplan::Slot.current_paid_count > slot.max_paid_employees` for all slots in the loaded week.
**Visibility:** ALL roles (consistent with Phase 23 D-23-05).
**Dismissible:** No — persistent per week load (reloads disappear naturally; no dismiss button).

Reuse `WarningList` component from `shifty-dioxus/src/component/warning_list.rs`:
- The existing `WarningTO::PaidEmployeeLimitExceeded` rendering already produces "X/Y" output via `Key::BookingWarningPaidLimitExceeded`.
- However, the week-overage section uses a section heading (new i18n key, see below) instead of the WarningList's internal booking-warning header.
- Preferred approach: render a custom `div` container matching the conflict section style (`mx-4 my-3 px-4 py-3 bg-warn-soft border border-warn rounded-md`) with a heading and a list of overage slots — parallel to the conflict block at `shiftplan.rs:869-903`.
- Alternative (also acceptable): pass a synthesized `Vec<WarningTO>` to `WarningList` with `suppress_header: false` — but this requires constructing `WarningTO` values from state, which means the section heading key is unused.
- **Recommended approach:** custom section (not `WarningList`) — keeps the section heading localized and avoids synthesizing `WarningTO` from state.

Each row in the overage list shows: slot name + day/time label + "X/Y" count. Format: `"{slot_name}: {current}/{max}"` — derived from slot state, no backend round-trip.

## Interaction Contracts

### Settings toggle interaction

1. User sees toggle button showing current state (On/Off text).
2. User clicks toggle.
3. Frontend immediately sends `PUT /toggle/paid_limit_hard_enforcement/enable` or `/disable`.
4. On success: toggle state flips, brief `SettingsSaved` confirmation shown inline (same `text-small text-ink-muted` pattern, disappears after 2s or on next interaction).
5. On error: `SettingsSaveError` shown as `text-bad text-small font-normal` inline below toggle. Toggle reverts to previous state.
6. No loading spinner needed (toggle call is fast). Disable the button during the in-flight request to prevent double-click.

### Hard-block inline error interaction

1. User clicks "+" on a slot.
2. Backend returns the new `PaidLimitExceeded` HTTP status (not 403).
3. Frontend renders `BookingBlockedPaidLimit` text inline at the slot (non-dismissible).
4. Error clears when the user performs another booking action on that slot, or navigates away.
5. No sound, no toast, no modal — strictly inline.

### Overage warning section behavior

1. Section is computed client-side on every render of the shiftplan page (no extra API call).
2. Section appears between the booking-warnings banner and the `ShiftplanTabBar` div.
3. Section is NOT dismissible.
4. Section shows all overage slots for the loaded week, sorted by slot time/day order (natural order from `state::Shiftplan.slots`).
5. Each row is non-interactive (no click action; it is purely informational).

## Accessibility

- Settings toggle button: `aria-pressed="true"/"false"` reflects current state. Label text must remain visible (no icon-only toggle).
- Inline error messages: rendered as plain text in a `div` with no special ARIA role needed (they are adjacent to the action that triggered them).
- Overage section: headings use correct semantic level (`h2` or `div` with `text-h2` class, consistent with existing conflict section at `shiftplan.rs:871`).
- All new i18n strings must be present in all three locales (En, De, Cs) — enforced by `i18n_*_present_in_all_locales` test pattern.
