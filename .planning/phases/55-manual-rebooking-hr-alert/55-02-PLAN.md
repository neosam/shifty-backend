`generate_route<RestState: RestStateDef>() -> Router<RestState>` mit vier
Routen (siehe Aufteilung in `rest/src/lib.rs`, Task C):

- `POST "/rebooking/manual"` → `post_manual::<RestState>`
- `GET "/rebooking-suggestions"` → `get_pending::<RestState>`
- `POST "/rebooking-suggestions/{id}/approve"` → `post_approve::<RestState>`
- `POST "/rebooking-suggestions/{id}/reject"` → `post_reject::<RestState>`

Handler-Details:

- `post_manual`: nimmt `axum::Json<ManualRebookingRequestTO>`, parst
  `direction` in `service::rebooking_reconciliation::RebookingDirection`
  über die `From`-Impl aus rest-types, ruft
  `rebooking_reconciliation_service().rebook_manual(...)`. Fehlerbehandlung:
  bei `ServiceError::EntityAlreadyExists(_)` → 409 + Body
  `{"error":"RebookingErrorSlotTaken"}` (i18n-Key als String, kein
  SQL-Leak, T-4). Bei `ServiceError::ValidationError(_)` → 400. Sonst via
  `error_handler`. Erfolg → 200 + JSON `RebookingBatchTO::from(&entity)`.
- `get_pending`: nimmt keine Query-Parameter (v1: alle pending Suggestions
  über alle Personen, HR-gated), ruft
  `rebooking_reconciliation_service().list_pending_for_sales_person(None,
  context, None)` (Plan 55-01 liefert die `Option<Uuid>`-Signatur — `None`
  = alle phase-weit). Rückgabe ist bereits `Arc<[RebookingSuggestion]>`
  (Plan 55-01 hydratisiert intern per `hydrate_pending_to_suggestion`).
  Handler mapped `RebookingSuggestionTO::from(&s)` für jede Suggestion.
  Response: `Json(Vec<RebookingSuggestionTO>)`.
- `post_approve`: `Path(batch_id)`, ruft `approve_suggestion`. Bei
  `ServiceError::BatchAlreadyResolved` → 409 + Body
  `{"error":"RebookingErrorAlreadyResolved"}`. Erfolg → 200 +
  `RebookingBatchTO`.
- `post_reject`: analog approve, ruft `reject_suggestion`.

ApiDoc-Struct `RebookingApiDoc` analog `ReportApiDoc` in `report.rs`,
`components(schemas(RebookingBatchTO, RebookingSuggestionTO,
ManualRebookingRequestTO, RebookingBatchKindTO, RebookingBatchStateTO,
RebookingDirectionTO))`. Tag `Rebooking`.
