**B) `shifty-dioxus/src/loader.rs`:**

Ergänze am Ende (nach `load_voluntary_stats`) vier neue async Funktionen.
Nutze das bestehende `Config`-Argument-Muster + `reqwest::Client::new()`.
Base-URL-Muster: analog `load_voluntary_stats` (`config.backend`, siehe
Zeilen ~310).

1. `pub async fn submit_manual_rebooking(config: Config, request:
   ManualRebookingRequest) -> Result<RebookingBatch, RebookingSubmitError>`:
   - POST `/rebooking/manual` mit JSON-Body =
     `ManualRebookingRequestTO::from(&request)`.
   - 200 → parse `RebookingBatchTO`, gib `RebookingBatch::from(&to)` zurück.
   - 409 mit Body `{"error":"RebookingErrorSlotTaken"}` → `Err(SlotTaken)`.
   - 400 → `Err(Other(<error-body>))`.
   - Sonstige Fehler → `Err(Other(...))`.

2. `pub async fn load_rebooking_suggestions_pending(config: Config) ->
   Result<Vec<RebookingSuggestion>, ShiftyError>`:
   - GET `/rebooking-suggestions` → parse `Vec<RebookingSuggestionTO>` →
     `RebookingSuggestion::from(&to)` collect.

3. `pub async fn approve_rebooking_suggestion(config: Config, batch_id:
   Uuid) -> Result<RebookingBatch, RebookingSubmitError>`:
   - POST `/rebooking-suggestions/{batch_id}/approve` (kein Body).
   - 200 → `RebookingBatch`. 409 mit
     `{"error":"RebookingErrorAlreadyResolved"}` →
     `Err(AlreadyResolved)`.

4. `pub async fn reject_rebooking_suggestion(config: Config, batch_id:
   Uuid) -> Result<RebookingBatch, RebookingSubmitError>`: analog.

Der bestehende `ShiftyError` in `shifty-dioxus/src/error.rs` bleibt für
GET-Fehler; `RebookingSubmitError` ist der Approve/Reject/Submit-spezifische
Fehler-Typ (weil die Modals inline reagieren müssen).
