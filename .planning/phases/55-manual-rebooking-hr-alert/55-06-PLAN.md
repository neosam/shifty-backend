**`## Manuelle Umbuchung (F3)` (EN: `## Manual Rebooking (F3)`)**

Inhalt (Prosa, keine Code-Fences außer für kurze Beispiel-JSON):

1. Trigger: HR-User navigiert zu `/employees/{id}` und öffnet den
   „Manuelle Umbuchung"-Menu-Item (TopBar bzw. Header).
2. Modal-Struktur: Wochen-Feld (ISO year + ISO week), Direction-Radio
   (VolunteerWork↔ExtraWork), Hours-Input, Preview-Sektion, Submit.
   D-55-05 (HR wählt Woche frei, auch retrospektiv), D-55-06 (Direction
   im Modal, nicht abgeleitet).
3. Submit-Flow: `POST /rebooking/manual` mit `ManualRebookingRequestTO`.
   Backend `RebookingReconciliationService::rebook_manual`:
   - HR-Gate zuerst.
   - Öffnet Transaktion.
   - Schreibt zwei ExtraHours-Rows (`ExtraHoursSource::Rebooking`,
     Kategorien nach Direction), verknüpft sie mit einem Batch
     (`kind=Manual, state=Approved`) + einem Entry.
   - `RebookingBatchService::create` macht Pre-Check auf UNIQUE-Slot
     `(sales_person_id, iso_year, iso_week)` — Kollision → 409.
4. State-Machine: manueller Batch startet direkt in `Approved` — kein
   Pending-Zwischenschritt (D-55-04: kein Undo, keine State-Änderungen
   nach Approve).
5. Fehler: 409 mit i18n-Key `RebookingErrorSlotTaken`; Modal bleibt
   offen, User kann Woche ändern.
6. Datenmodell-Zustand nach Erfolg: zwei ExtraHours + Batch + Entry;
   Read-Aggregate (VOL-STAT-01, VOL-ACCT-01, Balance) bleiben durch
   den `source==Manual`-Filter im ReportingService neutral (VOL-ACCT-03,
   Property-Test 55-03).

**`## HR-Alert + Vorschlags-Modal (F5)` (EN: `## HR-Alert + Suggestion Modal (F5)`)**

Inhalt:

1. Predicate (D-55-01): `alert_predicate(balance, voluntary_ist,
   cap_active) = cap_active && balance <= -0.5h && voluntary_ist > 0`.
   Float-Noise-Tolerance via `<= -0.5h`. Pure fn in
   `service/src/rebooking_reconciliation.rs`; Truth-Table-Test in
   `service_impl/src/test/rebooking_reconciliation.rs`.
2. Backend-Ripple (D-55-02): `ShortEmployeeReportTO` bekommt
   `has_pending_rebooking: bool` + `pending_rebooking_id: Option<Uuid>`.
   Reporting-Chain prüft Predicate-gated + HR-gated + DAO-Query
   `find_pending_for_sales_person`. Non-HR sieht `false`/`None`.
3. Alert-UI: `component/rebooking_alert_banner.rs` — Inline-Banner (KEIN
   Dialog, MEMORY-Regel `feedback_warnings_inline_not_dialog`) in der
   Employees-Overview-Liste.
4. Suggestion-Modal (`component/rebooking_suggestion_modal.rs`):
   - IST-Spalte: Balance, Voluntary-Ist, Voluntary-Soll,
     Voluntary-Delta — jeweils `_before` (Backend-computed, D-55-03,
     inkl. `voluntary_delta_before` — kein FE-Subtraktion).
   - DANN-Spalte: dieselben mit `_after` (Backend-berechnet, D-55-03,
     inkl. `voluntary_delta_after` — kein FE-Subtraktion).
   - `proposed_hours = min(|balance|, voluntary_ist)` — Backend-computed.
   - `RebookingSuggestion` enthält `voluntary_delta_before/after` als
     eigene Backend-computed Felder (Fat-Backend-Regel MEMORY
     `feedback_fat_backend_thin_client`).
   - Approve / Reject als zwei Button-Aktionen.
5. State-Machine:
   - Suggest (`suggest_for_week`): Batch `kind=HrSuggestion,
     state=Pending`. UNIQUE-Slot ist per Claim-on-Suggest belegt
     (HR-ALERT-04) — verhindert Doppelvorschläge in derselben Woche.
     Keine ExtraHours-Rows in Pending.
   - Approve (`approve_suggestion`): state-conditional UPDATE Pending →
     Approved (`WHERE state='pending'`, affected-rows-Check) + schreibt
     die 2 Pair-ExtraHours-Rows. Bei affected==0 (Race, T-1) → 409 mit
     i18n-Key `RebookingErrorAlreadyResolved`.
   - Reject (`reject_suggestion`): state-conditional UPDATE Pending →
     Rejected. UNIQUE-Slot bleibt belegt bis nächste ISO-Woche
     (D-55-07 — verhindert Neu-Vorschlag in derselben Woche).
6. Alert-Terminierung (D-55-07): Approve UND Reject beenden den Alert;
   Backend liefert `has_pending_rebooking=false` nach beiden Terminal-
   States.
7. No-Undo (D-55-04, Anti-Feature REB-UNDO-01): Approve und Reject sind
   one-shot. Keine State-Rückwärtstransitionen.
