## Spacing Scale

All spacing follows the Tailwind p-/gap-/m- scale mapped to CSS pixel values.
Only multiples of 4 are used; the existing Settings page establishes the baseline.

| Token | Tailwind class | Value | Usage in this phase |
|-------|---------------|-------|---------------------|
| xs | `p-1` / `gap-1` | 4px | label-to-description gap in card rows |
| sm | `p-2` / `gap-2` | 8px | button row gaps, list row internal gaps |
| md-tight | `gap-3` | 12px | inter-row gap inside a card (matches Card-1/Card-2) |
| md | `p-4` / `gap-4` | 16px | card padding, input width constraint, page padding unit |
| lg | `mt-4` | 16px | card-to-card vertical gap (matches existing `mt-4` between Card-1 and Card-2) |
| xl | `px-4 py-4 md:px-6 lg:px-8` | 16–32px | page wrapper padding (inherited, no change) |

**Exceptions:**
- Date input width cap: `max-w-[200px]` — matches Card-2 date input width exactly.
- Time input width cap: `max-w-[140px]` — compact time-of-day input.
- Year picker width: `w-24` (96px) — 4-digit number, no wider needed.
- List row internal vertical padding: `py-2` — consistent with existing table/list row patterns.
- Delete button inside list row: `px-2 py-1` — compact badge-button pattern from `billing_periods.rs`.

## Typography

Sourced from `tailwind.config.js` `fontSize` extension. All font sizes are canonical and
must not be substituted with Tailwind default sizes.

| Role | Tailwind class | Size / Line-height | Weight | Usage in this phase |
|------|---------------|-------------------|--------|---------------------|
| Micro label | `text-micro` | 11px / 14px | 600 | Type badges (Holiday/ShortDay), table-header-style labels |
| Small | `text-small` | 12px / 16px | 500 | Form hints, inline errors, context suffix, delete feedback |
| Body | `text-body` | 14px / 20px | 400 | Card descriptions, list date strings, form field values, dropdown options |
| Section heading | `text-body font-semibold` | 14px / 20px | 600 | Card feature label row (top of each card) — matches Card-1/Card-2 pattern |
| Page heading | `text-h2 font-semibold` | 18px / 24px | 600 | Existing `<h1>` on Settings page — no change |

**Inherited three-weight exception (400 / 500 / 600):**

This phase uses three font weights, which exceeds the generic two-weight guideline.
This is an explicit, justified exception: all three weights are **inherited verbatim from
the existing project-wide design system** — they are not introduced by this phase and
reducing them would break app-wide visual consistency.

- **Weight 400** — baked into `text-body` in `tailwind.config.js` (`fontWeight: '400'`).
- **Weight 500** — baked into `text-small` in `tailwind.config.js` (`fontWeight: '500'`).
  Verified in `settings.rs`: Card-1 and Card-2 both use `text-small text-ink-soft` and
  `text-small text-ink-muted`, so 500 is the existing app baseline for secondary/hint text.
  Card-3 must use `text-small` in the same positions to stay consistent.
- **Weight 600** — used by `font-semibold` on card feature labels and `text-micro`/`text-h2`.
  Verified in `settings.rs`: both Card-1 and Card-2 use `text-body text-ink font-semibold`
  for their section label rows.

**Conclusion:** Implementing Card-3 with only 2 weights would require either (a) removing
`text-small` from secondary labels (breaking Card-1/Card-2 consistency) or (b) overriding
`text-small`'s baked-in weight with `font-normal` everywhere it appears (non-standard and
fragile). Neither is acceptable. The three weights are the project's canonical type scale.

**Rules:**
- Card feature label: ALWAYS `text-body text-ink font-semibold`.
- Card description: ALWAYS `text-small text-ink-soft` on the line below the feature label.
- Form field labels: `text-small text-ink-muted` above each input.
- Inline success flash: `text-small text-ink-muted`.
- Inline error: `text-small text-bad` (or `text-bad text-small` — class order does not matter).
- List entry primary line: `text-body text-ink`.
- List context suffix `(Samstag, KW 33, 2026)`: inline within the `text-body text-ink` string,
  not a separate element — the full formatted string is built in Rust and rendered as a single span.

## Surface 1: Shiftplan Per-Weekday Dropdown

**Gate:** `is_shiftplanner` — already derived at component top in `shiftplan.rs` line 102-105.
Only rendered when `is_shiftplanner == true`. No change to the page-level auth.

**Placement:** Inside the existing WeekView per-day column, below the weekday header
and above/alongside the slot list. One `DropdownTrigger` per weekday (Mo–So). The trigger
label reflects the current state for that day.

**Trigger label states:**

| State | Label (de) | Label (en) | Label (cs) |
|-------|-----------|-----------|-----------|
| No special day | `—` or `{i18n.t(Key::ShiftplanDayTypeNone)}` | `—` | `—` |
| Holiday active | `{i18n.t(Key::ShiftplanDayTypeHoliday)}` = "Feiertag" | "Holiday" | "Svátek" |
| ShortDay active | `{i18n.t(Key::ShiftplanDayTypeShortDay)}` = "Kurzer Tag" | "Short Day" | "Zkrácený den" |

**Trigger visual:** compact, no border by default; on hover `bg-surface-alt`. When a special
day is active, the trigger gets a colored dot indicator beside the text:
- Holiday: `inline-block w-2 h-2 rounded-full bg-accent mr-1.5`
- ShortDay: `inline-block w-2 h-2 rounded-full bg-warn mr-1.5`

This keeps the grid compact without wide persistent badges.

**Dropdown entries (from `DropdownEntry`):**

| Entry text (de) | Entry text (en) | Action | Enabled when |
|----------------|----------------|--------|-------------|
| "Feiertag" | "Holiday" | `create_special_day(Holiday, time_of_day=None)` then reload | Always |
| "Kurzer Tag..." | "Short Day..." | Opens inline time-of-day prompt (see below) | Always |
| "Nichts" | "None" | `delete_special_day(existing_id)` then reload | Special day exists on that day |

**ShortDay time-of-day interaction (from dropdown):**
When the user selects "Kurzer Tag...", the `DropdownBase` closes and a compact inline
form appears in that day's column (replacing the trigger button area) showing:
- `input[type=time].form-input.max-w-[140px]` — time-of-day field
- `Btn { variant: Primary }` with label `i18n.t(Key::ShiftplanDayShortDayConfirm)` = "Uhrzeit speichern" / "Save time" / "Uložit čas"
- `Btn { variant: Ghost }` with label `i18n.t(Key::CancelLabel)` = "Abbrechen" / "Cancel" / "Zrušit"

Submit is disabled until the time field is non-empty. On confirm:
→ `create_special_day(ShortDay, time_of_day=Some(parsed_time))` → reload `for-week`.
On cancel: the inline form closes, trigger state reverts to previous.

**Error handling (shiftplan dropdown):**
On API error (create or delete), show a non-modal inline error below the trigger in that
day column: `span.text-small.text-bad { i18n.t(Key::SettingsSaveError) }`. Clears on next
successful action or week navigation.

**No duplicate check on shiftplan surface:** The dropdown "Feiertag" and "Kurzer Tag"
options are always available. If a Holiday already exists and the user picks "Feiertag"
again, the backend will return an error (duplicate), which surfaces via the inline error
span. The "Nichts" option is disabled (grayed out in the DropdownEntry with `disabled: true`)
when no special day exists for that day.

## Surface 2: Settings Page — Card-3 (Special Days)

**Gate:** Per-Card shiftplanner gate — NOT the page-level admin gate.
The Settings page currently gates the whole page on `has_privilege("admin")`. Card-3 adds
its own inner guard:

```rust
let is_shiftplanner = AUTH.read().auth_info.as_ref()
    .map(|a| a.has_privilege("shiftplanner"))
    .unwrap_or(false);
if is_shiftplanner {
    // render Card-3
}
```

The page-level admin gate remains unchanged (D-33-02). A user who is shiftplanner-but-not-admin
cannot reach this page via the nav — that is an existing limitation noted in the
`admin-rolle-bekommt-alle-privilegien` todo. The Card-3 gate ensures no 403 mismatch for
users who DO reach the page with both privileges.

**Card-3 structure:**

```
div.bg-surface.border.border-border.rounded-md.p-4.flex.flex-col.gap-3.mt-4
│
├── Row A — Feature label + description
│   div.flex.flex-col.gap-1
│   ├── span.text-body.text-ink.font-semibold  {i18n.t(Key::SettingsSpecialDaysSectionLabel)}

│   │

│   │   div.flex.flex-col.gap-1   [time-of-day input]
│   │   ├── label.text-small.text-ink-muted  {i18n.t(Key::SettingsSpecialDaysTimeLabel)}
│   │   └── TextInput { input_type: "time", value: time_str }  [max-w-[140px]]
│   │
│   └── Btn { variant: Primary, disabled: !form_valid }
│       {i18n.t(Key::SettingsSpecialDaysAddBtn)}   ← "Sondertag anlegen" / "Add Special Day"
│
├── Row D — Inline hints and errors (shown conditionally)
│   ├── if duplicate detected:
│   │   span.text-small.text-bad  {i18n.t(Key::SettingsSpecialDaysDuplicateHint)}
│   ├── if add_error:
│   │   span.text-small.text-bad  {i18n.t(Key::SettingsSaveError)}   ← reuse existing key
│   └── if add_success (flash, auto-clear after ~2s or on next action):
│       span.text-small.text-ink-muted  {i18n.t(Key::SettingsSaved)}  ← reuse existing key
│
└── Row E — Chronological list (loaded via get_special_days_for_year)
    ├── [Empty state — when list is empty for selected year]
    │   div.py-6.text-center
    │   └── p.text-body.text-ink-muted  {i18n.t(Key::SettingsSpecialDaysEmptyBody)}
```

**Form validity rules:**
- `form_valid` = `!date_str.is_empty() && (type != ShortDay || !time_str.is_empty())`
- Submit button `disabled: !form_valid` using `Btn`'s `disabled` prop → `opacity-50 cursor-not-allowed`.
- Duplicate detection: after user changes date, check loaded list for same
  `(year, calendar_week, day_of_week)` derived from the chosen date. If match found,
  show `Key::SettingsSpecialDaysDuplicateHint` in Row D immediately (before submit).
  The Add button MAY remain enabled (letting the backend also reject); the hint is the
  primary UX signal. This avoids an extra API call for pre-flight check.

**Date-to-KW mapping (WASM caveat D-25-06):**
`<input type="date">` signal update is unreliable in WASM — use the same workaround
as Card-2 in `settings.rs`: bind via `oninput` event (not `onchange`), reading
`event.data.value()`. The Rust-side mapping uses `time::Date::parse` +
`time::Date::from_iso_week_date` + `as_shifty_week` (as in SPD-01 / D-33-04).

**Date context string construction (SPD-02):**
Built in Rust from the `SpecialDayTO` fields:
```
let date = time::Date::from_iso_week_date(entry.year as i32, entry.calendar_week, weekday)?;
let weekday_key = match entry.day_of_week { /* → Key::Monday .. Key::Sunday */ };
let display = format!(
    "{} ({}, KW {}, {})",
    locale.format_date(&date),        // locale-aware: "15.08.2026" / "08/15/2026"
    i18n.t(weekday_key),              // "Samstag" / "Saturday" / "Sobota"
    entry.calendar_week,
    entry.year
);
```
The "KW" abbreviation for calendar week also gets its own i18n key
`SettingsSpecialDaysCalendarWeekAbbr` so CS/EN can use a different abbreviation.

**Year picker load behavior:**
- On mount: load `get_special_days_for_year(current_year)`.
- On year input change (debounce or on-blur): reload list for new year.
- Year signal starts at `js::get_current_year()`.

**Delete feedback:**
- No confirmation dialog (D-33-08 intent: "Liste aktualisiert sich sofort").
- On success: list reloads, entry disappears. No explicit flash needed.
- On error: `span.text-small.text-bad` inline below the list:
  `{i18n.t(Key::SettingsSpecialDaysDeleteError)}`.

## Interaction Contract Summary

### Create flow (Settings Card-3)

1. User enters date + selects type (+ time if ShortDay).
2. As user changes date: live duplicate check against loaded list → show/hide Row D hint.
3. Submit button enabled only when `form_valid` (date non-empty + time non-empty if ShortDay).
4. On click: `Btn` disabled during save (prevent double-submit); call `create_special_day`.
5. On success: reload `get_special_days_for_year(year)`, clear form fields, show `SettingsSaved` flash (auto-clear after 2 s).
6. On error: show `SettingsSaveError` in Row D; button re-enables.

### Delete flow (Settings Card-3)

1. User clicks "Löschen" Btn (Danger variant) in list row.
2. No confirmation dialog.
3. Button disabled during delete.
4. On success: reload list; entry disappears.
5. On error: show `SettingsSpecialDaysDeleteError` inline below list.

### Shiftplan dropdown flow

1. Per-day dropdown trigger shows current state (—, Feiertag, Kurzer Tag).
2. Click trigger → `DropdownBase` opens with entries.
3. "Feiertag" → immediate `create_special_day` (no time needed) → reload `for-week`.
4. "Kurzer Tag..." → dropdown closes, inline time prompt appears in that day column.
5. User enters time → "Uhrzeit speichern" → `create_special_day(ShortDay, time_of_day)` → reload.
6. "Nichts" (enabled only when entry exists) → `delete_special_day(id)` → reload.
7. On any API error: show `span.text-small.text-bad` inline in that day column.
