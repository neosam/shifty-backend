<objective>
Close UV-05: after converting an hours-based Vacation entry (extra_hours) into an
absence_period, the Employee-Report must still show `vacation_days > 0` (deckungsgleich
mit den Stunden) — not 0. Same for `sick_leave_days` and `absence_days`.

ROOT CAUSE (verified): `vacation_days()` (`service/src/reporting.rs:118-123`) =
`vacation_hours / hours_per_day()`, where the per-week `GroupedReportHours.vacation_hours`
is built in `hours_per_week` (`reporting.rs:1227-1231`) EXCLUSIVELY from
`filtered_extra_hours_list` (extra_hours category Vacation). The absence_period-derived
hours of the week (`derived_absence`) feed only `derived_absence_hours -> absence_hours`
(reporting.rs:1139-1148, the expected/balance reduction). They never reach the per-week
category fields. After conversion the extra_hours row is soft-deleted, so per-week
`vacation_hours = 0 -> vacation_days = 0`, even though the top-level display
`vacation_hours` is correct (it adds a year-lump `absence_derived_vacation_hours`,
reporting.rs:719-724).

FIX (D-18-03 + D-18-04): merge the per-week derived absence hours into the per-week
category fields (so days are computed correctly and UNGATED, D-18-05), THEN remove the
top-level year-lump add in `get_report_for_employee` so the derived hours are counted
exactly once (single source of truth = per-week fields summed over by_week).

SNAPSHOT (D-18-07): this changes the computation that produces the persisted
`BillingPeriodValueType::VacationDays` (and SickLeave/UnpaidLeave days). Per CLAUDE.md
§ Snapshot Versioning, BUMP `CURRENT_SNAPSHOT_SCHEMA_VERSION` 9 -> 10 in this commit.

Purpose: correct + regression-proof vacation/sick/absence day counting after conversion.
Output: per-week category merge, de-double-counted employee report, snapshot bump, tests.
</objective>
