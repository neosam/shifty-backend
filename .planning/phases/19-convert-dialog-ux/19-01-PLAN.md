<objective>
Add backend pre-computation of the smart convert-range suggestion for hourly markers.
For each not-yet-converted hourly extra_hours marker, the backend computes:
1. `suggested_end` — the workday-based end date (skipping weekends/holidays and respecting
   the per-week cap) such that the inclusive range `[when, suggested_end]` covers exactly
   `derived_days` active workdays (UV-01).
2. `is_full_week` — `true` iff the marker's `amount` exactly equals the active contract's
   weekly soll (`expected_hours`) within an f32 epsilon (UV-02).

Purpose: The frontend modal needs (a) the weekly-soll exact-match flag and (b) a
workday-aware end date. All three required inputs (contract workday flags, holidays,
weekly soll) live backend-side already (`AbsenceService` deps + `EmployeeWorkDetails`).
Replicating the workday/holiday calendar in the WASM frontend would duplicate the exact
logic that caused the historical `vacation-hours-overcounted` bug — CLAUDE.md and
19-CONTEXT.md (D-19-01) explicitly steer toward backend pre-compute. Decision: FE+BE,
backend owns the calendar math.

Output: Two new fields on `ExtraHoursMarkerTO`, one new `AbsenceService` trait method
with implementation, wiring into both REST list handlers, and backend unit tests.
</objective>
