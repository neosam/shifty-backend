**Konkrete Pattern-Beispiele für die häufigsten Fälle:**

**Pattern P1 — Match-Arm-Erweiterung in `rsx!`-Kontext (UI-SPEC Regel 1):**
```rust
match warning {
    WarningTO::ExistingVariant1 { .. } => rsx! { /* bestehender Code */ },
    WarningTO::ExistingVariant2 { .. } => rsx! { /* bestehender Code */ },
    // Neu (Phase 6, no-op-Render — v1.3 FUI-* renders):
    WarningTO::PaidEmployeeLimitExceeded { .. } => rsx! {},
    WarningTO::NewVariant1 { .. } => rsx! {},
    WarningTO::NewVariant2 { .. } => rsx! {},
}
```

VERBOTEN: `unimplemented!()`, `todo!()`, `panic!()`, `span { class: "hidden", ... }`.

**Pattern P2 — Match-Arm-Erweiterung in non-RSX-Kontext (UI-SPEC Regel 3):**
```rust
match status {
    InvitationStatus::Valid => "valid".to_string(),
    InvitationStatus::Expired => "expired".to_string(),
    InvitationStatus::Redeemed => "redeemed".to_string(),
    // Neu (Phase 6, semantischer Default):
    InvitationStatus::SessionRevoked => "sessionrevoked".to_string(),
}
```

**Pattern P3 — Field-Mirror-Default (UI-SPEC Regel 2):**

Falls eine state-struct `Foo` gegen ein erweitertes `FooTO` mappt und das `FooTO` neue Felder hat:
```rust
// Im state/Foo:
pub struct Foo {
    // ... bestehend ...
    pub new_field: Option<u8>,  // mirrored, kein Render
}

// In From<&FooTO>:
new_field: foo_to.new_field,
```

**Pattern P4 — `state/employee.rs::from_identifier` Panic-Defense (UI-SPEC Regel 3, falls erforderlich):**

NUR anwenden, wenn der Compiler einen Match-Arm erzwingt oder eine andere Wave-Sequenz die Funktion berührt. Falls nicht erzwungen: UNVERÄNDERT lassen.

```rust
// Zeile 89 (state/employee.rs):
// VORHER: _ => panic!("Unknown working hours category: {}", identifier)
// NACHHER (option A — neuer Variant):
//   Erfordert: pub enum WorkingHoursCategory { ..., Unknown(Rc<str>) }
//   Match-Arm: _ => WorkingHoursCategory::Unknown(identifier.into())
//
// NACHHER (option B — bestehenden semantischen Default wählen):
//   Match-Arm: _ => WorkingHoursCategory::ExtraWork  // sicherster Default
```

ENTSCHEIDUNG-CRITERIA für A vs B: Wenn der Compiler keine `unreachable_patterns`-Warnung wirft und kein Aufrufer eine Distinction zu `Unknown` braucht, Option B nehmen (kleinerer Diff). Falls die Funktion downstream zwischen "bekannt" und "unbekannt" unterscheiden muss, Option A.
