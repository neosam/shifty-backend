## Summary

Phase 8.5 ist eine brownfield Mixed-Phase (Backend + Frontend). Sie hat drei unabhängige Bausteine, die in einer klaren Reihenfolge voneinander abhängen: (1) Backend-Kern — neuer `AbsenceConversionService` (BL-Tier) + Backlink-DAO-Umzug + Convert-Endpoint + Read-Projektion-Erweiterung der bestehenden Absence-Read-Endpoints; (2) Frontend-Kern — Absence-Liste um stundenbasierte Marker erweitern + Inline-Stunden-Edit-Modal + HR-Convert-Modal-Aktion; (3) Dialog-Umbau — `add_extra_hours_form.rs` schlanken und Inline-Hinweis einbauen.

Die entscheidende Erkenntnis ist, dass der Code für die Convert-Tx bereits fertig existiert: `service_impl/src/cutover.rs:527-582` (Steps 6/7/8 in `convert_quarantine_entry`) enthält die exakten drei Writes. Der neue Service extrahiert genau diese drei Writes, ohne den umgebenden Heuristik-Ballast (Steps 3/4/9). Die drei Writes sind: `absence_dao.create` (Z. 551-553), `cutover_dao.upsert_migration_source` (Z. 561-570), `extra_hours_service.soft_delete_bulk` (Z. 575-582). Das `ManualConvertModal` aus Phase 8.2 kann 1:1 wiederverwendet werden — es trägt bereits die Halb/Ganz-`DayFractionTO`-Auswahl und die P-7-Submit-Defense.

Die Backlink-DAO-Topologie-Entscheidung (MigrationSourceDao vs. AbsenceDao-Erweiterung) ist zugunsten eines **neuen kleinen `MigrationSourceDao`-Traits** zu treffen: (a) `AbsenceDao` wird nicht durch Backlink-Semantik aufgebläht, (b) Phase 8.6 kann `CutoverDao.upsert_migration_source` subtraktiv löschen und die neue Heimat bleibt stabil, (c) das DB-Schema `absence_period_migration_source` überlebt wie entschieden (D-04). Der `cutover_run_id`-Ballast wird durch eine **additive nullable-Migration** abgelöst — da Prod/INT keine Rows haben, ist das schmerzlos. Phase 8.6 kann die Spalte dann per Drop-Migration entfernen.

**Primary recommendation:** Backend in Wave 1 (Service+DAO+Endpoint+Projektion) → Frontend Wave 2 (AbsenceList-Erweiterung + Modals) → Frontend Wave 3 (Dialog-Umbau + i18n-Vollständigkeit + Gates). Kein CURRENT_SNAPSHOT_SCHEMA_VERSION-Bump.

### Pattern 3: Read-Projektion auf bestehenden Absence-Endpoints

**Ziel-Shape (Discretion-Entscheidung):** Separates Feld auf einer neuen Wrapper-Response ist der sauberste Weg — vermeidet Typ-Vermischung, hält `AbsencePeriodTO` stabil:

```rust
// rest-types/src/lib.rs — NEUES TO
#[derive(Serialize, Deserialize, Clone, Debug, ToSchema)]
pub struct ExtraHoursMarkerTO {
    pub extra_hours_id: Uuid,
    pub sales_person_id: Uuid,
    pub when: time::Date,        // aus date_time.date()
    pub amount: f32,
    pub category: ExtraHoursCategoryTO,
    pub description: Arc<str>,
    // Nur in HR-Response: optional person_name für Anzeige
    #[serde(default)]
    pub person_name: Arc<str>,
}

#[derive(Serialize, Deserialize, Clone, Debug, ToSchema)]
pub struct AbsenceListWithProjectionTO {
    pub absence_periods: Vec<AbsencePeriodTO>,
    pub hourly_markers: Vec<ExtraHoursMarkerTO>,
}
```

**Warum separates Wrapper-TO statt unified list:** Das Frontend muss beide Typen unterschiedlich rendern (Range vs. Tages-Marker); typisierte Unterscheidung ist sicherer als ein `kind`-Discriminant-String. [ASSUMED — alternative wäre ein unified enum-TO, aber das erfordert serde-flatten-Ergonomie]

**REST-Handler-Erweiterung:**
- `get_all_absence_periods` → return type wechselt von `Vec<AbsencePeriodTO>` zu `AbsenceListWithProjectionTO` — **BREAKING für bestehende Frontend-Callers** (api.rs `list_absence_periods` muss angepasst werden).
- `get_absence_periods_for_sales_person` → analog.
- Lade-Call: `extra_hours_service.find_by_sales_person_id_and_year_range(sp_id, from, to, Auth::Full, tx)` — Range-Bound: Caller muss sinnvolle Bounds wählen. Empfehlung: laufendes Jahr + letztes Jahr (zwei-Jahres-Fenster), konsistent mit wie Absence-Reports gebaut werden. [ASSUMED — exakte Bound-Logik ist Planner-Detail]

**Kategorie-Filter:** Nur Vacation, SickLeave, UnpaidLeave — identisch zur Cutover-Semantik (`extra_hours_category_to_absence`-Funktion, cutover.rs:1470-1479).

## Common Pitfalls

### Pitfall 1: VacationDays-Match-Arm im `add_extra_hours_form.rs` Submit-Handler
**What goes wrong:** `WorkingHoursCategory::VacationDays` ist eine Rust-Enum-Variante. Wenn der Branch entfernt wird aber die Variante noch in der Select-Option auftaucht, kann der Nutzer sie auswählen und der Submit-Handler schickt dann ggf. ein leeres Extra-Hours-Formular. Außerdem bleibt `vacation_days_str` als totes i18n-Key-Reference übrig.
**How to avoid:** (a) VacationDays aus dem `<select>`-RSX entfernen (der Kategorie-Dropdown), (b) toten `let vacation_days_str`-Binding entfernen, (c) `from`/`to` use_signal entfernen wenn keine anderen Stellen sie lesen.
**Warning signs:** Compile-Warning `unused variable` für `vacation_days_str`.

### Pitfall 2: Locale::En statt Locale::De in de.rs (historischer Bug)
**What goes wrong:** i18n-Keys für neue Strings werden in `de.rs` mit `Locale::En` eingetragen statt `Locale::De`. Tests für `Key::AbsencePageTitle` etc. fangen das ab, aber neue Keys haben keinen Ref-Matcher bis einer angelegt wird.
**How to avoid:** Per-Locale-Reference-Matcher-Tests für neue Keys anlegen (Plan 08-04/08-05-Pattern — `i18n_*_match_german_reference`). Mindestens 3-4 neue Keys stichprobenartig testen.
**Warning signs:** `i18n.t(Key::SomeNewKey)` gibt en-String zurück wenn Locale::De gesetzt ist.

### Pitfall 3: Breaking-Change auf bestehende api.rs-Callers bei Response-Shape-Änderung
**What goes wrong:** `list_absence_periods` und `list_absence_periods_by_sales_person` in `api.rs` erwarten heute `Vec<AbsencePeriodTO>`. Wenn der Endpoint auf `AbsenceListWithProjectionTO` wechselt, breaken alle Callers.
**How to avoid:** `api.rs`-Funktionen in der gleichen Wave wie die REST-Handler-Änderung anpassen. Service-Coroutine in `service/absence.rs` auf den neuen Typ migrieren. SSR-Snapshot-Tests für `AbsenceList` müssen Props-Typ-Änderung mitmachen.
**Warning signs:** Compile-Error in `shifty-dioxus` nach Backend-Änderung.

### Pitfall 4: `cutover_run_id NOT NULL`-Constraint beim Backlink-Write
**What goes wrong:** Die aktuelle `absence_period_migration_source`-Tabelle hat `cutover_run_id BLOB(16) NOT NULL`. Wenn der neue `AbsenceConversionService` `Uuid::nil()` schreibt, funktioniert das technisch (nil ≠ NULL), aber ist semantisch falsch.
**How to avoid:** Migration anlegen, die die Tabelle neu erstellt ohne `cutover_run_id` oder mit nullable `context TEXT`. Da Prod/INT keine Rows haben, ist DROP+RECREATE-Strategie in der Migration safe.
**Warning signs:** Kein Compile-Error (Uuid::nil ist ein valider Uuid), aber die Tabelle trägt toten Ballast.

### Pitfall 5: ExtraHoursService.soft_delete_bulk permission bypass
**What goes wrong:** `soft_delete_bulk` hat in seiner Docstring: "Bypasses per-row permission checks: caller MUST hold `cutover_admin`" — die alte Prämisse aus Phase 4. Im neuen Service ist der Caller HR (nicht cutover_admin). Die aktuelle Impl in `service_impl/src/extra_hours.rs` muss auf die actual Permission-Handling-Semantik geprüft werden.
**How to avoid:** Prüfen ob `soft_delete_bulk` tatsächlich eine cutover_admin-Prüfung hat oder ob `Authentication::Full` bypasst. Falls ja, den HR-Caller über `Authentication::Full` (internal call after hr-check im REST-Handler) übergeben — analog zur Pattern in `compute_forward_warnings` (`absence.rs:219-230`).
**Warning signs:** 403-Fehler bei HR-Convert trotz korrektem Privileg.

### Pitfall 6: ManualConvertModal Prop-Mismatch beim Reuse
**What goes wrong:** `ManualConvertModal` in `cutover_admin.rs:924` hat als erstes Prop `entry: EntryRef` (ein `Arc<DriftEntryRowData>`-Newtype). Das ist cutover-spezifisch.
**How to avoid:** Beim Reuse auf der Absence-Seite: ManualConvertModal entweder (a) extrahieren nach `component/manual_convert_modal.rs` mit generalisierten Props (extra_hours_id: Uuid, initial_date: time::Date, amount: f32, category: AbsenceCategoryTO), oder (b) das Modal inline in `absences.rs` neu implementieren — gleiche Logik, allgemeinere Props. Option (a) ist sauberer für Phase 8.6 (Cutover-Page-Entfernung).
**Warning signs:** Compile-Error wenn `EntryRef`/`DriftEntryRowData` nicht im Absence-Page-Kontext verfügbar ist.

### Pitfall 7: Year-Range bei Read-Projektion zu eng/weit
**What goes wrong:** `ExtraHoursService::find_by_sales_person_id_and_year` hat ein `until_week`-Parameter und ist auf ein Kalenderjahr beschränkt. Bei der Read-Projektion auf dem Absence-Endpoint müsste über mehrere Jahre geladen werden (HR-View sieht alle Absence-Periods beliebiger Jahre).
**How to avoid:** Stattdessen `find_by_sales_person_id_and_year_range` nutzen (Signatur: `sp_id, from_date, to_date, context, tx`) — diese Variante hat keine Jahr-Beschränkung. Die Bounds können aus dem Bereich der vorhandenen Absence-Periods abgeleitet werden, oder mit einem pragmatischen Fenster (z.B. aktives Jahr ± 1).
