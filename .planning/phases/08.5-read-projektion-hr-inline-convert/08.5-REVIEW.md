# Phase 08.5: Code Review Report

**Reviewed:** 2026-06-09T20:12:14Z
**Depth:** standard
**Files Reviewed:** 23
**Status:** issues_found

## Summary

Die Phase fügt einen schlanken `AbsenceConversionService`, einen hr-gateten Convert-Endpoint, eine Read-Projektion lebender Stunden-Marker und die zugehörige Frontend-Verdrahtung hinzu. Permission-Gating (hr vor Tx, hr∨self auf der per-sales-person-Marker-Route), Overlap-Check-Signatur, die Migration (DROP+RECREATE bei 0 Rows) und die i18n-Abdeckung über alle drei Locales sind korrekt umgesetzt. Modul-Registrierung und DI-Wiring sind vollständig.

Der zentrale Befund ist ein **BLOCKER in der Soft-Delete-Atomarität**: Der Convert-Service löscht die Quell-`extra_hours`-Row über deren **logical_id** statt über die **physische `id`**. Da `extra_hours::update` ein versioniertes Schreibmodell verwendet (neue physische `id` bei gleichbleibender `logical_id`), trifft die Soft-Delete bei jeder zuvor aktualisierten Row keine Zeile — die `absence_period` und der Backlink entstehen, aber die lebende Stunden-Row überlebt. Resultat: sichtbare Doppelzählung (Marker + Range) und Reporting-Drift. Die vorhandenen Tests verschleiern den Fehler (Mockall ignoriert die übergebenen IDs; die Fixtures setzen `id == logical_id`).

Zusätzlich: fünf WARNINGs (verlorenes Fehler-Feedback im Convert-Modal, Datenverlust der `description` beim Convert, falscher `created`-Zeitstempel, Read-Projektion ignoriert den Backlink-Filter, schwache Test-Abdeckung der `id != logical_id`-Konstellation) und vier Info-Punkte.

## Critical Issues

### CR-01: Convert soft-deletet die Quell-Row über logical_id statt physischer id → Doppelzählung

**File:** `service_impl/src/absence_conversion.rs:162-171`
**Issue:**
Der Service lädt die aktive Row korrekt via `find_by_logical_id(extra_hours_id, …)` (Zeile 94-102), übergibt aber für die Soft-Delete denselben `extra_hours_id` (= logical_id) weiter:

```rust
self.extra_hours_service
    .soft_delete_bulk(
        Arc::from([extra_hours_id]),   // <-- logical_id
        "absence_conversion::convert",
        Authentication::Full,
        Some(tx.clone()),
    )
    .await?;
```

`ExtraHoursDaoImpl::soft_delete_bulk` löscht jedoch über `WHERE deleted IS NULL AND id IN (...)` — also über die **physische `id`**, nicht `logical_id`. `extra_hours::update` (`service_impl/src/extra_hours.rs`, Zeile ~280) erzeugt bei jedem Update eine **neue physische `id`** und behält die `logical_id`. Für jede Row, die mindestens einmal aktualisiert wurde, gilt damit `id != logical_id`.

Konsequenz für solche Rows:
- `absence_dao.create` schreibt die neue Range (✓)
- `migration_source_dao.upsert` schreibt den Backlink mit `extra_hours_id` = logical_id (✓)
- `soft_delete_bulk` findet **keine** Row mit `id == logical_id` → die lebende `extra_hours`-Row bleibt bestehen

Da die Read-Projektion (`rest/src/absence.rs`) lebende Rows allein über `deleted IS NULL` lädt (siehe CR-Querverweis WR-04), erscheint die Row danach **gleichzeitig** als Stunden-Marker und als Range → Doppelzählung in Vacation-Balance/Reporting und in der HR-Liste.

Im cutover-Original (`service_impl/src/cutover.rs:750/756`, Bulk-Pfad) wurde bewusst `row.id` (physische id) für die Soft-Delete genutzt; die dortige `id == logical_id`-Annahme galt nur, weil Legacy-Rows nie über den Service aktualisiert wurden. Diese Annahme trägt im neuen per-Row-Convert nicht.

Die Tests fangen den Fehler nicht: Der Mockall-Test (`service_impl/src/test/absence_conversion.rs:173-176`) prüft nur `.times(1)`, nicht die übergebenen IDs; Unit- und Integration-Fixtures setzen durchweg `id == logical_id` (Zeile 45-46 bzw. seed mit identischem Wert für `id`/`logical_id`).

**Fix:** Die geladene Entity trägt die physische ID. Diese verwenden:

```rust
// entity stammt aus find_by_logical_id(extra_hours_id, …)
self.extra_hours_service
    .soft_delete_bulk(
        Arc::from([entity.id]),   // physische id der aktiven Row
        "absence_conversion::convert",
        Authentication::Full,
        Some(tx.clone()),
    )
    .await?;
```

Zusätzlich einen Regressionstest ergänzen, der eine Row mit `id != logical_id` seeded (z. B. via `extra_hours_service.update` vor dem Convert) und nach dem Convert asserted, dass die Row tatsächlich `deleted IS NOT NULL` trägt.

## Warnings

### WR-01: Convert-Modal verschluckt 422/Network-Feedback — Modal wird vor dem Ergebnis geschlossen

**File:** `shifty-dioxus/src/page/absences.rs:1821-1830`
**Issue:**
Im `on_submit`-Handler des `AbsenceConvertModal` wird unmittelbar nach dem `send` das `convert_target` auf `None` gesetzt — das Modal schließt synchron, bevor die async-Antwort vorliegt:

```rust
on_submit: move |(id, start, end, day_fraction)| {
    absence_service.send(AbsenceAction::ConvertExtraHours { … });
    convert_target.set(None);   // schließt sofort
},
```

Im Fehlerfall schreibt `absence_service` (`shifty-dioxus/src/service/absence.rs:199-205`) das Resultat in `ABSENCE_MODAL_EVENT::Validation(text)` bzw. `::Network(msg)`. Dieses Side-Channel wird aber ausschließlich vom CRUD-`AbsenceModal` konsumiert — das beim Convert gar nicht offen ist. Ein 422 (DateOrderWrong / OverlappingPeriod) erreicht den User somit nie sichtbar; nur der Network-Fall landet zusätzlich im `ERROR_STORE`. Der User sieht ein „verschwundenes" Modal ohne Fehlerhinweis.

**Fix:** Das Convert-Modal geöffnet halten, bis ein Erfolgs- oder Fehler-Event vorliegt, und einen eigenen Fehler-Slot rendern (analog `error_msg` im Modal). Bei Erfolg `convert_target.set(None)`; bei `Validation`/`Network` das Modal mit Fehlertext bestehen lassen. Alternativ ein dediziertes `ABSENCE_CONVERT_EVENT`-Signal einführen, das der Page-Convert-Flow auswertet.

### WR-02: Convert verwirft die ursprüngliche `description`

**File:** `service_impl/src/absence_conversion.rs:137`
**Issue:**
Die erzeugte `AbsencePeriodEntity` setzt `description: Arc::from("")`, obwohl die Quell-`extra_hours`-Row eine `description` trägt (im Integrationstest z. B. `"Urlaub 2026-04-10"`). Beim Convert geht dieser Kontext ersatzlos verloren — für SickLeave/UnpaidLeave kann das eine fachlich relevante Notiz sein.

**Fix:** `description: entity.description.clone()` übernehmen (ggf. trimmen/begrenzen), damit der HR-Kontext erhalten bleibt.

### WR-03: `created`-Zeitstempel der Absence wird auf `migrated_at` (jetzt) gesetzt statt aus der Quell-Row übernommen

**File:** `service_impl/src/absence_conversion.rs:90-91, 138`
**Issue:**
`created: migrated_at` stempelt das Convert-Datum als Anlage-Datum der neuen `absence_period`. Für einen reinen Daten-Umzug ist das vertretbar, weicht aber von der Semantik anderer create-Pfade ab (dort markiert `created` den fachlichen Anlagezeitpunkt). Da der Convert „write-once" ist und ein späterer Validator/Audit das `created` interpretieren könnte, sollte die Wahl bewusst und dokumentiert sein.

**Fix:** Entweder `entity.created` (Quell-Anlagedatum) übernehmen, oder den aktuellen Stempel beibehalten und im Code/Doc-Kommentar explizit begründen, dass `created == migrated_at` gewollt ist.

### WR-04: Read-Projektion ignoriert den `absence_period_migration_source`-Backlink

**File:** `rest/src/absence.rs:160-178, 329-343`
**Issue:**
Die Marker-Projektion lädt lebende Rows über `find_by_sales_person_id_and_year_range`, das ausschließlich `deleted IS NULL` filtert. Sie berücksichtigt **nicht** den Backlink-Filter `id NOT IN (SELECT extra_hours_id FROM absence_period_migration_source)`, den der cutover-Reader (`dao_impl_sqlite/src/cutover.rs:79-81`) bewusst anwendet.

Solange die Soft-Delete im Convert atomar erfolgt, fällt das nicht auf (die Row ist dann `deleted`). In Kombination mit CR-01 — oder falls eine Soft-Delete je teilweise fehlschlägt — wird eine bereits konvertierte (Backlink vorhandene) Row trotzdem als Marker angezeigt → Doppelzählung. Der Backlink-Filter wäre eine zweite, unabhängige Verteidigungslinie.

**Fix:** Primär CR-01 fixen. Zusätzlich defensiv die Marker, deren `extra_hours_id` in `absence_period_migration_source` steht, aus der Projektion filtern (z. B. via `MigrationSourceDao::find_by_extra_hours_id` oder einen Batch-Lookup), damit ein Backlink-vorhanden-aber-nicht-deleted-Zustand nicht doppelt zählt.

### WR-05: Tests verifizieren weder die übergebene Soft-Delete-ID noch die `id != logical_id`-Konstellation

**File:** `service_impl/src/test/absence_conversion.rs:173-176, 483-589`
**Issue:**
Der Mockall-Happy-Path prüft `soft_delete_bulk` nur mit `.times(1)` und ignoriert die Argumente. Der Integrationstest seeded `id == logical_id` und ersetzt `soft_delete_bulk` durch einen `Ok(())`-Stub, der nichts gegen die DB tut. Damit ist der einzige sicherheitskritische Schritt (welche Row wird tatsächlich gelöscht) faktisch ungetestet — genau die Lücke, durch die CR-01 unbemerkt blieb. Das verletzt die Projekt-Regel „Always make sure you have tests for the changes" für den kritischen Pfad.

**Fix:** (a) Im Mockall-Test `withf(...)` auf `soft_delete_bulk` legen, das die erwartete physische ID asserted. (b) Einen End-to-End-Integrationstest ergänzen, der eine via `update` versionierte Row (`id != logical_id`) konvertiert und danach per SQL prüft, dass genau die aktive physische Row `deleted IS NOT NULL` trägt und kein lebender Marker mehr zurückkommt.

## Info

### IN-01: N+1-Marker-Loads im HR-`GET /absence-period`

**File:** `rest/src/absence.rs:162-178`
**Issue:** Pro SalesPerson wird ein separater `find_by_sales_person_id_and_year_range`-Aufruf abgesetzt (Schleife über `persons`). Bei vielen Mitarbeitern entsteht eine N+1-Last. Performance ist in v1 explizit out-of-scope; hier nur als Hinweis, falls die Liste wächst.
**Fix:** Optional einen Batch-Reader (`find_living_absence_markers_in_range`) ergänzen, der alle relevanten Rows in einem Query liefert.

### IN-02: Ungenutztes `_pool`-Feld in `MigrationSourceDaoImpl`

**File:** `dao_impl_sqlite/src/migration_source.rs:36-43`
**Issue:** `MigrationSourceDaoImpl` hält `_pool: Arc<Pool<Sqlite>>`, das nirgends gelesen wird (alle Queries laufen über die übergebene `tx`). Konsistent mit `CutoverDaoImpl._pool`, daher bewusst — der Unterstrich unterdrückt die Warnung. Kein Bug, nur toter Zustand.
**Fix:** Optional `new()` auf `()`-Konstruktion ohne Pool-Feld reduzieren, falls der Pool wirklich nie gebraucht wird.

### IN-03: Hartkodierter Fallback-String "Invalid date format" nicht i18n-lokalisiert

**File:** `shifty-dioxus/src/component/absence_convert_modal.rs:177-179`
**Issue:** Der Parse-Fehlerzweig setzt eine englische, nicht übersetzte Meldung, während alle anderen Modal-Texte über `i18n.t(...)` laufen. In De/Cs-Locales erscheint hier inkonsistentes Englisch. Tritt nur bei manipuliertem/leerem date-Input auf (selten).
**Fix:** Einen i18n-Key (z. B. `CutoverManualConvertErrInvalidDate`) für alle drei Locales ergänzen und hier verwenden.

### IN-04: `AbsencePeriodTO::version` mit `#[serde(rename = "$version")]` — Convert-Response-Konvention prüfen

**File:** `rest/src/extra_hours.rs:240`, `rest-types/src/lib.rs:1641-1643`
**Issue:** Der Convert-Endpoint gibt `AbsencePeriodTO` mit `$version` zurück. Der Frontend-Convert-Pfad (`api.rs:536`) deserialisiert das Ergebnis, verwirft es aber (`Ok(_result)` in `service/absence.rs:195`) und lädt stattdessen die Liste neu. Funktional unkritisch; nur Hinweis, dass die `$version`-Konvention hier ungenutzt durchläuft und keine Regression-Absicherung trägt.
**Fix:** Kein Handlungsbedarf; bei späterer Nutzung des Rückgabewerts sicherstellen, dass `$version` korrekt gemappt wird (bereits durch die bestehende `From`-Impl gegeben).
