## Component Contracts

### 1. BookingWarningDialog

The dialog is assembled using the existing `component/dialog.rs` `Dialog` primitive with `DialogVariant::Auto` (Bottom on mobile, Center on desktop).

**Dialog props:**
- `open`: signal driven by `pending_warnings.read().is_empty() == false`
- `variant`: `DialogVariant::Auto`
- `title`: i18n-resolved header string (Singular or Plural, see Copywriting Contract)
- `on_close`: routes to `ShiftPlanAction::RollbackBooking(booking_id)` — ALL close paths (X button, backdrop click, ESC) must dispatch rollback. "Trotzdem buchen" clears `pending_rollback_id` BEFORE the dialog closes.

**Dialog body:**
- `WarningList` component (moved from `absences.rs` to `component/warning_list.rs`) with `dense = false` (or without the prop if executor drops it — see Spacing section)
- `person_name: Option<ImStr>` prop set to the booked person's display name from `sales_persons_resource`

**Dialog footer (two buttons, left-to-right):**
1. Secondary button — `BtnVariant::Secondary`, label from `Key::BookingWarningDialogCancel`, dispatches `ShiftPlanAction::RollbackBooking(booking_id)` on click
2. Primary button — `BtnVariant::Primary`, label from `Key::BookingWarningDialogConfirm`, clears `pending_rollback_id`, closes dialog, calls `update_shiftplan()` on click

**Width:** default dialog width (existing `Dialog` default — no override needed).

**Animation:** inherits existing `shifty-modal-pop` (Center) / `shifty-modal-slide-up` (Bottom) from `dialog.rs` — no changes.

### 2. WarningList (shared component — moved to `component/warning_list.rs`)

Visually identical to the current `WarningList` in `absences.rs:197–244`, with container padding normalised to `p-4` (16px). The move must preserve the exact Tailwind classes:

**Container:**
```
border-l-[3px] border-warn bg-warn-soft rounded-md p-4 flex flex-col gap-2
```

The padding is `p-4` (16px) on all paths. The previously used `p-2.5` (10px) and `p-3` (12px) values from `absences.rs` are normalised to `p-4` as part of extracting the component (D-09) — this aligns the warning box inner padding with the dialog panel body padding for visual consistency. If the executor determines both dense and non-dense paths are now identical, the `dense` prop may be dropped entirely.

**Header row:**
```
text-micro text-warn font-semibold uppercase
```
Text: Singular `"Hinweis · 1 Konflikt (nicht blockierend)"` / Plural `"Hinweis · {count} Konflikte (nicht blockierend)"` — using the existing keys `AbsenceWarningHeaderSingular`/`AbsenceWarningHeaderPlural` (unchanged for absence-path callers) OR the new booking-dialog-specific header keys (see Copywriting Contract).

**Item list:**
```
list-disc pl-4 text-body text-ink
```

**New match arms (booking-path variants) — exact text templates:**

| Variant | Template | Notes |
|---------|----------|-------|
| `BookingOnAbsenceDay { date, category, .. }` | `{person_name} ist am {date} als {category} abwesend.` | `date` via `i18n.format_date(date)`; `category` via `AbsenceCategory` i18n key; `person_name` from prop |
| `BookingOnUnavailableDay { year, week, day_of_week, .. }` | `{person_name} ist in KW {week}/{year} ({day}) als nicht verfügbar markiert.` | `day_of_week` mapped to weekday i18n key; no `time::Date` reconstruction |
| `PaidEmployeeLimitExceeded { current_paid_count, max_paid_employees, .. }` | `Bezahlt-Limit überschritten ({current}/{max}).` | No `person_name` needed (slot-level warning); `current_paid_count` + `max_paid_employees` as numeric interpolations |

The existing two absence-path variants (`AbsenceOverlapsBooking`, `AbsenceOverlapsManualUnavailable`) remain unchanged.

**`person_name` prop:**
- Type: `Option<ImStr>`
- Default: `None`
- Absence-path callers (no single person context) pass `None`; booking-dialog callers pass the booked person's name
- If `None` and a booking-path variant is rendered, fall back to a neutral dash `"–"` for the person placeholder rather than omitting the field

## Interaction Contract

### Dialog Open / Close Lifecycle

```
State machine for booking_warning_dialog:

CLOSED
  → [AddUserToSlot: warnings non-empty] → OPEN (pending_rollback_id set, pending_warnings set)

OPEN
  → [X button / ESC / backdrop] → dispatch RollbackBooking(id) → CLOSED (pending cleared)
  → ["Abbrechen"] → dispatch RollbackBooking(id) → CLOSED (pending cleared)
  → ["Trotzdem buchen"] → clear pending_rollback_id → close → update_shiftplan() → CLOSED
```

**Critical invariant:** `pending_rollback_id` must be passed as a parameter of `RollbackBooking(booking_id: Uuid)` — NOT read from a signal inside the handler (Pitfall 5 guard). See RESEARCH.md Pitfall 5.

**Rollback failure path (D-04):**
- DELETE returns error → call `error_handler(e)` (surfaced as toast) AND call `update_shiftplan()`. Never silent-swallow.

**403 on initial POST (D-13):**
- Silent-swallow — no dialog opened, no toast. Consistent with existing `AddUserToSlot` 403 pattern.

**422 on initial POST (D-13):**
- Surface via `error_handler`. No dialog opened.

### Accessibility

- Dialog uses `component/dialog.rs` which provides ESC dismissal and backdrop click to close — both must dispatch rollback (same as X button).
- Focus trap: handled by existing Dialog primitive.
- No new ARIA attributes needed beyond what the Dialog primitive already provides.

## States and Conditions

| State | Visual Outcome |
|-------|---------------|
| Booking POST returns `warnings: []` | No dialog; `update_shiftplan()` called immediately |
| Booking POST returns `warnings: [one item]` | Dialog opens; title uses `BookingWarningDialogHeaderSingular` |
| Booking POST returns `warnings: [N items, N>1]` | Dialog opens; title uses `BookingWarningDialogHeaderPlural` with `{count}` = N |
| Rollback DELETE succeeds | Dialog closes; `update_shiftplan()` called; no toast |
| Rollback DELETE fails | `error_handler` toast shown; `update_shiftplan()` still called; dialog closes |
| 403 on booking POST | No dialog; no toast; silent |
| 422 on booking POST | No dialog; `error_handler` toast shown |
