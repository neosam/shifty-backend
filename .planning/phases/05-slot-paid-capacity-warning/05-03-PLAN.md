Wire `max_paid_employees: Option<u8>` through the basic-tier `SlotService`: domain DTO `Slot`, both `From` impls (entity ↔ domain), the `SlotServiceImpl::create_slot` / `update_slot` flows, and **all `Slot { ... }` / `SlotEntity { ... }` literals in 5 test files** (`test/slot.rs`, `test/booking.rs`, `test/block.rs`, `test/absence.rs`, `test/shiftplan_edit.rs`) so the test crate compiles after the new field is added.

Purpose: Bridges the DAO layer (Plan 01) up to the service layer that REST handlers and `ShiftplanEditService` consume. Implements D-02 (entity field), D-10 (Slot DTO carries it — service-tier first, REST mirror in Plan 05), D-11 (update permission already gated by `SHIFTPLANNER_PRIVILEGE` in `update_slot`; new field is mutable in-place, not locked).

**Approach (revision):** `service::slot::Slot` does NOT derive `Default`. Existing test fixtures use full struct literals with no `..spread`. Adding the new field to `Slot` (and to `SlotEntity` via Plan 01) immediately breaks compilation in `service_impl` test code. **Plan 03 owns the fixture migration in 5 test files; Plan 04 (Wave 2, parallel sibling) owns the migration of `service_impl/src/test/shiftplan.rs` because Plan 04 is already modifying that file to add new tests** — same-wave file ownership rule keeps the two plans on disjoint files. The alternative (deriving `Default` on `Slot` + `SlotEntity` and migrating to `..Default::default()`) is bigger surface and changes existing fixture semantics — rejected. The chosen approach: explicit `max_paid_employees: None,` added to every literal site in this plan's owned files. Acceptance gate is `cargo test -p service_impl --lib slot::` exits 0 plus a workspace-wide `cargo build` to catch any E0063 leakage outside the slot tests.

Output: `Slot` carries the field; service tests still green; downstream consumers (`ShiftplanEditService` in Plan 06, `ShiftplanViewService` in Plan 04, REST DTOs in Plan 05) have a populated `Slot.max_paid_employees` to read.
