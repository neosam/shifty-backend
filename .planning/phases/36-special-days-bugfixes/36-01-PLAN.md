<objective>
Fix SDF-01 entirely in the backend: the special-day `create` service path must replace an existing same-date entry in place (update, not a second insert) when a day's type is switched between Feiertag (Holiday) and Kurzer Tag (ShortDay). Today the second POST hits the duplicate guard and returns `ServiceError::ValidationError([Duplicate])` (HTTP 422), so the shift-plan dropdown shows an error and the type never changes.

Purpose: One central, atomic backend fix (per D-01/D-04) makes both entry surfaces (Schichtplan per-day dropdown and Settings Card-3) robust without any frontend change and without a new endpoint (D-02).

Output: An extended DAO `update` that persists `day_type`/`time_of_day`, a `create` service path that replaces the existing same-date row atomically, converted/added service tests, and a regenerated `.sqlx` cache.
</objective>
