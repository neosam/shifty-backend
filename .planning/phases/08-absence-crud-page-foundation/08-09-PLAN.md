<objective>
Das aktuelle Cutover-Modell (v1.0 Phase 4, D-Phase4-01/02) erzwingt strict
1:1-Mapping pro Tag: jede `extra_hours`-Row muss an einem Vertragstag liegen
und exakt `contract.hours_per_day()` Stunden enthalten. Sonst Quarantäne.

Diese Annahme passt nicht zu einer **etablierten Buchungs-Konvention** im
realen Datenstand: Bei Teilzeit-Verträgen wird der Wochenurlaub oft als
**Pauschalbuchung an einem beliebigen Wochentag** eingetragen, statt pro
Vertragstag aufgesplittet.

**Live-Beispiel:** Max Schmidt (3-Tage-Vertrag, 20h/Woche, hours_per_day ≈ 6.67h)
hat 20h Vacation am 2026-05-08 (Freitag) eingetragen. Das ist sein Wochenurlaub,
nicht ein Tageseintrag. Strict-Match wirft Quarantäne (`ContractHoursZeroForDay`),
Cutover-Gate stoppt mit drift = 20h.

**Ziel dieses Plans:** Erweiterte Cutover-Heuristik, die solche
Wochenpauschalen erkennt und korrekt auf einen `absence_period { from: Mo,
to: So }` der entsprechenden Kalenderwoche mappt. Bestehende Strict-Match-Logik
bleibt als Fallback erhalten.
</objective>

<spec>

### Erkennungsregel "Wochenpauschale"

Eine `extra_hours`-Row qualifiziert als Wochenpauschale, wenn ALLE drei
Bedingungen erfüllt sind:

1. **Amount-Match:** `|row.amount − Σ(contract.hours_per_day_for_weekday(d) für jeden Vertragstag d in der ISO-Kalenderwoche der Row)| < CONTRACT_HOURS_EPSILON`
   - In der Praxis bei einem ganzwöchig-aktiven Vertrag: `amount ≈ contract.expected_hours`
   - Bei Vertragswechseln innerhalb der Woche: dynamisch pro Wochentag summieren
2. **Single-Row-pro-Woche:** Keine andere `extra_hours`-Row derselben `(sales_person_id, category)` liegt in derselben ISO-Kalenderwoche (live, nicht-soft-deleted, nicht-bereits-migriert).
3. **Wochentag beliebig:** Eintrag-Datum darf jeder Wochentag sein (Mo/Di/Mi/Do/Fr/Sa/So) — auch Nicht-Vertragstage.

### Mapping

Wenn die Row matcht:
- `absence_period.from_date = montag der ISO-Kalenderwoche der Row`
- `absence_period.to_date = sonntag der ISO-Kalenderwoche der Row`
- Cluster-Builder darf diese Row NICHT mit Vorgänger/Nachfolger verschmelzen — sie ist ein eigenständiger 1-Row-Cluster mit explizit gesetztem Range.
- Quarantine-Liste bleibt **leer** für diese Row (sie ist kein Drift, sondern erfolgreich gemappt).

### Reihenfolge im Algorithmus

Im `migration_phase`-Loop (`service_impl/src/cutover.rs:485+`):

```
für jede legacy_row:
  1. Pre-check: ist row eine Wochenpauschale (Erkennungsregel oben)?
     → JA: bilde Cluster-of-1 mit { from: Mo-der-Woche, to: So-der-Woche },
            close_current_cluster, push to migrations, continue
     → NEIN: weiter mit existing logic (Strict-Match)

  2. (existing) Workday-check: ContractHoursZeroForDay quarantine
  3. (existing) Strict-Match: AmountBelow/AmountAbove quarantine
  4. (existing) Cluster-Building
```

### Backwards-Compat

- `extra_hours`-Rows, die strict-match passen (1 Eintrag = 1 Vertragstag = hours_per_day), funktionieren unverändert.
- Cluster-of-N (mehrere aufeinanderfolgende Vertragstage) funktioniert unverändert.
- Quarantine-Reasons (`AmountBelow/Above`, `ContractHoursZeroForDay`, `ContractNotActiveAtDate`, `Iso53WeekGap`) bleiben unverändert.
- Drift-Threshold (0.01) und Drift-Berechnung im Gate-Phase unverändert.

### Stunden-Rekonstruktion im Carryover

Das neue Modell rekonstruiert Stunden eines `absence_period` durch Range-Iteration
über Vertragstage und Aufsummierung von `hours_per_day` pro Vertragstag.
Bei einer Wochenpauschale `{Mo-So}` mit 3-Tage-Vertrag (Mo/Di/Mi) ergibt das:
3 × 6.67h ≈ 20h. → Drift = 0 (legacy_sum 20h, derived_sum 20h). Gate passes.

</spec>

### Task 1 — Wochenpauschalen-Erkennung implementieren

**File:** `service_impl/src/cutover.rs` (Migration-Phase, ~Line 480+)

1. Helper-Function in `service_impl/src/cutover.rs`:
   ```rust
   /// Returns Some(week_range) if this row qualifies as a weekly lump-sum
   /// per Plan 08-09 spec (single row per (sp, cat, ISO-week) where amount
   /// matches Σ(contract.hours_per_day for each contract weekday in that ISO-week)).
   /// Returns None otherwise — caller falls back to strict-match logic.
   fn detect_weekly_lump_sum(
       row: &LegacyExtraHoursRow,
       all_rows_for_sp_cat: &[LegacyExtraHoursRow],
       contract_at: impl Fn(time::Date) -> Option<&EmployeeWorkDetails>,
   ) -> Option<(time::Date, time::Date)> { ... }
   ```
   - Bestimmt ISO-Woche von `row.date_time`
   - Filtert `all_rows_for_sp_cat` auf andere Rows in derselben ISO-Woche → wenn ≥1 anderer: None
   - Iteriert Mo-So der Woche, ruft `contract_at(weekday)` für jeden Tag, summiert `hours_per_day` falls Vertragstag und Vertrag aktiv
   - Vergleicht Summe gegen `row.amount` (Epsilon CONTRACT_HOURS_EPSILON)
   - Bei Match: returns `Some((monday, sunday))`

2. Integration in den Migration-Loop:
   - Vor dem `(b) workday check`: `if let Some((from, to)) = detect_weekly_lump_sum(...)` → bilde 1-Row-Cluster mit explizit gesetzten {from, to}, `close_current_cluster`, `migrations.push(...)`, `continue`.
   - Cluster-Builder muss den 1-Row-Cluster nicht mit Vorgänger/Nachfolger verschmelzen — daher `close_current_cluster()` davor und ein direkt-gepushter Cluster-of-1 (oder ein neuer Helfer `push_explicit_range_cluster(...)`).

3. Cluster-Strukturen prüfen: ggf. existing Cluster trägt nur die rows, das `from/to_date` wird beim Migration-zu-AbsencePeriod-Übergang berechnet (z.B. `cluster.first().date` bis `cluster.last().date`). Für Wochenpauschalen müssen wir das überschreiben — entweder durch ein optionales `Option<(from, to)>` Override-Feld am Cluster, oder durch eine separate "WeeklyLumpSumCluster"-Variante.
