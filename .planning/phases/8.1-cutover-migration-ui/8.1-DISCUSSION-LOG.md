## Convert-Atomarität & Backend-Endpoints

### Q1 — Wie soll 'Convert single quarantine entry → AbsencePeriod' atomic-tx-mäßig laufen?

| Option | Description | Selected |
|--------|-------------|----------|
| Neuer Backend-Endpoint, atomar | POST /admin/cutover/convert-quarantine-entry mit {extra_hours_id, target_range_from, target_range_to, category}. Single Tx. | |
| Frontend-Sequenz, DELETE + POST | Frontend ruft DELETE /extra-hours/{id} dann POST /absence-period. NICHT atomar. | |

## Feiertag-Heuristik-Fix (Plan 08-09 gap (c))

### Q1 (initial) — Wie wird der Feiertag-Konsistenz-Fix gehandhabt?

| Option | Description | Selected |
|--------|-------------|----------|
| In 8.1 mitnehmen als eigener Backend-Plan | detect_weekly_lump_sum-Erweiterung mit SpecialDayService-Lookup. | |
| In 8.1 NICHT — Operator löst manuell | UI bietet Edit / Convert; Plan 08-09 dokumentiert Heuristik-Limit. | |
| Out-of-Scope in eigene Phase 8.2 | Phase 8.1 nur Frontend; 8.2 = Heuristik-Polish. | |

**User's choice (free-text):** "Ich bin mir nicht sicher, ob das notwendig ist. Feuertage werden zusätzlich zum Urlaub dazugerechnet. Das sind oft Bruchteile eines Arbeitstages."

**Notes:** Domain-Insight des Users: Feiertage sind orthogonal zu Vacation verbucht (über SpecialDayService); User trägt Vacation-Pauschale ein, ohne Holidays abzuziehen. Code-Verifikation: `derive_hours_for_range` skipt Holidays explizit (`service_impl/src/absence.rs:483-485`). Konsequenz: Ein Heuristik-Fix, der Holidays IGNORIERT, würde permanente Drift in Reports erzeugen. Daher Reflection-Frage gestellt.

### Q1b (reflection) — Mit dem Code-Befund: Heuristik-Fix in 8.1 — was ist die richtige Strategie?

| Option | Description | Selected |
|--------|-------------|----------|

| Doch automatischer Fix mit Amount-Override | Heuristik schreibt amount=derived auf die absence_period. | |

**User's choice:** KEIN automatischer Heuristik-Fix — Operator löst via Edit/Convert.
