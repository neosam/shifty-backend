# Phase 8.1: Cutover-Migration-UI - Context

**Gathered:** 2026-05-08
**Status:** Ready for planning

<domain>
## Phase Boundary

Admin-Frontend in `shifty-backend/shifty-dioxus/` für die `extra_hours` → `absence_period`-Datenmigration. 3-Stage-Stepper-Wizard (Profile → Dry-Run → Commit) mit Drift-Resolution-Liste, atomaren Per-Eintrag-Aktionen (Delete / Edit / Convert / Skip) und atomaren Bulk-Aktionen pro `(sales_person, category, year)`-Gruppe. Ergänzt um zwei NEUE Backend-Endpoints (Single-Convert + Bulk-Convert), die jeweils in einer Tx `extra_hours`-Soft-Delete + `absence_period`-Insert atomar zusammenführen. Closure-Phase für Phase 8 — entsperrt den dort deferred 35-Schritt-HUMAN-UAT (`08-HUMAN-UAT.md` gap-1) durch Operator-getriebene Auflösung der int-Drift-Patterns, die die Plan-08-09-Auto-Heuristik nicht abdeckt.

**Misch-Phase Backend + Frontend** (analog Phase 8). Backend-Erweiterung ist additive Cutover-Surface; kein Schema-Änderung an `extra_hours` / `absence_period`.

**In Scope:**
- **Backend:** `POST /admin/cutover/convert-quarantine-entry` (Single-Convert, atomic Tx, Range-Auto-Compute via `detect_weekly_lump_sum`-Logik); `POST /admin/cutover/bulk-convert-quarantine-rows` (Bulk-Convert per `(sales_person_id, category, year)`-Scope, single Tx); beide mit `cutover_admin`-Privileg, OpenAPI-Annotations, `ToSchema` auf neuen DTOs, OpenAPI-Surface-Test-Eintrag.
- **Frontend:** Route `/admin/cutover` mit 3-Stage-Stepper; Stage 2 zeigt Drift-Liste gegliedert nach `(person, category, year)` mit Inline-Action-Buttons (Convert / Edit / Delete / Skip) + Per-Group-Bulk-Convert; Stage 3 mit Type-to-confirm "CUTOVER" + Migration-Summary; Auto-Re-Run von gate-dry-run nach jeder Aktion; Erfolgs-Page mit Idempotenz-Hinweis; Permanent-Banner-State bei Re-Open nach Commit.
- **Inline-Edit-Modal:** Re-uses existing `PUT /extra-hours/{id}`; editierbar sind nur `amount` und `date` (nicht `category`, nicht `description`).
- **Diagnose-Plan für gap-1 (a):** Reproduce der Vertragsdaten-Edge-Case-Drifts (Lila/Anina/Karin) mit Hypothesen-Suche; Fix wenn klar, sonst dokumentiert als bleibender gap.
- **i18n De/En/Cs vollständig** für Page-Chrome (Stage-Labels, Action-Buttons, Confirm-Dialog-Texte). Backend-Reason-Texte aus `QuarantineReason::{human_text,suggested_action}` bleiben Englisch (Plan 08-08 Konvention) — Frontend rendert unverändert.
- **8.1-eigener UAT-Plan** für die Cutover-UI selbst (Wizard-Stages, Drift-Resolution-Aktionen, Bulk-Convert, Type-to-confirm, Post-Commit-State).
- **Phase-8-HUMAN-UAT (35 Schritte) als separater Subsumption-Plan** post-Cutover; setzt `08-HUMAN-UAT.md` gap-1 auf `resolved` und schließt Phase 8 + 8.1 gemeinsam.

**Out of Scope (explizit):**
- Automatischer Backend-Heuristik-Fix für Feiertag-Konsistenz in `detect_weekly_lump_sum` (gap-1 (c)) — würde permanent inkonsistente AbsencePeriod-Daten erzeugen, weil `derive_hours_for_range` Holidays bewusst skipt (`absence.rs:483-485`); Operator löst manuell via Edit/Convert.
- Heuristik-Erweiterung für Teil-Wochen-Pauschalen (gap-1 (b)).
- Audit-Log-Ansicht im UI (Backend-File-Artefakte unter `.planning/migration-backup/cutover-gate-{ts}.json` bleiben File-only).
- Cutover-History-Page (Multi-Run-Sicht).
- Multi-Tenant-Awareness.
- Pre-Cutover-Backup-Integrity-UI (Backend-implizit; UI rendert nur Standard-Error-Toast bei 500).
- Force-Commit-Override / Escape-Hatch unter Umgehung des `quarantined_rows == 0`-Gates.
- Multi-Step-Confirm vor Commit (einzelner Type-to-confirm reicht).
- Falls neue Drifts auftauchen, die UI-Tools nicht lösen → als gap-2 in `08-HUMAN-UAT.md` dokumentiert + Backlog-Plan für 8.2/9.x; 8.1 closed trotzdem.

<decisions>
## Implementation Decisions

### Backend-Erweiterung (8.1 ist Misch-Phase)

- **D-01:** Neuer Endpoint `POST /admin/cutover/convert-quarantine-entry` ist atomic-tx — Body `{ extra_hours_id }`; Backend leitet `{from, to}`-Range automatisch aus der `detect_weekly_lump_sum`-Logik ab; in einer Tx: soft-delete `extra_hours` + insert `absence_period` + return `AbsencePeriodTO`. Idempotent via existing `extra_hours.deleted IS NULL`-Check. Alternative "Range-vom-Frontend-vorgegeben" wurde verworfen — weniger Fehlerquellen, weniger Logik im Frontend.
- **D-02:** Neuer Endpoint `POST /admin/cutover/bulk-convert-quarantine-rows` mit Body `{ sales_person_id, category, year }` (optional zusätzlich explizite `extra_hours_ids: Vec<Uuid>` für Multi-Selection-Erweiterung). Single Tx über alle Rows. Konsumiert die Heuristik aus Plan 08-09 für Range-Auto-Compute pro Row. Atomar — alle migrieren oder keiner.
- **D-03:** Service-Tier-Klassifizierung beider neuen Operationen ist **Business-Logic-Tier** auf `CutoverServiceImpl` (existing) — sie kombinieren `AbsenceService` + (Soft-Delete-Pfad auf) `ExtraHoursService`/-DAO + `WorkingHoursService`. Plan-Phase entscheidet, ob das auf `CutoverServiceImpl` direkt landet oder einen Sub-Service kriegt.
- **D-04:** Inline-Edit nutzt das existing `PUT /extra-hours/{id}`. Editierbar im UI-Modal sind nur `amount` und `date`. `category` und `description` sind im UI nicht editierbar (für Cutover-Resolution-Anwendungsfall ausreichend); falls User sie ändern möchte, geht das via existing Extra-Hours-Page.
- **D-05:** "Skip" ist Frontend-only — Row wird lokal aus der UI-Liste ausgeblendet; kein Backend-Persistenz. Nach Page-Reload ist sie wieder da. Kein neuer DAO / Schema-Field nötig.
- **D-06:** KEIN Backend-Heuristik-Fix für Feiertage in `detect_weekly_lump_sum`. Konsistenz-Argument: `derive_hours_for_range` skipt Holidays explizit (`service_impl/src/absence.rs:483-485`); würde die Heuristik Holidays nicht abziehen, hätten migrierte AbsencePeriods permanent Drift in Reports. Operator löst gap-1 (c) manuell via Edit (Amount korrigieren) oder Convert + Edit.

### Wizard-Topologie

- **D-07:** Stepper mit harter Stage-Reihenfolge: Profile → Dry-Run → Commit. Stage 2 nur erreichbar nach Stage 1 abgeschlossen; Stage 3 nur wenn `quarantined_rows == 0` aus letztem Dry-Run. Linear, klare "wo bin ich gerade"-Visual.
- **D-08:** Auto-Re-Run von `POST /admin/cutover/gate-dry-run` nach jeder Resolve-Aktion (Single-Convert / Bulk-Convert / Edit / Delete). Skip triggert KEINEN Re-Run (Frontend-only). Kostet einen Backend-Roundtrip pro Aktion, dafür sieht der User Liste-schrumpft sofort.
- **D-09:** Loading-State während Re-Run: Inline-Spinner auf Action-Button + Liste-Skeleton (Default-Pattern aus shifty-dioxus). Page bleibt sonst interaktiv, aber Drift-Liste blockiert Klicks bis Re-Run komplett.
- **D-10:** Profile-Stage zeigt strukturierte Stat-Boxen (Aggregat: Total `extra_hours`-Rows zu migrieren, geplante AbsencePeriods, betroffene Personen, Carryover-Diff-Vorschau) + Per-Person-Kacheln (Anzahl Rows + Hours-Summe pro Mitarbeiter). Die Profile-Response-Shape wird in der Plan-Phase finalisiert (vermutlich Erweiterung von `CutoverRunResultTO` oder neuer DTO `CutoverProfileResultTO`).

### Drift-Resolution-Liste

- **D-11:** Drift-Liste ist gegliedert nach `(sales_person_name, category, year)` — 1:1-Mapping zur Backend-`DriftRow`-Struktur. Outer-Sections per Gruppe; innerhalb Quarantine-Entry-Liste + Section-Aktionen.
- **D-12:** Per-Eintrag-Aktionen: Inline-Action-Buttons in der Row am Zeilenende — Convert / Edit / Delete / Skip. Convert + Delete ohne extra Confirm (User hat schon den Wizard-Schritt erreicht, der explizit Resolution erwartet); Edit öffnet Modal; Skip ist instant.
- **D-13:** Bulk-Convert: Per-Group-Section-Action "Alle in dieser Gruppe konvertieren" (1 Click → `POST /bulk-convert-quarantine-rows` für die `(sp, cat, year)`-Triple). Kein separater Multi-Select-State; Group-Struktur aus D-11 ist die Selection.
- **D-14:** Pro Drift-Row sichtbar: ISO-Datum + Wochentag-Code (z.B. "2026-05-08 (Fri)"); Amount + Reason-Code (Hover-Tooltip mit `reason_text`); Suggested-Action als sichtbarer Hint (aus `QuarantineReason::suggested_action()`, hilft dem User die richtige Aktion zu wählen).

### Commit-Sicherheit & Idempotenz-UX

- **D-15:** Commit-Confirm ist Type-to-confirm: Modal verlangt Eingabe des Strings "CUTOVER" + zeigt Migration-Summary (X Rows migriert, Y AbsencePeriods erstellt, Carryover-Diff Z h). Commit-Button erst aktiv nach korrekter Eingabe. Keine Multi-Step-Confirm.
- **D-16:** Nach erfolgreichem Commit: Erfolgs-Page mit Summary-Stats + Idempotenz-Hinweis ("Re-Runs sind no-ops") + Backup-Pfad-Hinweis. Stages werden read-only / disabled. Optional Link "Zur AbsencePeriod-Liste".
- **D-17:** Idempotenz beim Re-Open der UI nach Commit: Page detectet `absence_range_source_active=true` via `GET /feature-flag/{key}` (Plan 08-07-Endpoint); rendert Banner "Cutover bereits abgeschlossen am {date}" oben. Stages funktionieren weiter (Profile + Dry-Run liefern "0 Rows zu migrieren"); Commit-Button permanent disabled mit Tooltip "Bereits durchgeführt". Re-Runs sind backend-seitig no-ops (atomic dedupe per `logical_id`).

### Heuristik-Scope

- **D-18:** gap-1 (b) Teil-Wochen-Pauschalen ("3 von 4 Tagen Urlaub als 15h-Eintrag") — Operator löst manuell via Convert oder Edit (kein automatischer Detection-Patch).
- **D-19:** gap-1 (a) Vertragsdaten-Edge-Case (Lila/Anina/Karin matchen nicht trotz scheinbar passendem Pattern) — Diagnose-Plan in 8.1: Reproduce mit Test-Fixtures, Hypothesen prüfen (mid-week-Vertragswechsel, Hire-Date-Edge-Cases, Inactive-Contract-Tage in `lookup_active_contract`), Fix wenn klar. Falls non-trivial → bleibender gap dokumentiert; Operator löst die betroffenen Rows manuell via UI.

### UAT & Phase-8-Closure

- **D-20:** Phase-8-35-Step-HUMAN-UAT läuft post-Cutover, einmalig nach 8.1-UI-Build: Operator nutzt 8.1-UI für int-Cutover (Profile → Dry-Run → Resolution → Commit) → dann 35-Step-UAT auf migrierten Daten. Kein Pre-Cutover-Run (Phase-8-AbsencePeriod-Page wäre dort leer). UAT pinnt die Funktionalität auf realer Datenbasis.
- **D-21:** Eigener 8.1-UAT-Plan für die Cutover-UI selbst (separat vom Phase-8-UAT) — Wizard-Stages, Drift-Resolution-Aktionen (alle vier Aktionen je einmal), Bulk-Convert auf Group-Section, Type-to-confirm-Verhalten, Idempotenz-State nach Commit. Plus separater Phase-8-UAT-Subsumption-Plan, der die 35 Steps dokumentiert + gap-1 auf `resolved` setzt.
- **D-22:** Falls 8.1-UI auf int neue Drift-Patterns trifft, die Tools (Convert / Edit / Delete) nicht lösen → neues gap-2 in `08-HUMAN-UAT.md` dokumentiert + Backlog-Plan für Phase 8.2 / 9.x. Phase 8.1 closed trotzdem; Phase 8 bleibt `milestone-complete`.

### Privilege-Gate & Routing

- **D-23:** UI sichtbar für `cutover_admin` UND `hr` (beide Privilegien). HR-User sehen die Page und können Profile + Dry-Run nutzen (Diagnose); nur `cutover_admin` darf Commit-Button bedienen. Spiegelt Backend-Permission-Modell 1:1.
- **D-24:** Menü-Eintrag in "Verwaltung"-Submenu (Plan 08-07-Pattern) — neuer Entry "Datenmigration" oder "Cutover". Konsistent mit existing Submenu-Konvention.
- **D-25:** URL-Pfad `/admin/cutover` (1:1 zum Backend-Pfad `/admin/cutover/*`). Klar als Admin-Tool erkennbar; URL nicht-discoverable für Non-Admin.

### i18n

- **D-26:** Page-Chrome (Stage-Labels, Stat-Box-Titel, Action-Buttons, Confirm-Dialog-Texte, Banner-Texte, Toast-Messages, Error-Texte) ist vollständig in De / En / Cs sofort. Pflicht laut FUI-A-09 + STATE.md-Constraints. Pattern: neuer i18n-Comment-Block `// Cutover migration` in `src/i18n/mod.rs`, Befüllung in `en.rs`/`de.rs`/`cs.rs` analog Plan 08-04 D-26 (60 Keys waren das). - **D-27:** `QuarantineReason`-Texte (`reason_text`, `suggested_action`) bleiben Englisch (Plan 08-08-Konvention) — Frontend rendert sie unverändert. Lokalisierung dieser Backend-Strings ist explizites Backlog-Item für später (FE würde dann `reason_code` als i18n-Key behandeln).

### Claude's Discretion

- Konkrete CSS / Tailwind-Utility-Klassen, Spaltenbreiten und Spacing folgen `input.css` Design-Tokens und existing Frontend-Patterns.
- Wahl zwischen Erweiterung von `CutoverServiceImpl` direkt vs. Sub-Service für Convert-Endpoints liegt in Plan-Phase.
- Profile-Response-Shape (Erweiterung von `CutoverRunResultTO` vs. neuer `CutoverProfileResultTO`) liegt in Plan-Phase — Datenfelder sind aber durch D-10 vorgegeben (Aggregat-Stats + Per-Person-Slice).

- Reason-Code → i18n-Key-Mapping als optionale Folge-Phase ausgelagert (D-27); 8.1 rendert Backend-Texte unverändert.

- Stage-Transition-Animations bleiben minimal (Default-Tailwind-Transitions).

### Established Patterns

**Backend:**
- **Inline-Drift-Report-Pattern** (Plan 08-08): Convert-Response sollte ähnliche Inline-Diagnostics liefern — z.B. die neue `AbsencePeriodTO` + Summary "extra_hours_id soft-deleted, absence_period_id created". Plan-Phase entscheidet die genaue Shape.
- **Cross-Phase-Daten-Bucketing** (Plan 08-08): Bulk-Convert kann `HashMap<(Uuid, AbsenceCategory, u32), Vec<CutoverQuarantineEntry>>` reusen für Pre-Validation oder Progress-Reporting.
- **Heuristik-Pre-Check vor Quarantine-Pfaden** (Plan 08-09): Migration-Loop-Pattern. Convert-Endpoint nutzt nicht das Migration-Loop, sondern direkt die Heuristik-Helper für Range-Compute.
- **`gen_service_impl!`-Macro** für CutoverServiceImpl-Erweiterung (Business-Logic-Tier).
- **Transaction-Pattern** `Option<Self::Transaction>` + `transaction_dao.use_transaction(tx).await?` + `commit(tx)` — beide Convert-Endpoints sind atomic-tx.
- **REST-OpenAPI-Annotation** `#[utoipa::path]` + `ToSchema` auf neuen DTOs + Eintrag in `EXPECTED_PATHS` + `EXPECTED_SCHEMAS` der `rest/tests/openapi_surface.rs` (Plan 08-03-Pattern).
