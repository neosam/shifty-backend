## Pattern Findings (Reference-Code-Excerpts)

### 1. Backend: Where to splice `manual_range` in `convert_quarantine_entry`

**Live code:** `service_impl/src/cutover.rs:412..547`. Der 10-Step-Flow ist:

```
1. Privilege-Gate (CUTOVER_ADMIN_PRIVILEGE)
2. Open Tx
3. find_legacy_extra_hours_not_yet_migrated → row + sibling-set
4. employee_work_details_service.find_by_sales_person_id → contracts
5. detect_weekly_lump_sum → (from_date, to_date)         ← MANUAL-RANGE BRANCHES HERE
6. absence_dao.create(AbsencePeriodEntity)
7. cutover_dao.upsert_migration_source (synthetic_run_id)
8. extra_hours_service.soft_delete_bulk
9. inline refreshed_drift_report (replay migrate_legacy_extra_hours_to_clusters + compute_gate_diagnostic)
10. transaction_dao.commit
```

**Splice-Punkt:** Step 5. Vor Step 5 (Argument-Match) prüfen: `manual_range` is `Some(r)` → skip `detect_weekly_lump_sum`-Aufruf, statt dessen:

```rust
// Pseudo-code skeleton, NOT plan-final
let (from_date, to_date) = if let Some(r) = manual_range {
    // D-30 validations
    let start = parse_iso_date(&r.start_date)?;     // → ServiceError::ValidationError on parse fail
    let end = parse_iso_date(&r.end_date)?;
    if start > end {
        return Err(ServiceError::ValidationError(/* "start_date must be <= end_date" */));
    }
    let row_year = row.date_time.date().year();
    if start.year() != row_year || end.year() != row_year {
        return Err(ServiceError::ValidationError(/* "range must stay inside the quarantine row's year" */));
    }
    // Overlap-Check via existing DAO surface (see Pattern-Finding 3 below).
    let dao_category = extra_hours_category_to_absence(&row.category);
    let range = DateRange::new(start, end).map_err(|_| ServiceError::DateOrderWrong(start, end))?;
    let conflicts = self.absence_dao
        .find_overlapping(row.sales_person_id, dao_category, range, None, tx.clone())
        .await?;
    if !conflicts.is_empty() {
        return Err(ServiceError::ValidationError(Arc::from([
            ValidationFailureItem::OverlappingPeriod(conflicts[0].logical_id),
        ])));
    }
    (start, end)
} else {
    // Existing heuristic path — unchanged.
    detect_weekly_lump_sum(&row, all_legacy.as_ref(), |d| {
        lookup_active_contract(work_details.as_ref(), d)
    }).ok_or_else(|| /* ServiceError::ValidationError as before */)?
};
```

**Steps 6-10 bleiben unverändert** — der einzige sichtbare Unterschied ist, dass bei `manual_range` der `extra_hours_category_to_absence(&row.category)`-Call in Step 6 die category liefert, die der Operator über D-31 explizit unverändert übernehmen muss (= identisch zum Heuristik-Pfad — beide Pfade lesen die category der Source-Row).

**Audit-Trail-Konsistenz (deine Frage 2):**
- Step 7 schreibt `cutover_dao.upsert_migration_source(MigrationSourceRow { extra_hours_id, absence_period_id, cutover_run_id: synthetic_run_id, migrated_at })`. Das ist in `migrations/sqlite/<…>_absence_period_migration_source.sql` als `absence_period_migration_source`-Tabelle persistiert.
- Manual-Range schreibt **dieselbe Tabelle mit demselben Schema** — Reporting / Validatoren können die Source einer migrierten AbsencePeriod weiterhin auflösen. Es ist KEIN neues Audit-Feld nötig, der Manual-Pfad ist ununterscheidbar von Heuristik-Pfad auf dem persistierten Audit-Row.
- **Discretion-Frage für Plan-Phase:** Falls Reporting später unterscheiden möchte ("welche Rows wurden manuell aufgelöst"), könnte Plan-2 eine neue NULLABLE Column `migration_strategy: TEXT` (heuristic|manual) auf `absence_period_migration_source` ergänzen. **Empfehlung: NEIN — out-of-scope für 8.2** (kein User-Bedarf bislang; CONTEXT.md sagt "Audit-Log-UI" ist deferred).

### 2. `derive_hours_for_range` semantics (deine Frage 2)

**Live code:** `service_impl/src/absence.rs:383..510`.

`derive_hours_for_range(from, to, sales_person_id, ctx, tx)` baut eine `BTreeMap<Date, ResolvedAbsence>` per Tag im Range, indem es:
1. Range-Validation (`DateRange::new` → `DateOrderWrong` wenn `from > to`).
2. Permission-Check (`HR ∨ self`).
3. Lädt alle Absences + Contracts + Holidays für die ISO-Wochen des Range.
4. **Per Tag** im Range: aktiver Contract suchen → `has_day_of_week`-Check → Holiday-Skip → `hours_per_day` aus Contract → dominante Absence-Kategorie picken.

**Wichtig für 8.2 — was der manuelle Pfad SKIPT vs. INKLUDIERT:**
- Der **Convert-Pfad** ruft `derive_hours_for_range` **NICHT explizit auf**. Er schreibt nur den Range `{from_date, to_date}` in `absence_period`. Die Stunden-Wirkung (Bilanz, Reports) wird erst von Downstream-Konsumenten (Reporting, Carryover) durch wiederholten `derive_hours_for_range`-Aufruf re-derived.
- Heuristik (`detect_weekly_lump_sum`) und Manual-Range schreiben **denselben** Output: ein `AbsencePeriodEntity { from_date, to_date, category, sales_person_id, … }`. Ab dem Persist sind beide Pfade ununterscheidbar.
- **Holiday-Skip** (Z. 483-485) gilt für beide Pfade gleichermaßen — der Operator kann nicht per Manual-Range einen Holiday als bezahlten Urlaubstag erzwingen. Das ist die strukturelle Konsistenz, die D-06 in 8.1 rechtfertigt.
- **Pre-Contract-Tage / Sundays in Range** (Lila / Anina-Pattern aus 8.1-10): `derive_hours_for_range` skipt sie sowieso (no `active_contract` → `continue` Z. 477-479; no `has_day_of_week` → `continue` Z. 480-482). **Manual-Range darf also einen Range angeben, der z.B. einen Sonntag oder einen Tag vor Vertragsbeginn enthält** — das verfälscht die Bilanz nicht, weil derive die Tage rauswirft. Das macht den Manual-Range robust gegenüber Operator-Tippfehlern.

**Konsequenz:** Der Manual-Range-Backend-Pfad muss `derive_hours_for_range` selbst NICHT aufrufen — er ist nur für die persistierte Range zuständig. Der **Validator-Pfad** (gate-dry-run nach Convert) wird `derive_hours_for_range` aber implizit konsumieren, wenn er die `legacy_sum` mit dem `derived_sum` vergleicht.

### 3. `absence_period`-DAO Overlap-Validation (deine Frage 3)

**Live code:** `dao/src/absence.rs:78..86` + `service_impl/src/absence.rs:189..207`.

```rust
async fn find_overlapping(
    &self,
    sales_person_id: Uuid,
    category: AbsenceCategoryEntity,
    range: DateRange,
    exclude_logical_id: Option<Uuid>,   // None für Create-Path
    tx: Self::Transaction,

```

- **Existing usage** in `AbsenceServiceImpl::create` Z. 193-207: gibt `Arc<[AbsencePeriodEntity]>` zurück; bei `!conflicts.is_empty()` → `ServiceError::ValidationError(ValidationFailureItem::OverlappingPeriod(conflicts[0].logical_id))`.

- **`exclude_logical_id` ist `None`** im Convert-Pfad: das ist semantisch eine `create`-Operation, nicht `update`. Es gibt keine eigene zu exkludierende Row.
- **Category-Berechnung:** `extra_hours_category_to_absence(&row.category)` gibt direkt `AbsenceCategoryEntity` (siehe `cutover.rs:481`). Das matcht `find_overlapping`'s Signatur 1:1.

Migration is privileged."). Manual-Range ist dieselbe Klasse: privilegierte Cutover-Operation, daher direkt-DAO.

### 4. REST DTO Additivität (deine Frage 4)

**Live code:** `rest/tests/openapi_surface.rs:23..36, 92..136`.

Zentrales Snippet aus dem Test-Header (Z. 22-29):
```
//! Schema bodies are intentionally NOT pinned — we only check the schema
//! NAMES exist, so adding/removing fields on a DTO does not require a test update.
```

**Konsequenz für 8.2:** Hinzufügen eines `manual_range: Option<ManualRangeTO>`-Feldes auf `CutoverConvertQuarantineEntryRequest` ist:
- **Path-Test (Z. 138-157):** unverändert (Path bleibt `/admin/cutover/convert-quarantine-entry`).
- **Schema-Test (Z. 159-182):** `EXPECTED_SCHEMAS` enthält bereits `CutoverConvertQuarantineEntryRequest` (Z. 131). **Wenn ein neuer Sub-Struct `ManualRangeTO` (oder analog) neu eingeführt wird, muss er als neuer named-Schema in `EXPECTED_SCHEMAS` ergänzt werden** — utoipa generiert für jedes `ToSchema`-derivierte Type einen separaten Schema-Eintrag, der in `components.schemas` landet. Falls 8.2 stattdessen die Felder flach inline (`Option<String>` × 2), bleibt der Schema-Test komplett unverändert.
- **Runtime-Backwards-Compat:** Per `#[serde(default)]` auf dem optionalen Feld können alte Clients (8.1-Frontend mit dem alten DTO) den 8.2-Endpoint weiterhin ohne `manual_range` aufrufen — exakt wie der existing `extra_hours_ids: Option<Vec<Uuid>>`-Pattern in `CutoverBulkConvertQuarantineRowsRequest` Z. 2349-2350 von rest-types.

**Zusatz-Test, der nicht busted:** Die existing `cutover_convert_dto_tests` (rest-types/src/lib.rs:2435..2506) enthalten `bulk_response_defaults_optional_fields`. Plan-2 sollte einen analogen Test `convert_request_omits_manual_range` ergänzen, der zeigt: serialisiertes Request **ohne** manual_range deserialisiert weiterhin sauber (= D-29 Backwards-Compat-Garantie).

**D-34 Review:** CONTEXT.md sagt "kleiner Update auf das neue Feld" — das ist im aktuellen Test-Code falsch ausgedrückt. Der Test prüft **nur Schema-Namen**, keine Bodies. **Action für Plan-2:** Wenn Sub-Struct `ManualRangeTO`, dann ein neuer Eintrag in `EXPECTED_SCHEMAS`. Sonst kein Test-Update.

## Common Pitfalls

### P-1: Range spannt eine Category-Boundary (z.B. start in Vacation-Year, end in Next-Year)

**Was geht schief:** Operator gibt versehentlich Range `2025-12-29` bis `2026-01-04`. `find_overlapping` würde funktionieren, aber die Quarantäne-Entry-Year-Buckets in `compute_gate` brechen (das Quarantäne-Bucket war für ein bestimmtes Jahr; eine AbsencePeriod, die Jahresgrenze überspannt, könnte beide Year-Buckets affichieren → mismatch).

**Wie verhindern (D-30):** Backend lehnt ab, wenn `start.year() != row.date_time.date().year() || end.year() != row.date_time.date().year()`. Validation-Error vor dem Tx-Commit.

**Warning sign:** Test-Case `manual_range_rejects_year_boundary_crossing`.

### P-2: Heuristic-Mismatch wird zu Manual-Mismatch

**Was geht schief:** Operator gibt einen Range an, dessen Tage-Sum != `legacy_sum` (z.B. legacy 40h, aber Range deckt nur 36h Vertragstage ab — exakt das Karin-Pattern aus 08.1-10). Was macht der Validator?

**Antwort:** Der Validator (gate-dry-run nach Convert) ruft `derive_hours_for_range(from, to, sp, …)` auf und vergleicht das Ergebnis mit `legacy_sum`. Wenn `derive` 36h zurückgibt aber das aufgelöste `legacy_sum` 40h war (Karin), gibt es **Drift** — der Gate würde wieder `passed=false` melden.

**Wichtig — die Wahl des Operators:**
- Karin-Pattern: Operator stellt Range so ein, dass `derive_hours_for_range` das gewünschte Ergebnis (36h) liefert. **Aber legacy_sum war 40h.** Drift bleibt.
- **Daher muss der Operator-Workflow sein:** ZUERST die `extra_hours`-Row via `PUT /extra-hours/{id}` korrigieren (40h → 36h, da die User-Eingabe falsch war), DANN Manual-Range-Convert anwenden. **Oder** Drift akzeptieren (gate-dry-run zeigt es prominent an).

**Plan-Phase-Action:** ManualConvertModal-Helptext (D-29 / D-32-aware) sollte explizit warnen: "Wenn die Stunden des Quarantäne-Eintrags nicht zur Range passen, bleibt eine Drift-Differenz bestehen. Korrigiere ggf. zuerst die Stunden via Edit, oder akzeptiere die Drift." — entscheide das in Discuss-Phase oder lasse es als deferred i18n-Erweiterung.

**Warning sign:** Integration-Test `manual_range_resolves_karin_quarantine` muss klar entscheiden, was es asserted: (a) Quarantäne wird zu 0 (Range matcht ein verkleinertes/korrigiertes legacy_sum) ODER (b) Quarantäne wird zu 0 + Drift-Row wird zu 0 (vollständiges Cleanup). **Empfehlung (a)** — der Test isoliert die 8.2-Surface; Drift-Cleanup ist Operator-Verantwortung über mehrere UI-Aktionen.

### P-3: Race mit konkurrentem heuristischem Bulk-Convert dropped denselben extra_hours_id

**Was geht schief:** Operator A klickt "Manual-Convert" auf Row X, Operator B klickt "Bulk-Convert" auf die Drift-Group, die Row X enthält. Wer gewinnt?

**Antwort (existing 8.1-Backend-Verhalten):**
- Beide Pfade laufen `find_legacy_extra_hours_not_yet_migrated` (filtert `WHERE deleted IS NULL`) UND `extra_hours_service.soft_delete_bulk` UND `cutover_dao.upsert_migration_source` (mit `ON CONFLICT(extra_hours_id) DO NOTHING`).
- Die SQLite-Tx-Isolation (`BEGIN IMMEDIATE`) serialisiert Schreibers. Wer zuerst commit-ed gewinnt; der Verlierer findet die Row nicht mehr in `find_legacy_extra_hours_not_yet_migrated` und bekommt `ServiceError::EntityNotFoundGeneric` (Single-Convert) oder `target_rows.is_empty()` → `EntityNotFoundGeneric` (Bulk-Convert).
- Der `ON CONFLICT(extra_hours_id) DO NOTHING` auf `migration_source` ist der finale Idempotenz-Lock — selbst bei seltsamem Timing kann nicht doppelt gemappt werden.

**Konsequenz für 8.2:** Manual-Range-Branch erbt das Race-Behaviour 1:1 (Steps 3+8 sind unverändert). **Keine zusätzliche Locking-Logik nötig.** Operator-UX: Wenn der Verlust-Pfad `EntityNotFoundGeneric` (HTTP 404) zurück bekommt, muss das Frontend das als "schon konvertiert" rendern (existing 8.1-Behaviour — siehe `service/cutover.rs::cutover_service`-Coroutine, `ConvertSingle`-Branch).

**Warning sign:** Optional ein Integration-Test `concurrent_bulk_and_manual_convert_idempotent` — aber das duplicates 8.1-02-Tests; **Empfehlung: skippen, weil 8.2 keine neue Race-Surface einführt.**

### P-4: Operator gibt Range an, die VOR/NACH dem aktiven Contract liegt

**Was geht schief:** Range `2024-01-01` bis `2024-01-31`, aber Sales-Person hatte ihren Vertrag erst ab `2024-06-01`. Was macht der Backend?

**Antwort (deeply-correct):**
- `absence_dao.create` schreibt die Row ohne Contract-Check.
- `derive_hours_for_range` skipt Tage ohne aktiven Contract (Z. 477-479 in absence.rs).
- Konsequenz: Die AbsencePeriod existiert, hat aber **0h Bilanz-Wirkung**. Das ist konsistent mit dem Lila/Anina-Pattern aus 8.1-10 — pre/post-contract-Tage werden korrekt geskipt.
- Aber: `legacy_sum` der Quarantäne-Row **war** != 0 (sonst wäre sie nicht quarantined). Drift bleibt nach Convert weiterhin sichtbar im gate-dry-run — Operator-Erkennbar.

**Plan-Phase-Action:** **Keine zusätzliche Backend-Validation** für "Range innerhalb Contract" — das ist ein graceful-degrade durch derive-Skip, kein Bug. Aber das Modal-Helptext kann erwähnen: "Tage außerhalb deines aktiven Vertrags werden vom Reporting ignoriert."

### P-5: ISO-8601 String mit Timezone-Suffix vs. plain Date

**Was geht schief:** Frontend sendet `"2026-05-09T00:00:00Z"` statt `"2026-05-09"`. `time::Date::parse(s, format_description!("[year]-[month]-[day]"))` schlägt fehl.

**Wie verhindern:** D-29 sagt explizit "ISO-Datumsstrings konsistent zu `CutoverQuarantineEntryTO.date`". Das existing CutoverQuarantineEntryTO.date ist `String` mit dem Format `"2026-05-08"` (rest-types/src/lib.rs:1945-1946 doc-comment). Plan-2 muss sicherstellen, dass die FE-Submit-Logik den `<input type="date">`-Value 1:1 einsendet (= `"YYYY-MM-DD"`).

**Warning sign:** Backend sollte einen freundlichen Error werfen statt Panic: `ServiceError::ValidationError` mit Klartext.

### P-6: 8.2 Tx-Replay-Failure dropped die success Tx

**Was geht schief:** Step 9 (inline `refreshed_drift_report`-Replay) wirft. Existing 8.1-02-Verhalten ist non-fatal: `refreshed_drift_report = None`, Convert wird trotzdem committet (`cutover.rs:526-537`). Manual-Range erbt das.

**Wie verhindern:** Plan-2 muss im Coroutine-Branch (`CutoverAction::ConvertSingleManualRange`) den Fall behandeln, wo Response `refreshed_drift_report.is_none()` ist → fallback auf separaten `cutover_gate_dry_run`-Call (genau wie der `UpdateExtraHours`-Branch in `service/cutover.rs:142`). Dispatch-Logik:
```
let resp = api::cutover_convert_quarantine_entry_with_manual_range(...).await?;
let drift = match resp.refreshed_drift_report {
    Some(r) => Some(r),
    None => api::cutover_gate_dry_run(config).await.ok().and_then(|r| r.gate_drift_report),
};
CUTOVER_STORE.write().last_dry_run = drift;
```

### P-7: EditExtraHoursModal-Submit-Fallback-Date

**Was geht schief:** EditExtraHoursModal Z. 887-888:
```rust
let parsed = time::Date::parse(date_str.read().as_str(), date_format)
    .unwrap_or_else(|_| time::macros::date!(2026 - 01 - 01));
```
Bei Parse-Fail wird **lautlos** ein hardcoded Datum eingesetzt. Schlecht für 8.2's Manual-Range-Modal — der Operator würde nicht merken, dass sein Range fehlerhaft ist, sondern ein 2026-01-01 würde als Range-Boundary submitted.

**Plan-Phase-Action:** ManualConvertModal **NICHT** denselben `unwrap_or_else`-Fallback verwenden. Stattdessen: Submit-Button disabled solange beide Date-Strings nicht parsbar sind, oder per-Field-Validation-Error rendern.

5. **i18n-Reuse vs. New-Keys:** Ist `CutoverEditModalTitle` ("Eintrag bearbeiten") reuse-fähig für ManualConvert oder muss `CutoverManualConvertModalTitle` neu sein? **Empfehlung neu** — semantisch ist Manual-Convert keine Bearbeitung, sondern eine explizite Migration; getrennte Keys sind kommunikativer.
