## Phase Boundary

Misch-Phase Backend + Frontend im Monorepo (`shifty-backend/` + `shifty-dioxus/`). **Reine Aufräum-/Subtraktiv-Phase:** Die Batch-Cutover-Maschinerie wird **ersatzlos entfernt**. Phase 8.5 hat das per-row Convert-Plumbing bereits aus dem Cutover-Namespace herausgelöst (neuer `AbsenceConversionService` (BL-Tier) + `MigrationSourceDao` + Migration `20260609120000` die `cutover_run_id` dropt). 8.6 ist daher überwiegend **subtraktiv** — kein neuer Fach-Code, nur Entfernung toter Cutover-Wege plus die wenigen Folge-Entkopplungen.

**In Scope:**
- **Frontend-Abriss:** `page/cutover_admin.rs`, `service/cutover.rs`, `state/cutover_state.rs`, die `/admin/cutover`-Route (`router.rs` + `app.rs`), der `AdminCutover`-Menü-Eintrag + `admin_cutover`-Visibility (`top_bar.rs`), alle Cutover-i18n-Keys (de/en/cs + `i18n/mod.rs`) und die Cutover-`api.rs`-Calls werden entfernt.
- **Backend-Abriss:** `rest/src/cutover.rs` (alle 5 Handler: gate-dry-run / commit / profile / convert-quarantine-entry / bulk-convert-quarantine-rows), `service_impl/src/cutover.rs` + `service/src/cutover.rs` (Gate-/Quarantäne-/Profile-/Commit-/Bulk-Logik + Heuristik `detect_weekly_lump_sum` / `iso_week_range` / `lookup_active_contract`), `dao/src/cutover.rs` + `dao_impl_sqlite/src/cutover.rs` werden entfernt. DI-Konstruktion in `shifty_bin/src/main.rs` entkoppelt.
- **Tabellen-Drops (forward-only Migrations):** `absence_migration_quarantine` (Migration `20260503000000`) **und** `employee_yearly_carryover_pre_cutover_backup` (Migration `20260503000002`) werden gedroppt. `absence_period_migration_source` **bleibt** (wird vom 8.5-`AbsenceConversionService` aktiv beschrieben).
- **Flag-Spezifik raus, generische Infra bleibt:** Der `absence_range_source_active`-Seed wird entfernt + die letzte Schreibstelle (Cutover-Commit) verschwindet mit `cutover.rs`. Das **generische** Feature-Flag-Subsystem (Backend `feature_flag` DAO/Service/REST-Endpoint + Tabelle, Frontend `service/feature_flag.rs` + `state/feature_flag.rs` Mechanismus) **bleibt als wiederverwendbarer Mechanismus stehen**.
- **Absences-Menü-Entkopplung:** `top_bar.rs:59` (`absences: logged_in && cutover_active`) wird auf reines `logged_in` umgestellt; der `cutover_active`-Parameter von `nav_visibility` entfällt.
- **Test-Abriss:** Cutover-Integration-Tests (`shifty_bin/src/integration_test/cutover.rs`), `integration_test/feature_flag.rs` (soweit cutover-flag-spezifisch), obsolete Diagnose-Tests (Karin/Lila/Anina) werden ersatzlos entfernt. Der 8.5-`absence_conversion.rs`-Test bleibt.
- **Gates:** OpenAPI-Surface-Test (`rest/tests/openapi_surface.rs`) um Cutover-Pfade/-Schemas bereinigt; `cargo test --workspace` + WASM-Build (`cargo build --target wasm32-unknown-unknown`) + `cargo check --workspace` **ohne Dead-Code-Warnungen** grün.

**Out of Scope:**
- **`add_vacation`-Endpoint aufräumen** — die 8.5-D-12-Bedingung („falls 8.5 die letzte Call-Site entfernt hat") ist **NICHT erfüllt**: `shifty-dioxus/src/component/extra_hours_modal.rs:200` ruft `api::add_vacation` (→ `PUT /vacation` auf `shiftplan_edit`) weiterhin live auf. Der Endpoint **bleibt** → kein Aufräumen in 8.6.
- **`absence_period_migration_source`-Tabelle droppen** — bleibt (D-04 aus 8.5, lebender Backlink).
- **Un-Convert-UI/-Endpoint**, Modell B/C (Überlappungs-Block / Per-Tag-Präzedenz), neue Fach-Features — alle weiterhin außerhalb v1.3.
- **Komplett-Abriss des generischen Feature-Flag-Subsystems** — bewusst verworfen (User-Entscheidung GA-A): die generische Infra bleibt für künftige Flags erhalten.

<decisions>
## Implementation Decisions

> **Modell-Framing (geerbt aus 8.4/8.5-CONTEXT, gilt 8.4–8.6).** Koexistenz statt Ablösung. **Modell-Fakt (kalibriert 2026-06-09):** Es gibt **keine produktive Migrations-Altlast** — Prod steht auf dem Stand *vor* Cutover/`absence_period` (keine `absence_period`-Daten, keine `migration_source`-Rows, kein Cutover lief je). INT-DB ist komplett wipebar. Konsequenz: Die gesamte Cutover/Quarantäne-Maschinerie kann **ersatzlos** weg, es existieren keine zu schützenden konvertierten Daten.

### Feature-Flag-Subsystem (GA-A)
- **D-01 (Generische Infra BEHALTEN, nur spezifischen Flag entfernen):** `absence_range_source_active` ist der einzige Flag und hat nach Cutover-Entfernung keinen Reader/Writer mehr. Statt das ganze Subsystem abzureißen bleibt der **generische** Mechanismus stehen: Backend `feature_flag` DAO/Service/REST-Endpoint (`GET /feature-flag/{key}`) + `feature_flag`-Tabelle, Frontend `service/feature_flag.rs` + `state/feature_flag.rs`. Entfernt werden nur (a) der `absence_range_source_active`-**Seed-Row** (forward-only `DELETE`-Migration; Tabelle bleibt) und (b) die letzte **Schreibstelle** im Cutover-Commit (`service_impl/src/cutover.rs:1319`, fällt mit `cutover.rs` weg). SC4-konform: der *Flag* ist vollständig entfernt, der *Mechanismus* bleibt.
- **D-02 (Dead-Code-Auflösung — Planner-Detail, mit Vorgabe):** D-01 (Infra behalten) kollidiert am **Frontend** mit dem No-Dead-Code-Gate (D-09): der generische `FeatureFlagsState` verliert mit dem Cutover-Fetch + der Absences-Gate-Entkopplung (D-03) seinen einzigen Konsumenten. **Backend ist sauber** — der REST-Endpoint ist im Router registriert, also nicht tot. **Frontend:** Planner entscheidet zwischen `#[allow(dead_code)]` auf dem generischen Flag-Modul (Mechanismus konserviert, kompiliert warnungsfrei) vs. auf eine minimale Shell trimmen. Vorgabe: Mechanismus erhalten, WASM-Build + `cargo check` warnungsfrei.

### Absences-Menü-Gate (GA-B)
- **D-03 (Gate auf reines `logged_in`):** `top_bar.rs:59` ist heute `absences: logged_in && cutover_active` — das Absence-Menü hängt am Cutover-Flag und ist in Prod (Flag seeded-`false`) **versteckt** (Route nur per Direkt-URL erreichbar). Im Koexistenz-Modell ist die Absence-Seite das dauerhafte Haupt-Erfassungs-Feature → Gate wird auf reines `logged_in` umgestellt. Backend `hr ∨ self` (`rest/src/absence.rs`) bleibt die **autoritative** Sichtbarkeits-/Datengrenze (Self sieht eigene, HR alle — konsistent mit 8.5 D-06). Der `cutover_active`-Parameter von `nav_visibility(...)` + die Ableitung im Frontend entfallen.

### Tabellen-Drops (GA-C)
- **D-04 (Drei-Tabellen-Bilanz):** `absence_migration_quarantine` (`20260503000000`) **drop** (SC3). `employee_yearly_carryover_pre_cutover_backup` (`20260503000002`) **drop** — reines Cutover-Backup-Artefakt, nur von `dao_impl_sqlite/src/cutover.rs:327` (Cutover-Commit) je beschrieben, in Prod nie befüllt; SC3 nennt es nicht, wird aber bewusst mitgenommen (kein totes Schema). `absence_period_migration_source` (`20260503000001`) **bleibt** (lebender 8.5-Backlink). Alte Create-Migrationen bleiben unverändert; Drops als **neue** forward-only Migrationen.

### Test-/Migrations-/Verifikations-Strategie (GA-D)
- **D-05 (Forward-only Drop-Migrations):** Keine Down-Migrations nötig — INT wipebar, kein Prod-Data. Drop läuft sauber auf einer DB ohne Quarantäne-/Backup-Rows.
- **D-06 (Cutover-Tests ersatzlos):** `shifty_bin/src/integration_test/cutover.rs` (~1500 Zeilen), die cutover-flag-spezifischen Teile von `integration_test/feature_flag.rs` und die Diagnose-Tests (Karin/Lila/Anina) werden gelöscht, nicht umgebaut. Generische `feature_flag`-Tests, die den behaltenen Mechanismus (D-01) decken, **bleiben** (auf einen neutralen/generischen Key umstellen statt `absence_range_source_active`, falls der Seed wegfällt — Planner-Detail).
- **D-07 (8.5-Tests unangetastet):** `service_impl/src/test/absence_conversion.rs` (8.5) bleibt grün — der neue Convert-Pfad teilt nichts mit der Cutover-Maschinerie (8.5 D-03).
- **D-08 (OpenAPI-Surface bereinigen):** Cutover-Pfade + -Schemas aus `rest/tests/openapi_surface.rs` (`EXPECTED_PATHS` / `EXPECTED_SCHEMAS`) entfernen. Der generische `/feature-flag/{key}`-Pfad bleibt in der Surface (D-01).
- **D-09 (Verifikations-Gate):** `cargo test --workspace` + `cargo build --target wasm32-unknown-unknown` (WASM-Gate, via `nix develop` / `nix-shell -p openssl pkg-config lld`) + `cargo check --workspace` **ohne Dead-Code-Warnungen auf entfernte Symbole** müssen grün sein.

### Snapshot-Schema-Versioning
- **D-10 (KEIN Bump):** `CURRENT_SNAPSHOT_SCHEMA_VERSION` wird **nicht** erhöht. 8.6 entfernt nur tote Cutover-Maschinerie; keine `BillingPeriodValueType`-Berechnung und keine Input-Menge des Snapshot-Builders ändert sich (Reporting summiert seit 8.4 additiv per-row, unberührt). Per `shifty-backend/CLAUDE.md` § „Billing Period Snapshot Schema Versioning" ist kein Bump fällig.

### Claude's Discretion
- Genaue Reihenfolge/Granularität der Subtraktion (eine große Wave vs. Backend-Abriss / Frontend-Abriss / Migrations getrennt) — Planner-Wave-Topologie.
- Frontend-Dead-Code-Auflösung `#[allow(dead_code)]` vs. Shell-Trim (D-02).
- Ob der `absence_range_source_active`-Seed via neue `DELETE`-Migration oder durch Editieren der bestehenden Seed-Migration entfernt wird (Maxime: alte Migrationen unverändert lassen → eher neue forward-only Migration; Planner entscheidet, ob ein DELETE überhaupt nötig ist oder der unbenutzte Seed-Row harmlos stehen bleiben darf).
- Welche generischen `feature_flag`-Tests auf einen neutralen Key umgestellt werden (D-06).
- DI-Auflösung in `shifty_bin/src/main.rs` nach Wegfall von `CutoverService`/`CutoverDao`.
