### 1. Drop-Migrationen (forward-only SQL)

**Analog:** `migrations/sqlite/20260609120000_migration-source-drop-cutover-run-id.sql`

**Vollständige Analog-Datei** (Zeilen 1–9):
```sql
-- Phase 8.5 (D-04): Entferne cutover_run_id aus absence_period_migration_source.
-- Der Backlink ueberlebt 8.6 (CutoverDao wird subtraktiv geloescht), aber loesgeloest
-- von Cutover-Run-Semantik. Prod/INT haben 0 Rows (CONTEXT.md Fakt 2026-06-09) -> DROP+RECREATE safe.
DROP TABLE IF EXISTS absence_period_migration_source;
CREATE TABLE absence_period_migration_source (
    extra_hours_id    BLOB(16) NOT NULL PRIMARY KEY,
    absence_period_id BLOB(16) NOT NULL,
    migrated_at       TEXT NOT NULL
);
```

**Muster extrahiert:**
- Kommentar-Header: Phase-Referenz + Begründung + Prod/INT-Fakt
- `DROP TABLE IF EXISTS` (nicht `DROP TABLE` — idempotent)
- Keine Down-Migration-Datei (forward-only per D-05)
- Dateiname-Konvention: `{YYYYMMDDHHMMSS}_{aktion}-{tabellenname}.sql`

**Neue Datei `20260611000000_drop-absence-migration-quarantine.sql`** — Muster:
```sql
-- Phase 8.6 (D-04): Drop absence_migration_quarantine.
-- Tabelle wurde ausschliesslich von dao_impl_sqlite/src/cutover.rs beschrieben.
-- In Prod nie befuellt (CONTEXT.md Modell-Fakt 2026-06-11). INT wipeable.
DROP TABLE IF EXISTS absence_migration_quarantine;
```

**Neue Datei `20260611000001_drop-employee-yearly-carryover-pre-cutover-backup.sql`** — Muster:
```sql
-- Phase 8.6 (D-04): Drop employee_yearly_carryover_pre_cutover_backup.
-- Reines Cutover-Backup-Artefakt; nur von dao_impl_sqlite/src/cutover.rs:327
-- (Cutover-Commit) je beschrieben. In Prod nie befuellt. INT wipeable.
DROP TABLE IF EXISTS employee_yearly_carryover_pre_cutover_backup;
```

**Neue Datei `20260611000002_delete-absence-range-source-active-seed.sql`** — Muster (analog zum INSERT in `20260501000000`):
