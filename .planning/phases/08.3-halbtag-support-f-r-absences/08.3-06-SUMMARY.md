# Phase 8.3 Plan 06: Halbtag-Frontend Surface Summary

**Full half-day operator surface across AbsenceModal, DriftEntryRow, and ManualConvertModal — selects render, submits thread DayFractionTO, suggestion hint surfaces at ½-day amounts, all on 564 green shifty-dioxus tests + green wasm32 build.**

## Performance

- **Duration:** ~30 min sequential execution
- **Completed:** 2026-05-17
- **Tasks:** 2 (plan-level), 4 atomic commits
- **Files modified:** 4

## Accomplishments

- `AbsenceModal` now exposes the Tageshälfte form input bound to a `day_fraction` Signal that defaults to `DayFraction::Full` on Create and pre-fills from `editing_period.day_fraction` on Edit. Submit threads the value via `(&*day_fraction.read()).into()` into `AbsencePeriodTO.day_fraction`.
- `DriftEntryRow` carries a compact per-row `<select>` (Full / Half) with `aria_label = CutoverDriftDayFractionAria`, placed BEFORE the Convert button. The Convert dispatch sends `CutoverAction::ConvertSingle { extra_hours_id, day_fraction }`.
- D-08.3-FE-02 suggestion hint renders as a `col-span-12 text-small text-ink-muted` row beneath the action buttons when `(amount − 8.0 × 0.5).abs() < 0.5`. 8h is the contract-day fallback because `CutoverGateDriftRowTO` does not carry `contract_hours_per_day`.
- `ManualConvertModal` gained a third `<label>` (Tageshälfte) between End-Date and Error, plus a local `day_fraction` signal. `on_submit` signature migrated from 3-tuple to 4-tuple — all existing 8.2 SSR tests adjusted accordingly.
- `CutoverAction::ConvertSingle` migrated from tuple variant `ConvertSingle(Uuid)` to struct variant `ConvertSingle { extra_hours_id, day_fraction }`. `ConvertSingleManualRange` + `BulkConvert` gained a `day_fraction` field. `BulkConvert` call site in `DriftGroupSection` hardcodes `DayFractionTO::Full` per D-08.3-FE-03 (no group-wide toggle in this phase).
- API client (`cutover_convert_quarantine_entry`, `cutover_bulk_convert_quarantine_rows`) extended with `Option<DayFractionTO>` parameter.
- 6 new dioxus-ssr snapshot tests pin the new surfaces (2 absences + 4 cutover_admin including the 2 D-08.3-FE-02 hint cases).

## Decisions Made

- **D-08.3-FE-01 (UI-Pattern):** `SelectInput` (Full/Half) für alle drei Surfaces. Rationale: UX-Konsistenz mit dem existing Kategorie-Feld in `AbsenceModal`, semantisch klarer als Checkbox (2 explizite Optionen), gleiche Spacing-Tokens. `FormCheckbox`-Alternative wurde verworfen (würde Field+SelectInput-Konsistenz brechen).
- **D-08.3-FE-02 (Suggestion-Hint sichtbar):** Hint wird sichtbar in `DriftEntryRow` gerendert wenn `(entry.amount − 8.0 × 0.5).abs() < 0.5`. Default-Toggle bleibt Full (Operator quittiert den Hint bewusst und flippt manuell). `contract_hours_per_day` ist nicht im `CutoverGateDriftRowTO`-Payload — 8.0 ist Frontend-Fallback (dokumentiert im Code-Kommentar).
- **D-08.3-FE-03 (Bulk-Toggle):** KEIN gruppen-weiter Halb/Ganz-Toggle in `DriftGroupSection`. `BulkConvert`-Dispatch verwendet hardcoded `DayFractionTO::Full`. i18n-Key `CutoverBulkDayFractionLabel` bleibt deklariert + übersetzt (Future-Reservation), wird aber in dieser Phase nicht konsumiert.

## Deviations from Plan

### Minor adaptations

**1. Existing 8.2 snapshot-test `manual_convert_modal_renders_two_date_inputs` invariant tightened**
- **Found during:** Task 2 (DriftEntryRow + ManualConvertModal UI)
- **Issue:** The test asserted `!html.contains("<select")` to encode D-31 (no editable category surface). Our new legitimate `day_fraction` `<select>` violates that broad pattern even though the D-31 intent (no category mutation) is still honored.
- **Fix:** Replaced the assertion with two more specific checks: `!html.contains(r#"value="vacation""#)` and `!html.contains(r#"value="sick_leave""#)` — pins the category-mutation invariant by checking for the category-option values rather than excluding `<select>` elements entirely.
- **Verification:** `manual_convert_modal_renders_two_date_inputs` passes; D-31 intent preserved.

**2. `cutover_admin.rs` call sites for the enum migration co-located with Task 1 commit 1/4**
- **Found during:** Task 1 (API + CutoverAction)
- **Reason:** Migrating `CutoverAction::ConvertSingle` from tuple to struct variant is a breaking change that the Rust compiler enforces at every call site. The call sites in `cutover_admin.rs` (DriftEntryRow Convert button, DriftGroupSection BulkConvert, ManualConvertModal on_submit/on_submit-shape) must therefore be migrated atomically in the same commit, so `cargo build` stays green after each commit. Plan's verify step for Task 1 includes `cargo build` + `cargo test absence_modal_*`, both of which would otherwise fail.
- **Fix:** Commit 1/4 contains all enum-migration changes incl. `cutover_admin.rs` call sites; commit 3/4 (Task 2) layers the actual UI selects + hint atop that scaffold.
- **Impact on plan:** None — the work is the same; only the commit boundary differs from the plan's nominal split.

**Total deviations:** 2 minor adaptations, both inherent to enum-shape migrations + invariant-test refinement. No scope creep.
