# Phase 8.3 Plan 02: DAO + Service-CRUD — Persist day_fraction round-trip

**Komplettes Threading von `day_fraction` durch die DAO-Schicht (alle 6 SELECT-Queries + INSERT + TryFrom-Mapping) und Verdrahtung im Service-`update`-Pfad. Ersetzt die Plan-01-Bridges durch echte Persistierung; pinnt no-drift via In-Memory-SQLite-Integration-Test.**

## Performance

- **Duration:** ~20 min (Task 1 + Task 2 + SUMMARY)
- **Tasks:** 2 (DAO-Threading + Service-Update + Tests)
- **Files modified:** 3 + 1 created
- **Net LOC added:** ~80 (DAO ~25 + service_impl 1-Liner ersetzt + Integration-Tests ~110)

## Accomplishments

### Task 1 — DAO-SQLite-Threading (`a91d60d9`)

- **`AbsencePeriodDb.day_fraction: String`** als neues Pflicht-Feld am Ende des Structs (sqlx-Symmetrie).
- **`TryFrom<&AbsencePeriodDb> for AbsencePeriodEntity`** mapped `day_fraction` mit explizitem Match:
  - `"full"` → `DayFractionEntity::Full`
  - `"half"` → `DayFractionEntity::Half`
  - sonst → `DaoError::EnumValueNotFound(value.into())` (analog `AbsenceCategory`-Pattern, Defense-in-Depth gegen DB-Korruption — der CHECK-Constraint aus Plan 01 macht das in der Praxis unmöglich, aber das typisierte Error-Fallback bleibt korrekt).
- **`day_fraction_to_str(&DayFractionEntity) -> &'static str`**-Helper direkt nach `category_to_str` (lowercase 'full'/'half').
- **Alle 6 SELECT-Queries** projizieren jetzt `day_fraction` als letztes Feld (entspricht Struct-Reihenfolge):
  - `find_by_id`
  - `find_by_logical_id`
  - `find_by_sales_person`
  - `find_all`
  - `find_overlapping` (mit + ohne `exclude_logical_id`)
  - `find_overlapping_for_booking`
- **`create`-INSERT** schreibt die neue Spalte (`day_fraction_str` als zusätzliches Binding).
- **`update`** UNVERÄNDERT — Tombstone-Pattern setzt nur `deleted`, `update_version`, `update_process`. Neue `day_fraction`-Werte landen via `create()` der neuen Row (PATTERNS.md Pattern 6 dokumentiert das explizit; sonst würde die Versionierungs-Semantik brechen).
- **Bridge-Kommentar** in `TryFrom` ersetzt durch echten Plan-02-Kommentar.
- **Verifikation:** `cargo build -p dao_impl_sqlite` exit 0 — sqlx-compile-time-Query-Check bestätigt vollständige SELECT/INSERT-Symmetrie.

### Task 2 — Service-CRUD + Integration-Tests (`811d96c2`)

- **`AbsenceServiceImpl::update`**: in der `new_entity`-Konstruktion (Z. 313-327) ersetzt die Plan-01-Bridge durch `day_fraction: (&request.day_fraction).into()`. **KRITISCH:** `request.day_fraction`, NICHT `active.day_fraction` — sonst könnte der User das Feld nie ändern.
- **`create`-Pfad**: KEIN Code-Change nötig — `create` ruft `AbsencePeriodEntity::try_from(&entity)` (Z. 215), und das `TryFrom<&AbsencePeriod>`-Impl in `service/src/absence.rs::Z. 122-139` propagiert `day_fraction` bereits seit Plan 01.
- **Test-Fixtures**: KEINE bestehenden Test-Fixtures mussten in diesem Plan angefasst werden — Plan 01 hatte alle 18 Construction-Sites mit `DayFractionEntity::Full` / `DayFraction::Full` versorgt (siehe Plan-01-SUMMARY "Bridge-Defaults"). Plan 02 ersetzt nur den `dao_impl_sqlite::TryFrom`-Bridge und den `service_impl::absence::update`-Bridge. Alle 411 service_impl-Tests + alle 14 absence-bezogenen shifty_bin-Integration-Tests bleiben grün, ohne Fixture-Edits.
- **Neuer Integration-Test-File `shifty_bin/src/integration_test/absence_no_drift.rs`** (118 LOC):
  - `absence_period_default_day_fraction_is_full_on_legacy_rows`: simuliert legacy-Row-Insert per direktem SQL-INSERT, der die `day_fraction`-Spalte OMITS. Migration-Default `'full'` greift; SELECT via `find_by_id` liefert `DayFraction::Full`.
  - `absence_period_create_and_select_with_half_persists_correctly`: erzeugt eine Halbtag-Absence per `AbsenceService::create`, reloadet via `find_by_id`, verifiziert dass `day_fraction == Half`. Zusätzlich raw-SELECT auf `day_fraction`-Spalte → Wert ist die lowercase `'half'`-Repräsentation (pinnt die DB-Storage-Konvention).
- Modul-Registry in `shifty_bin/src/integration_test.rs` erweitert (alphabetisch eingeordnet, `absence_no_drift` vor `absence_period`).

## Deviations from Plan

### Deviation 1: Test-File-Pfad

- **Plan sagt:** `service_impl/src/test/absence_no_drift.rs`
- **Tatsächlich:** `shifty_bin/src/integration_test/absence_no_drift.rs`
- **Grund:** `service_impl` hat keine sqlx-Abhängigkeit (Cargo.toml zeigt nur `mockall`, `service`, `dao`, `tokio`, `time`, `uuid` — kein `sqlx`, kein `dao_impl_sqlite`). Der `TestSetup`-In-Memory-SQLite-Stack lebt im `shifty_bin`-Crate (Z. 266 in `integration_test.rs`). Analoge Plan-01-Tests (`absence_period.rs`, `extra_hours_update.rs`) liegen ebenfalls dort.
- **Inhaltliche Treue:** Beide Tests prüfen exakt das, was der Plan beschreibt (Legacy-Default + Half-Round-Trip).

### Deviation 2: Keine Test-Fixture-Mass-Edits nötig

- **Plan sagt:** "Bestehende Test-Fixtures hieven ... ergänze an jeder Konstruktions-Stelle ..."
- **Tatsächlich:** Plan 01 hat alle 18 Construction-Sites bereits mit `day_fraction: ...::Full` versorgt (siehe Plan-01-SUMMARY "Bridge-Defaults für workspace-weiten Compile"). Plan 02 musste keine Fixtures anfassen — alle 411 `service_impl`-Tests und alle 14 absence-bezogenen `shifty_bin`-Tests waren schon grün.
- **Impact:** Reduziert die Plan-02-Änderungsfläche von "DAO + Service + 18 Fixture-Edits" auf "DAO + Service + 2 neue Tests" — sauberer Cut zwischen Wave-0 (Plan 01: alle Site-Updates) und Wave-2 (Plan 02: echte Persistierung).
