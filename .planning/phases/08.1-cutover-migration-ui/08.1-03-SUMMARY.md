# Phase 8.1 Plan 03: Service-Bulk-Convert Backend Summary

**Strict-atomic Bulk-Convert lands on `CutoverServiceImpl` (Business-Logic-Tier per D-03) — single Tx over the `(sales_person_id, category, year)`-Triple (optionally narrowed by `extra_hours_ids`), `cutover_admin`-privilege-gated, range derived per row via the Plan 08-09 weekly-lump-sum heuristic. Any heuristic mismatch in the target set fails the whole batch with `ServiceError::ValidationError` (HTTP 422) — no DAO writes happen on the failure path. All converted rows share one `synthetic_run_id` (RESEARCH Q3) for audit cohesion. The inline `refreshed_drift_report` (D-08) reuses Plan 02's `compute_gate_diagnostic` helper verbatim — no duplicate gate-computation logic. 4 mockall unit tests cover atomic-success (3 rows), strict-atomic-422 mismatch, explicit-ids subset narrowing, and empty-match-set-not-found; the matching wire-tier `From`-impl in `rest-types` is gated behind the `service-impl` feature.**

### Service Impl (`service_impl/src/cutover.rs`)

The `bulk_convert_quarantine_rows` body follows the locked 10-step flow:

1. `permission_service.check_permission(CUTOVER_ADMIN_PRIVILEGE, ...)` — short-circuits before any Tx work if the caller lacks the privilege.
2. `transaction_dao.use_transaction(tx)` — opens the cutover Tx.
3. `cutover_dao.find_legacy_extra_hours_not_yet_migrated` returns the full not-yet-migrated set; the heuristic needs the full sibling set to detect single-row-per-week clusters.
4. Filter by `(sales_person_id, category, year)` triple, narrowed optionally by `explicit_ids`. Year is the calendar year of the row's `date_time` (matches the gate-bucket convention). Empty match-set → `EntityNotFoundGeneric` (HTTP 404).
5. `employee_work_details_service.find_by_sales_person_id` once for the sales person — replicates the migration-loop pre-fetch shape.
6. **Strict-atomic per-row heuristic:** iterate target_rows, call `detect_weekly_lump_sum(&row, all_legacy.as_ref(), |d| lookup_active_contract(&work_details, d))` for each. Any `None` returns `ServiceError::ValidationError(Arc::from([ValidationFailureItem::InvalidValue(format!("Row {} does not match weekly-lump-sum heuristic; manual edit required", row.id))]))` — implicit Tx-Drop rollback leaves the DB untouched.
7. For each `(row, from_date, to_date)` in the planned set: build `AbsencePeriodEntity` → `absence_dao.create(&entity, CUTOVER_MIGRATION_PROCESS, tx)` (mirrors Plan 02's direct DAO insert, bypassing AbsenceService forward-warning loop) → `cutover_dao.upsert_migration_source(MigrationSourceRow { extra_hours_id, absence_period_id, cutover_run_id: synthetic_run_id, migrated_at }, tx)`. The same `synthetic_run_id` is used for ALL rows (RESEARCH Q3 — audit-cohesion).
8. `extra_hours_service.soft_delete_bulk(Arc::from(deleted_ids.into_boxed_slice()), CUTOVER_MIGRATION_PROCESS, Authentication::Full, Some(tx))` — single bulk call.
9. **Inline refreshed drift-report** (D-08, RESEARCH P-03 option a): replay `migrate_legacy_extra_hours_to_clusters(synthetic_run_id, migrated_at, tx)` then call `compute_gate_diagnostic(synthetic_run_id, migrated_at, &bucket_map, tx)`. Failures here are non-fatal — the convert itself succeeded; we drop `refreshed_drift_report` to `None` rather than rolling back. **Reuses Plan 02's `compute_gate_diagnostic` verbatim — no duplicate logic.**
10. `transaction_dao.commit(tx)` — atomic flip.

The new private helper `absence_category_to_dao(c: &AbsenceCategory) -> AbsenceCategoryEntity` was added next to `extra_hours_category_to_absence` to map service-tier → dao-tier categories so the triple filter in step 4 can compare apples-to-apples.

### Wire-Tier From-Impl (`rest-types/src/lib.rs`)

`impl From<&service::cutover::BulkConvertQuarantineRowsOutcome> for CutoverBulkConvertQuarantineRowsResponse` (gated behind `feature = "service-impl"`). Each `AbsencePeriodEntity` is bridged through `service::absence::AbsencePeriod` so `rest-types` does not need a direct `From<&AbsencePeriodEntity> for AbsencePeriodTO` impl. `errors` is wire-mapped 1:1 from the service-tier `BulkConvertRowError`:

```rust
#[cfg(feature = "service-impl")]
impl From<&service::cutover::BulkConvertQuarantineRowsOutcome>
    for CutoverBulkConvertQuarantineRowsResponse
{
    fn from(o: &service::cutover::BulkConvertQuarantineRowsOutcome) -> Self {
        Self {
            converted_absence_periods: o.converted_absence_periods.iter()
                .map(|entity| {
                    let svc: service::absence::AbsencePeriod = entity.into();
                    AbsencePeriodTO::from(&svc)
                })
                .collect(),
            deleted_extra_hours_ids: o.deleted_extra_hours_ids.clone(),
            refreshed_drift_report: o.refreshed_drift_report.as_ref().map(CutoverGateDriftReportTO::from),
            errors: o.errors.iter()
                .map(|e| CutoverConvertErrorTO {
                    extra_hours_id: e.extra_hours_id,
                    reason: e.reason.clone(),
                })
                .collect(),
        }
    }
}
```

## Decisions Made

1. **Trait param shape `Option<Arc<[Uuid]>>`** mirrors the existing `extra_hours_service.soft_delete_bulk(ids: Arc<[Uuid]>, ...)` convention in the codebase. The REST-handler in Plan 04 will adapt the wire-tier `Option<Vec<Uuid>>` via `Arc::from(vec.into_boxed_slice())` at the boundary; the trait stays `Arc<[Uuid]>`-flavoured to keep cheap-clone semantics inside the hot path.
2. **Heuristic-mismatch fail-fast in step 6.** All `detect_weekly_lump_sum` checks complete BEFORE any DAO write in step 7. Strict-atomic (RESEARCH P-10) — any `None` aborts before any side effect, so the implicit Tx-Drop rollback is sufficient (no explicit rollback needed). The plan body suggested wrapping mismatch in `ValidationError(Arc<str>)` but the actual `ServiceError::ValidationError` variant takes `Arc<[ValidationFailureItem]>`; I wrapped the message in `ValidationFailureItem::InvalidValue(Arc::from(format!(...).as_str()))` and pushed a 1-element `Arc<[...]>`. This is the same idiom Plan 02 used for its single-convert validation error.
3. **Shared `synthetic_run_id` for the whole batch** (RESEARCH Q3). Single `Uuid::new_v4()` at the start; threaded into every `upsert_migration_source` AND into `migrate_legacy_extra_hours_to_clusters` (step 9 replay) AND into `compute_gate_diagnostic`. Test 1 asserts the cohesion via a `Mutex<Option<Uuid>>` shared across the 3 `withf` predicates — first call captures the run_id, subsequent calls assert equality.
4. **`absence_category_to_dao` helper added at module bottom.** Mirrors `extra_hours_category_to_absence` but in the inverse direction. No fallback variant needed because `service::absence::AbsenceCategory` is exhaustive over `{Vacation, SickLeave, UnpaidLeave}` — same three categories. Cleaner than reusing the existing helper because the existing one needs the `panic!` guard for non-legacy `ExtraHoursCategoryEntity` variants.
5. **Test 3 (explicit-ids) models replay-empty for isolation.** The plan body's behaviour spec ("triple matches 5 rows but `extra_hours_ids = Some([id1, id3])` → only 2 converted") is satisfied by returning `Arc::from([])` on the second `find_legacy_extra_hours_not_yet_migrated` call (Plan-02 convention). This isolates the bulk-convert phase's narrowing behaviour from the inherited Plan-02 replay-side-effect (where remaining valid lump-sum rows would also write absence_period rows during the diagnostic replay — a separate concern not in this plan's scope).
6. **Unused-import cleanup.** Initial Task-2 import block included `BulkConvertRowError`, but since the impl only constructs the empty `Vec::new()` for `errors` (no per-row error path is reachable in strict-atomic mode), the import was unused. Removed; the symbol stays exported from `service/src/cutover.rs` for the wire-tier From-impl in `rest-types/src/lib.rs`.

## Patterns Established

- **Strict-atomic batch validation pattern.** When a service method orchestrates a multi-row mutation under a single Tx, validate ALL rows BEFORE the first DAO write. Collect a `Vec<(Row, ValidationOutput)>` in a pre-pass; on the first invalid row, return `Err(ValidationError)` — the implicit Tx-Drop rollback removes all writes (mockall `.times(0)` guards make this assertable). No explicit `transaction_dao.rollback` needed.
- **Synthetic-run-id-as-cohort-marker.** When an ad-hoc batch operation needs an audit-trail anchor but is not part of an orchestrated run, generate a single synthetic `Uuid::new_v4()` at the start and thread it into every audit-trail-bearing DAO call (`upsert_migration_source.cutover_run_id` here). Tests assert cohesion via `Mutex<Option<Uuid>>` shared across `withf` predicates.
- **Helper reuse from prior plan in same wave.** Plan 02's `compute_gate_diagnostic` helper was extracted as `pub(crate)` precisely so Plan 03 could reuse it without duplicating the gate-computation logic. The pattern: when wave-N+1 follows wave-N in the same domain, refactor wave-N's IO-mixed methods to expose a pure-computation `pub(crate) fn` so wave-N+1 can layer different IO behaviour on top.
