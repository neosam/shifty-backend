| Fn                                            | HTTP Method | Path                                                  | Returns                                           | Error mapping                       |
| --------------------------------------------- | ----------- | ----------------------------------------------------- | ------------------------------------------------- | ----------------------------------- |

| `cutover_bulk_convert_quarantine_rows`        | POST        | `/admin/cutover/bulk-convert-quarantine-rows`         | `CutoverBulkConvertQuarantineRowsResponse`        | 422 → `ShiftyError::Validation(text)` else `error_for_status_ref()?` |

## Decisions Made

1. **Cutover fns appended at end of api.rs** with a Phase-8.1 comment-banner block. The plan suggested either appending or finding a "closest existing cutover-related block" — the existing api.rs has zero pre-existing cutover fns (none of the three Phase-4 endpoints had a frontend wrapper yet), so the natural location is at the end, immediately after `get_blocks`. The comment banner mirrors the AbsencePeriod-block pattern (api.rs:470-484) and groups all 5 endpoints together for easy discovery from search.
2. **`AbsenceCategoryTO` added to the rest-types import block** in api.rs alongside the new Cutover-* DTOs. It was previously only imported in `state/absence_period.rs` (Plan 08-04) and `service/absence.rs`. The bulk-convert fn signature requires it — `category: AbsenceCategoryTO` is a typed parameter forwarded directly into the `CutoverBulkConvertQuarantineRowsRequest` body.
3. **Single prefix-based `[[web.proxy]]` entry** instead of five separate full-path entries. `dx serve`'s proxy uses prefix matching — one entry `http://localhost:3000/admin/cutover` forwards every sub-path under that root (RESEARCH P-08). The existing `/absence-period`, `/feature-flag`, `/extra-hours`, etc. entries all follow the same pattern.
4. **WASM build gate run with `lld` injected via nix-shell.** The default cargo-build of the Dioxus frontend uses `lld` as the linker (per project setup), which is not on the bare nix-shell PATH. Adding `lld` to the shell command (`nix-shell -p openssl pkg-config lld --command "cargo build --target wasm32-unknown-unknown"`) makes the WASM gate green. This is a Wave-2 NixOS-toolchain hygiene tweak; future Wave-2 plans will follow the same recipe.

## Patterns Established / Reused

- **Frontend api.rs empty-body POST pattern** (cutover_profile / cutover_gate_dry_run / cutover_commit) — `client.post(url).json(&serde_json::json!({}))`. Backend's three Phase-4 cutover handlers accept an empty JSON body; the frontend matches that with the canonical empty-object literal.
- **Frontend api.rs typed-body POST + 422 mapping** (cutover_convert_quarantine_entry / cutover_bulk_convert_quarantine_rows) — verbatim reuse of the `create_absence_period` template (api.rs:532-555). Status-check before `error_for_status_ref()` catches 422s as `ShiftyError::Validation(text)` so the page can render the backend's reason text inline. All other 4xx/5xx flow through `error_for_status_ref()` as `ShiftyError::Reqwest`.
- **Dioxus.toml prefix-proxy single-entry pattern** — RESEARCH P-08 documents this; existing `/booking-information`, `/booking-log`, `/permission`, `/shiftplan-edit` etc. all use bare-prefix entries. Adding 5 separate entries would be redundant.
