## Decisions Made

1. **Boundary conversion at the REST handler.** `AbsenceCategoryTO` → `AbsenceCategory` and `Option<Vec<Uuid>>` → `Option<Arc<[Uuid]>>` are done inline in the bulk handler body (between `error_handler` open and the service call). No new From-impl was added — the existing `From<&AbsenceCategoryTO>` impl in `rest-types/src/lib.rs:1585` is reused. The `Arc<[Uuid]>` conversion uses `Arc::from(vec.into_boxed_slice())` per Plan 03 D-02.
2. **Explicit schema registration in `components(schemas(...))`** (per RESEARCH P-05). utoipa does NOT auto-discover schemas referenced through `request_body` / `body =` annotations alone — you have to register every DTO that should appear in the OpenAPI document. The 5 new convert DTOs are appended to the existing schema list.
3. **404 mapping for idempotent-replay.** The plan body said "second call's status is 404 (or 4xx)". I picked 404 as the strong assertion because:
   - Service returns `EntityNotFoundGeneric` (Plan 02 Task 2 line 444-448).
   - `rest/src/lib.rs:169` maps `EntityNotFoundGeneric` to HTTP 404.
   - This is the canonical "row not found" mapping; relaxing the assertion to "any 4xx" would weaken the contract verification.
4. **Audit cohesion query goes through a direct SQL read of `absence_period_migration_source`.** The synthetic `cutover_run_id` is generated inside the service and is NOT exposed through the trait or DTO surface. Querying the migration-source table directly is the cleanest test-side verification: the test depends only on the table schema (migration `20260503000001`) which is stable, not on any service-layer detail. Helper `distinct_run_ids_for_migrated` fetches `cutover_run_id` per `extra_hours_id` and deduplicates — Q3 audit cohesion is `len() == 1`.
5. **Test 5 fixture: heuristic mismatch at row #2 (mid-batch).** Placing the mismatch at index 2 (not 1, not last) verifies the strict-atomic pre-pass loop correctly aborts AFTER the validation of row 1 succeeded but BEFORE any DAO write happens (Plan 03 step 6 ordering). The post-call counts assertion (3 active, 0 absence_period, 0 cutover-softdeleted) is the strongest possible rollback witness in an in-memory SQLite without exposing Tx state.

## Patterns Established

- **`request_body = ...` + `axum::Json<...>` extractor for typed POST bodies in REST handlers.** Plan 04 introduces the first Cutover handler with a typed JSON request body (the existing 3 handlers all use empty `{}`). Pattern: `axum::Json(req): axum::Json<RequestTO>` extractor + `#[utoipa::path(... request_body = RequestTO ...)]` annotation. Used verbatim by both new handlers.
- **Boundary-Conversion-at-REST-Layer.** Wire-tier DTO with `Vec<Uuid>` / `AbsenceCategoryTO` is converted to service-tier `Arc<[Uuid]>` / `AbsenceCategory` inside the handler body before the service call. Keeps the trait signature lean (`Arc<[Uuid]>` for cheap-clone hot path) while matching the wire convention (`Vec<Uuid>` + camelCase JSON).
- **Audit-Cohesion-Direct-DAO-Read in Integration Tests.** When a service generates a synthetic correlation id (cutover_run_id) that's NOT exposed in the trait surface, the integration test reads the audit-trail table directly (`absence_period_migration_source` here) and asserts the cohesion property. Keeps the trait surface lean while still allowing test-side cohesion verification.
- **Strict-Atomic Mid-Batch Mismatch Test Fixture.** For a strict-atomic batch operation, place the failure-trigger at index 2 (mid-batch) so the pre-pass loop can verify it correctly aborts *between* rows. Coupled with pre/post-call count assertions, this is the strongest rollback witness possible in in-memory SQLite.
