# Phase 8.1: Cutover-Migration-UI - Research

**Researched:** 2026-05-09
**Domain:** Mixed Backend (Rust / Axum / SQLx) + Frontend (Dioxus 0.6 / WASM) — atomic-tx Cutover-Resolution-Endpoints + 3-Stage-Wizard + Drift-Resolution-UI
**Confidence:** HIGH (alle technischen Patterns sind durch Phase 8 Plans 08-04..08-09 vorgegeben und im Codebase verifiziert)

## Summary

Phase 8.1 ist eine Closure-Phase für Phase 8: Backend-additiv zwei neue atomic-tx Endpoints auf der existierenden `CutoverServiceImpl` (Business-Logic-Tier) plus ein 3-Stage-Stepper-Wizard im Frontend mit Drift-Resolution-Liste, Per-Eintrag-Aktionen und Bulk-Actions. Die technischen Patterns sind durch die abgeschlossenen Phase-8-Plans (08-04 Frontend-Foundation, 08-05 Page-Composition, 08-07 Feature-Flag-State + TopBar-Submenu, 08-08 inline drift-report DTOs, 08-09 `detect_weekly_lump_sum` + Helpers) vollständig vorgegeben — der Plan kann mechanisch reusen.

Wichtigste Befunde:
1. **Convert-Endpoint-Pattern** kommt aus `service_impl/src/cutover.rs::commit_phase` (Step c) und `migrate_legacy_extra_hours_to_clusters`: Soft-Delete via `extra_hours_service.soft_delete_bulk(ids, "phase-4-cutover-migration", Auth::Full, Some(tx))` + direkter `absence_dao.create(&AbsencePeriodEntity, CUTOVER_MIGRATION_PROCESS, tx)`-Insert in derselben Tx. **Nicht** `absence_service.create()` benutzen — der hat Forward-Warning-Loop und Permission-Layer, die hier eine privilegierte Cutover-Operation ausbremsen würden.
2. **Range-Auto-Compute** für Single-Convert reuses verbatim die Plan-08-09-Helper `detect_weekly_lump_sum(row, all_rows, contract_at)` + `iso_week_range(day) -> (Mo, So)` + `lookup_active_contract(work_details, day)`. Diese sind file-private (`fn` ohne `pub`) am Modulende von `service_impl/src/cutover.rs` — Plan-Phase muss entweder (a) sie zu `pub(crate)` heben oder (b) die neuen Convert-Methoden in derselben Datei platzieren.
3. **Idempotenz** kommt durch `extra_hours.deleted IS NULL`-Check + den existierenden `migration_source`-UPSERT (`ON CONFLICT(extra_hours_id) DO NOTHING`) frei. Jede Convert-Operation ist eine deklarative `(extra_hours.id → absence_period.id)`-Mapping-Erweiterung.
4. **Frontend-Skelett** ist 1:1 das Phase-8-Plan-04-Pattern: `src/api.rs` (5 fns: 3 existing remappings + 2 neue für Convert-Endpoints), `src/state/cutover_state.rs` (Wizard-Stage + Drift-Liste), `src/service/cutover.rs` (Coroutine + `CUTOVER_STORE` GlobalSignal + `CUTOVER_DRIFT_REFRESH: GlobalSignal<u64>` analog `ABSENCE_REFRESH`), `src/page/cutover_admin.rs` (Single-File-Page-Composition analog `absences.rs`).
5. **Validation-Strategie** orientiert sich Plan-für-Plan an Phase 8: Backend `mockall`-Unit-Tests + In-Memory-SQLite-Integration-Tests; Frontend dioxus-ssr-Snapshot-Tests + Per-Locale-Reference-Matcher-Tests (Pitfall-2-Guard); WASM-Build-Gate via `cargo build --target wasm32-unknown-unknown` aus `shifty-dioxus/`.

**Primary recommendation:** Plan-Phase legt 4-6 Waves an: Wave 1 Backend Convert-Endpoints (Service-Methods auf `CutoverServiceImpl` direkt, KEIN Sub-Service, weil die existierenden Helper file-private sind), Wave 2 Frontend Foundation (api / state / service / loader / i18n / Dioxus.toml-Proxy), Wave 3 Frontend Page (Wizard + Drift-Resolution-Liste + Inline-Edit-Modal + Type-to-confirm-Modal + Idempotenz-Banner), Wave 4 Diagnose-Plan gap-1 (a), Wave 5 8.1-eigener UAT, Wave 6 Phase-8-HUMAN-UAT-Subsumption.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Backend-Erweiterung (Misch-Phase):**
- **D-01:** Endpoint `POST /admin/cutover/convert-quarantine-entry` — Body `{ extra_hours_id }`; Backend leitet `{from, to}`-Range automatisch via `detect_weekly_lump_sum`-Logik ab; in einer Tx: soft-delete `extra_hours` + insert `absence_period` + return `AbsencePeriodTO`. Idempotent via existing `extra_hours.deleted IS NULL`-Check. Range-vom-Frontend-vorgegeben verworfen.
- **D-02:** Endpoint `POST /admin/cutover/bulk-convert-quarantine-rows` — Body `{ sales_person_id, category, year }` (optional zusätzlich `extra_hours_ids: Vec<Uuid>` für Multi-Selection-Erweiterung). Single Tx über alle Rows. Konsumiert die Heuristik aus Plan 08-09. Atomar.
- **D-03:** Service-Tier: **Business-Logic-Tier** auf `CutoverServiceImpl` (existing). Plan-Phase entscheidet, ob direkt auf `CutoverServiceImpl` oder Sub-Service.
- **D-04:** Inline-Edit nutzt existing `PUT /extra-hours/{id}`. Editierbar im UI nur `amount` und `date` (nicht `category`, nicht `description`).
- **D-05:** "Skip" ist Frontend-only — Row lokal aus UI-Liste ausgeblendet; kein Backend-Persist; nach Page-Reload wieder sichtbar.
- **D-06:** KEIN Backend-Heuristik-Fix für Feiertage in `detect_weekly_lump_sum`. `derive_hours_for_range` skipt Holidays explizit (`absence.rs:483-485`); würde der Pre-Check Holidays nicht abziehen, hätten migrierte AbsencePeriods permanent Drift.

**Wizard-Topologie:**
- **D-07:** Stepper Profile → Dry-Run → Commit; Stage 2 nur erreichbar nach Stage 1; Stage 3 nur wenn `quarantined_rows == 0` aus letztem Dry-Run.
- **D-08:** Auto-Re-Run von `POST /admin/cutover/gate-dry-run` nach jeder Resolve-Aktion (Single-Convert / Bulk-Convert / Edit / Delete). Skip triggert KEINEN Re-Run.
- **D-09:** Loading-State: Inline-Spinner auf Action-Button + Liste-Skeleton; Page sonst interaktiv, Drift-Liste blockiert Klicks bis Re-Run komplett.
- **D-10:** Profile-Stage zeigt Stat-Boxen (Aggregat) + Per-Person-Kacheln. Profile-Response-Shape Plan-Phase-Decision (Erweiterung `CutoverRunResultTO` ODER neuer `CutoverProfileResultTO`).

**Drift-Resolution-Liste:**
- **D-11:** Drift-Liste gegliedert nach `(sales_person_name, category, year)`. Outer-Sections per Gruppe; innerhalb Quarantine-Entry-Liste + Section-Aktionen.
- **D-12:** Per-Eintrag-Aktionen Inline am Zeilenende: Convert / Edit / Delete / Skip. Convert + Delete ohne extra Confirm; Edit öffnet Modal; Skip ist instant.
- **D-13:** Bulk-Convert: Per-Group-Section-Action "Alle in dieser Gruppe konvertieren" (1 Click → bulk-Endpoint).
- **D-14:** Pro Drift-Row: ISO-Datum + Wochentag-Code + Amount + Reason-Code (Hover Tooltip mit `reason_text`); Suggested-Action sichtbar.

**Commit-Sicherheit & Idempotenz-UX:**

- **D-16:** Erfolgs-Page mit Summary + Idempotenz-Hinweis + Backup-Pfad-Hinweis; Stages werden read-only.

### Pattern 1: Backend Convert-Endpoint (atomic-tx)

**What:** Atomic Tx kombiniert `extra_hours.soft_delete_bulk` + `absence_dao.create` + `cutover_dao.upsert_migration_source` in einer Tx; bei Fehler vollständiger Rollback.

**When to use:** Single + Bulk Convert (D-01 + D-02).

**Example** (skeleton, not final code):

```rust
// Source: pattern verified against service_impl/src/cutover.rs::commit_phase + migrate_legacy_extra_hours_to_clusters
impl<Deps: CutoverServiceDeps> CutoverServiceImpl<Deps> {
    pub async fn convert_quarantine_entry(
        &self,
        extra_hours_id: Uuid,
        context: Authentication<Self::Context>,
        tx: Option<Self::Transaction>,
    ) -> Result<ConvertQuarantineEntryOutcome, ServiceError> {
        // 1. Permission gate (D-23: cutover_admin only for Commit-Class actions)
        self.permission_service
            .check_permission(CUTOVER_ADMIN_PRIVILEGE, context.clone())
            .await?;

        // 2. Open Tx (single-Tx atomicity, D-Phase4-14 pattern)
        let tx = self.transaction_dao.use_transaction(tx).await?;

        // 3. Read the legacy row (must NOT be already soft-deleted — idempotency
        //    check via extra_hours.deleted IS NULL, mirrors existing DAO behaviour)
        let row = self
            .cutover_dao
            .find_legacy_extra_hours_not_yet_migrated(tx.clone())
            .await?
            .iter()
            .find(|r| r.id == extra_hours_id)
            .ok_or(ServiceError::NotFound)?
            .clone();

        // 4. Pre-fetch contract for sp (mirrors migrate_legacy_extra_hours_to_clusters
        //    step 2)
        let work_details = self
            .employee_work_details_service
            .find_by_sales_person_id(row.sales_person_id, Authentication::Full, Some(tx.clone()))
            .await?;

        // 5. Reuse Plan 08-09 helpers for range-auto-compute. NOTE: detect_weekly_lump_sum
        //    et al. are currently file-private — Plan-Phase MUST bump to pub(crate) before
        //    calling from this new method. (See "Common Pitfalls" → P-04.)
        let (from, to) = detect_weekly_lump_sum(&row, &all_rows, |d| {
            lookup_active_contract(&work_details, d)
        })
        .ok_or_else(|| ServiceError::ValidationError(
            "Row does not match weekly-lump-sum heuristic; manual edit required".into()
        ))?;

        // 6. Insert absence_period directly (skip AbsenceService.create — it has
        //    Forward-Warning loop + Permission-Layer; we are privileged here per
        //    D-Phase4 anti-pattern guidance — same as commit_phase does)
        let absence_period_id = Uuid::new_v4();
        let entity = AbsencePeriodEntity {
            id: absence_period_id,
            logical_id: absence_period_id,
            sales_person_id: row.sales_person_id,
            category: extra_hours_category_to_absence(&row.category),
            from_date: from,
            to_date: to,
            description: Arc::from(""),
            created: now_primitive_datetime(),
            deleted: None,
            version: Uuid::new_v4(),
        };
        self.absence_dao
            .create(&entity, CUTOVER_MIGRATION_PROCESS, tx.clone())
            .await?;

        // 7. Tag the migration-source mapping (idempotency for re-runs)
        self.cutover_dao
            .upsert_migration_source(
                &MigrationSourceRow {
                    extra_hours_id,
                    absence_period_id,
                    cutover_run_id: Uuid::new_v4(), // synthetic run-id for ad-hoc convert
                    migrated_at: now_primitive_datetime(),
                },
                tx.clone(),
            )
            .await?;

        // 8. Soft-delete the legacy extra_hours row (Auth::Full per cutover convention)
        self.extra_hours_service
            .soft_delete_bulk(
                Arc::from([extra_hours_id]),
                CUTOVER_MIGRATION_PROCESS,
                Authentication::Full,
                Some(tx.clone()),
            )
            .await?;

        // 9. Optional: build refreshed_drift_report inline — saves 1 FE roundtrip per
        //    Resolve-Action (D-08). Plan-Phase decides ON/OFF; Research recommends ON.
        let refreshed = self
            .compute_gate(/* synthetic run-id */, /* now */, /* dry_run = true */, &Default::default(), tx.clone())
            .await?;
        let inline_drift_report = build_gate_drift_report(/* ... */);

        // 10. Atomic commit
        self.transaction_dao.commit(tx).await?;

        Ok(ConvertQuarantineEntryOutcome {
            absence_period_id,
            deleted_extra_hours_id: extra_hours_id,
            refreshed_drift_report: inline_drift_report,
        })
    }
}
```

### Anti-Patterns to Avoid

- **`window.confirm` für Commit-Confirm verwenden:** Phase 8 hat das explizit verboten (siehe Plan 08-05 + `shifty-dioxus/CLAUDE.md` "Common Issues"). Nutze `Dialog(DialogVariant::Center)` für TypeToConfirm + EditExtraHoursModal.

- **Backend-`AbsenceService::create()` für Convert nutzen:** Forward-Warning-Loop ist hier unerwünscht. Direkter `absence_dao.create()` mit `CUTOVER_MIGRATION_PROCESS`-Tag.
- **Heuristik-Helper duplicieren:** `detect_weekly_lump_sum` ist die canonical Logik aus Plan 08-09. Plan-Phase muss sie auf `pub(crate)` heben statt zu re-implementieren.
- **`format!`-basierte Tailwind-Klassen-Konstruktion:** Pitfall 5 aus Plan 08-05. Per-Variant-`match`-Arm mit STATIC-Strings (siehe `CategoryBadge` in `absences.rs:111-128`).
- **Defensive Uuid::nil im Create-Body vergessen:** Plan-08-04-W-7 fordert `body.id = Uuid::nil(); body.version = Uuid::nil();` als ersten Schritt jeder POST-Funktion in `api.rs`.
- **Backend liefert Skip-State / Bulk-Failed-Rows ungeordnet:** Sortierung nach `(date, extra_hours_id)` analog `compute_gate` (Plan 08-08, `cutover.rs:776-781`).
- **`git commit`-Befehle in Plans:** Verboten (CLAUDE.local.md). Nur `jj describe` / `jj new`. Plans dokumentieren WAS committet wird, User entscheidet WANN.

## Common Pitfalls

### P-01: `detect_weekly_lump_sum` als file-private fn nicht aufrufbar von neuen Methoden auf demselben `impl`-Block

**What goes wrong:** Plan-Phase schreibt `convert_quarantine_entry` in `service_impl/src/cutover.rs` und ruft `detect_weekly_lump_sum(...)` auf — kompiliert.

**Why it happens:** Free-standing fns am Modulende sind im selben Crate aufrufbar. ABER wenn die Plan-Phase entscheidet, einen Sub-Service in einer separaten Datei zu bauen, müssen die Helper auf `pub(crate)` gehoben werden ODER über das Modul re-exportiert.

**How to avoid:** Plan-Phase entscheidet **direkt auf `CutoverServiceImpl`** (Empfehlung: ja, direkt). Falls Sub-Service: erste Task ist `pub(crate)`-Bump auf `detect_weekly_lump_sum`, `iso_week_range`, `lookup_active_contract` plus `MigratedCluster`-Struct + `extra_hours_category_to_absence`-Helper.

**Warning signs:** `cargo check --workspace` schlägt mit `function 'detect_weekly_lump_sum' is private` fehl.

[VERIFIED: `service_impl/src/cutover.rs:1064-1148` zeigt `fn` ohne `pub`-Modifier]

### P-02: Convert-Endpoint-Tx vs. dem `synthetic run-id`-Issue

**What goes wrong:** Convert-Endpoint braucht intern eine `cutover_run_id: Uuid` für das Migration-Source-Mapping (Idempotenz). Aber er gehört zu KEINEM existing Cutover-Run.

**Why it happens:** `MigrationSourceRow.cutover_run_id` ist Pflichtfeld; das Schema setzt es nicht NULL.

**How to avoid:** Plan-Phase generiert per Aufruf ein `Uuid::new_v4()` als synthetic run-id; das ist semantisch okay weil der run-id eh nur für Audit-Trail-Zwecke existiert. Alternative: separate Spalte `synthetic_run_origin: enum {automated_run, ad_hoc_convert}` — Plan-Phase entscheidet (Research empfiehlt: einfach synthetic Uuid + null-Override-Free).

**Warning signs:** Bulk-Convert-Tests zeigen alle Rows einer Bulk-Operation mit unterschiedlichen run_ids → falls das stört, derselben `Uuid::new_v4()` für alle Rows einer Bulk-Tx.

### P-03: Inline-Drift-Report-Re-Compute in derselben Tx ist subtil

**What goes wrong:** Plan-Phase calls `compute_gate(...)` innerhalb der Convert-Tx, vor `commit(tx)`. `compute_gate` schreibt selbst eine JSON-Datei nach `.planning/migration-backup/cutover-gate-{ts}.json` — das ist ein Side-Effect ohne Rollback.

**Why it happens:** `compute_gate` ist nicht für ad-hoc Re-Compute designed; es persistiert ein Audit-File.

**How to avoid:** Plan-Phase trifft eine von 3 Optionen: (a) inline `refreshed_drift_report` ohne Audit-File-Write — neuer Helper `compute_gate_in_memory_only`, der `compute_gate` bis kurz vor dem File-Write durchläuft und die Daten zurückgibt; (b) Frontend macht separaten `gate-dry-run`-Roundtrip nach Convert (D-08-Roundtrip-Cost akzeptiert); (c) inline-Re-Compute mit File-Write akzeptiert (jeder Convert produziert eine Audit-Datei).

**Warning signs:** Test-Suite generiert eine Inflation an `cutover-gate-*.json`-Dateien unter `.planning/migration-backup/`.

```rust
// Source: synthesized from service_impl/src/cutover.rs::commit_phase + Plan 08-08 bucket-map pattern
pub async fn bulk_convert_quarantine_rows(
    &self,
    sales_person_id: Uuid,
    category: AbsenceCategory,
    year: u32,
    explicit_ids: Option<Arc<[Uuid]>>, // optional — D-02 erwähnt "extra_hours_ids: Vec<Uuid>" als Multi-Selection-Erweiterung
    context: Authentication<Self::Context>,
    tx: Option<Self::Transaction>,
) -> Result<BulkConvertOutcome, ServiceError> {
    self.permission_service
        .check_permission(CUTOVER_ADMIN_PRIVILEGE, context.clone())
        .await?;

    let tx = self.transaction_dao.use_transaction(tx).await?;

    // Bucket-key match (Plan 08-08 pattern):
    //   - sp_id matches
    //   - extra_hours.category maps to AbsenceCategory equal to `category`
    //   - extra_hours.date_time.date().year() == year
    //   - id ∈ explicit_ids  (if Some)
    let all = self.cutover_dao
        .find_legacy_extra_hours_not_yet_migrated(tx.clone())
        .await?;
    let target_rows: Vec<&LegacyExtraHoursRow> = all
        .iter()
        .filter(|r| r.sales_person_id == sales_person_id)
        .filter(|r| matches_category(&r.category, category))
        .filter(|r| (r.date_time.date().year() as u32) == year)
        .filter(|r| match &explicit_ids {
            Some(ids) => ids.contains(&r.id),
            None => true,
        })
        .collect();

    if target_rows.is_empty() {
        return Err(ServiceError::NotFound);
    }

    let work_details = self
        .employee_work_details_service
        .find_by_sales_person_id(sales_person_id, Authentication::Full, Some(tx.clone()))
        .await?;

    let synthetic_run_id = Uuid::new_v4();
    let now = now_primitive_datetime();
    let mut converted_absence_periods: Vec<Uuid> = Vec::new();
    let mut deleted_extra_hours_ids: Vec<Uuid> = Vec::new();
    let mut failed_rows: Vec<(Uuid, String)> = Vec::new();

    for row in target_rows.iter() {
        match detect_weekly_lump_sum(row, &all, |d| lookup_active_contract(&work_details, d)) {
            Some((from, to)) => {
                let absence_period_id = Uuid::new_v4();
                let entity = AbsencePeriodEntity {
                    id: absence_period_id,
                    logical_id: absence_period_id,
                    sales_person_id,
                    category: extra_hours_category_to_absence(&row.category),
                    from_date: from,
                    to_date: to,
                    description: Arc::from(""),
                    created: now,
                    deleted: None,
                    version: Uuid::new_v4(),
                };
                self.absence_dao
                    .create(&entity, CUTOVER_MIGRATION_PROCESS, tx.clone())
                    .await?;
                self.cutover_dao
                    .upsert_migration_source(
                        &MigrationSourceRow {
                            extra_hours_id: row.id,
                            absence_period_id,
                            cutover_run_id: synthetic_run_id,
                            migrated_at: now,
                        },
                        tx.clone(),
                    )
                    .await?;
                converted_absence_periods.push(absence_period_id);
                deleted_extra_hours_ids.push(row.id);
            }
            None => {
                // Decision: strict atomic (P-10) → return 422 with failed_rows;
                // OR: collect failure, continue. Plan-Phase decides.
                failed_rows.push((row.id, "weekly-lump-sum heuristic mismatch".into()));
            }
        }
    }

    // Strict-atomic option (P-10 Empfehlung):
    if !failed_rows.is_empty() {
        // Tx wird vom Caller verworfen wenn ServiceError zurückkommt (use_transaction kontrakt)
        return Err(ServiceError::ValidationError(
            format!("{} rows did not match heuristic", failed_rows.len())
        ));
    }

    self.extra_hours_service
        .soft_delete_bulk(
            Arc::from(deleted_extra_hours_ids.clone().into_boxed_slice()),
            CUTOVER_MIGRATION_PROCESS,
            Authentication::Full,
            Some(tx.clone()),
        )
        .await?;

    self.transaction_dao.commit(tx).await?;

    Ok(BulkConvertOutcome {
        converted_absence_periods,
        deleted_extra_hours_ids,
        // Inline refreshed gate (P-03 option a):
        refreshed_drift_report: /* compute_gate_diagnostic(...) */,
    })
}
```

```rust
pub async fn cutover_convert_quarantine_entry(
    config: Config,

        info!("Convert returned 422 Validation: {}", text);

    }
    response.error_for_status_ref()?;
    let result = response.json().await?;
    info!("Converted");
    Ok(result)
```

## Open Questions (RESOLVED)

1. **Inline `refreshed_drift_report` vs. separater Roundtrip nach Convert?**
   - What we know: D-08 fordert Auto-Re-Run. Inline-Pattern spart 1 Roundtrip pro Aktion.
   - What's unclear: Backend-File-Write-Inflation bei vielen Aktionen pro Session; Server-CPU-Cost bei großen Drift-Sets.

2. **Bulk-Convert-Failure-Semantik?**

### Known Threat Patterns for {stack}

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| SQL Injection auf `extra_hours_id`-Input | Tampering | sqlx prepared statements (workspace-canonical) — keine string-Konkatenation |
| Privilege Escalation: HR-User triggert Commit | Elevation of Privilege | Backend `permission_service.check_permission(CUTOVER_ADMIN_PRIVILEGE)` als FIRST step im Convert/Bulk-Service-Method. Frontend-Disabled-Button ist NICHT die einzige Verteidigung. |
| Idempotency-Replay: Convert wird zweimal für dieselbe extra_hours_id getriggert (z.B. doppel-click) | Tampering | `extra_hours.deleted IS NULL`-Check + `cutover_dao.upsert_migration_source` mit `ON CONFLICT(extra_hours_id) DO NOTHING`; zweiter Aufruf ist no-op |
| Concurrent Cutover-Run + ad-hoc Convert: Race-Condition zwischen Bulk-Cutover-Run und Convert-Call | Tampering / Tx-Konflikt | Beide nutzen denselben Tx-Pattern; SQLite serialisiert Writes innerhalb einer Connection. Falls Production auf Postgres migriert: Plan-Phase prüft Isolation-Level (REPEATABLE READ minimum). |
| Cross-Site Request Forgery auf Commit-Endpoint | Spoofing | Existing Auth-Middleware (cookie + token); Backend rejected bei fehlendem Auth-Context — testbar via `oneshot` ohne `Extension(context)`. |
| OpenAPI-Schema-Drift: neuer Endpoint nicht in `EXPECTED_PATHS` | Tampering (Surface-Drift) | `rest/tests/openapi_surface.rs` als gate; Plan-08-08 Deviation 1 als Pattern. |
| Frontend-Backdoor: Direct-URL-Access `/admin/cutover` ohne Privileg | Elevation of Privilege | Plan-Phase implementiert Page-Internal-Privilege-Check (siehe `page/absences.rs:has_hr`-Pattern); 403-Render-State falls Privileg fehlt; Backend-Endpoint sind unabhängig privilege-gated. |
| Audit-Log-Schwund bei ad-hoc Convert (kein cutover_run_id-Trail) | Repudiation | `synthetic_run_id` + `cutover_dao.upsert_migration_source`-Tag mit `update_process = 'phase-4-cutover-migration'` (existing) → Audit-Trail bleibt intakt. |
