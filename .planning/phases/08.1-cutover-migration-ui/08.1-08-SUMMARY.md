## Decisions Made

1. **`PartialEq` not derived on `CutoverWizardState`.** The nested `rest-types` DTOs (`CutoverProfileTO`, `CutoverGateDriftReportTO`) do not derive `PartialEq` because their internal payloads include float / arc / date types without `Eq`-friendly semantics. Page-level reactivity in Plan 09 will read the store via `*CUTOVER_STORE.read()` and match on individual fields (`stage`, `last_dry_run.as_ref()`, etc.) — never on the full struct via `==`. Adding `PartialEq` would require either a wrapper (extra layer of indirection) or a `PartialEq`-derive contribution to the rest-types crate (out-of-scope for this plan; affects a foreign crate). `RunSummary` does derive `PartialEq + Eq` (its 5 fields are all `u32` / `Option<String>`-friendly). `WizardStage` already derives `PartialEq + Eq` per its `enum`-of-units shape.

2. **`apply_skip(id)` as a pure non-async helper.** Plan-Body's Test 2 sketch suggested a `tokio::runtime::Builder::new_current_thread().enable_all().build().unwrap().block_on(async { ... process_action(Skip(id), cfg).await })` invocation. That pulls in the tokio dependency just for one test path AND requires constructing a `Config` value (the existing `Config` impl-derives `Default`, so that part works), but more importantly it muddies the test by routing through an async-await chain whose only meaningful effect is the synchronous skipped-set append. Cleaner: extract the append logic into `apply_skip(id)` at module scope; the unit test calls that directly inside a `VirtualDom::new(assertion_app)` block (the runtime that `CUTOVER_STORE.write()` requires); the production async arm calls the same helper from inside its `match` branch. One source of truth, no async indirection in the test.

3. **`apply_skip` resets `skipped_extra_hours_ids` to a fresh empty `Arc<[Uuid]>` before asserting** — not strictly necessary for the assertion (`.iter().any(|x| *x == id)` works even on a multi-element vector), but defensive against cross-test global-state pollution. Future tests in the same file that also touch `CUTOVER_STORE` will start from a clean slate.

4. **`use_coroutine` short-path matches existing absence_service entry.** `app.rs` opens with `use crate::service;` so the canonical idiom for in-file coroutine registrations is `use_coroutine(service::cutover::cutover_service)` (NOT `use_coroutine(crate::service::cutover::cutover_service)`). The plan-body's done-criteria grep used the long form (`crate::service::cutover::cutover_service`); that grep would have failed against the corrected short form. Adjusted to match the established style — one less style-deviation to flag in code review. The verify table above uses the corrected form.

5. **`delete_extra_hour` Reqwest→ShiftyError lifting via `.map_err(ShiftyError::from)`.** The existing `delete_extra_hour` fn returns `Result<(), reqwest::Error>` (legacy signature from before Plan 04). The coroutine's `process_action` returns `Result<(), ShiftyError>`. `ShiftyError` already derives `#[from]` for `reqwest::Error`, so the conversion is a one-liner via `map_err(ShiftyError::from)`. Out-of-scope for this plan to refactor `delete_extra_hour` itself (would touch the broader `extra_hours` page; Plan 05 deliberately left it untouched per its scope of only adding new `cutover_*` fns).

6. **Edit / Delete trigger an explicit `gate-dry-run` Re-Run.** `update_extra_hour` and `delete_extra_hour` do NOT carry an inline `refreshed_drift_report` (they belong to the existing `/extra-hours` surface, not the new Cutover surface that was extended in Plan 04). The coroutine therefore explicitly calls `api::cutover_gate_dry_run` immediately after the mutation in those branches and writes its `gate_drift_report` into the store + bumps the refresh-token. This costs one extra HTTP roundtrip per Edit / Delete (versus the inline pattern Convert/Bulk uses), but mirrors what D-08 prescribes and is cheaper than extending the legacy extra-hours endpoints just for this UI flow.

### Auto-fixed Issues

**1. [Rule 1 - Bug] `#[derive(PartialEq)]` on `CutoverWizardState` breaks compile (E0369).**

- **Found during:** Task 1 first compile.
- **Issue:** `cargo test --bin shifty-dioxus state::cutover_state` failed with two `E0369` errors:
  - `binary operation '==' cannot be applied to type 'std::option::Option<CutoverProfileTO>'`
  - `binary operation '==' cannot be applied to type 'std::option::Option<CutoverGateDriftReportTO>'`
  Plan-body's pattern-skeleton specified `#[derive(Clone, Debug, Default, PartialEq)]` on `CutoverWizardState`, but `CutoverProfileTO` and `CutoverGateDriftReportTO` (from `rest-types`) do not derive `PartialEq`. The auto-derive on the enclosing struct cascades and hits the inner `Option<...>::eq` requirement.
- **Fix:** Removed `PartialEq` from the `CutoverWizardState` derive set. Doc-string above the struct explains why and what the page does instead (`*CUTOVER_STORE.read()` field-level matching, never full-struct `==`). `RunSummary` was tightened to `Clone, Debug, PartialEq, Eq` (its 5 fields are all `Eq`-friendly — `u32` / `Option<String>`).
- **Files modified:** `shifty-dioxus/src/state/cutover_state.rs`.
- **Verification:** `cargo test --bin shifty-dioxus state::cutover_state` exit 0 with all 3 tests passing.
- **Committed in:** `2fd95d2d` (Task 1 commit).

This is a Rule 1 auto-fix (bug — code didn't work as intended). Plan-body's pattern-skeleton over-specified `PartialEq` based on `state/absence_period.rs`'s pattern, which works for absence because every nested DTO in `AbsencePeriod`'s state-mirror derives `PartialEq` (the DTOs are simpler — `AbsenceCategoryTO` is a unit-variant enum, dates and Uuids are `Eq`). The Cutover wizard-state has structurally richer DTOs without `PartialEq`. Page-level reactivity does not need full-struct `==`; field-level matching is the canonical Dioxus idiom anyway.

### Plan-Body Test 2 Skeleton Adjustment (Decision, not a deviation)

The plan-body sketched Test 2 (`cutover_action_skip_appends_to_skipped_set`) using a `tokio::runtime::Builder::new_current_thread().enable_all().build().unwrap().block_on(async { ... process_action(...).await })` pattern. That works but is heavier than necessary — the Skip branch has zero `.await` calls, only a global-signal write. We extracted the append logic to a sync `apply_skip(id)` helper at module scope and the test calls it directly inside a `VirtualDom::new(assertion_app)` block (which provides the Dioxus runtime). The production `process_action::Skip(id)` arm calls the same helper. Cleaner test, single source of truth, no extra tokio invocation. The plan-body explicitly allowed this kind of decision via "If `Config::default()` is not impl'd, find the constructor pattern" guidance (Config does derive Default; we sidestepped the question entirely by not constructing a Config in the test). Documented under Decisions Made above (#2). Not flagged as a Rule-1/2/3 deviation — the plan's stated behavior outcome (Skip appends to the skipped set) is verified.
