# Phase 8.1 — 8.1-Eigener Cutover-UI UAT

> Independent UAT for the Cutover-Migration-UI itself (`/admin/cutover`).
>
> **Scope:** wizard-stages, all four per-row actions, bulk-convert, type-to-confirm,
> privilege gate, post-commit idempotenz state, error toast path.
>
> **Out of scope:** the 35-step Phase-8-HUMAN-UAT (post-Cutover absences-page UAT)
> runs separately as Plan 08.1-12 (Subsumption). Per D-21 the two UATs are
> independent so the Cutover-UI itself is verified before the dependent
> Phase-8-UAT consumes its output.

## Setup

- [ ] **0.1** Backend running on int with admin-auto-grant trigger (Plan 08-07)
      so the DEV-/SSO-User has all privileges including `cutover_admin`.
      Verify via:
      `curl -H 'Authorization: ...' /feature-flag/absence_range_source_active`
      → must return `{"key":"absence_range_source_active","enabled":false}`
      (Cutover not yet committed).
- [ ] **0.2** Frontend deployed to int (matching backend version).
- [ ] **0.3** Two test users available:
  - **`cutover_admin_user`** with `cutover_admin` privilege (DEVUSER on int via
    auto-grant trigger, or explicitly granted SSO user).
  - **`hr_only_user`** with `hr` privilege ONLY (no `cutover_admin`).
- [ ] **0.4** At least 1 quarantined `extra_hours` row exists in the int
      database. If `gate-dry-run` reports `quarantined_rows == 0`, seed one
      manually:
      - Easiest: insert a fractional Vacation entry (e.g. `13.33h` on a single
        workday) for a known sales person via the existing Extra-Hours page —
        Plan 08-09's `detect_weekly_lump_sum` will reject it and Plan 08-08's
        `compute_gate` will flag it as `FractionalAmount`.
      - Re-run `POST /admin/cutover/gate-dry-run` and confirm
        `quarantined_rows >= 1`.
- [ ] **0.5** Browser dev-tools open (F12) — needed for step 9.1 (manual fetch).

> **Pre-condition note:** The UAT is most useful with at least one quarantined
> row to be useful — if not, run `gate-dry-run` and seed test data via the
> existing Extra-Hours UI / REST.

## 4. Per-Row Actions (D-12)

#### 4.1 Convert (single)
expected: Click "Eintrag konvertieren" (`CutoverRowBtnConvert`) on a row whose `reason_code` is heuristic-friendly (e.g. a strict-match cluster). Action button enters loading-state (replaced with `"..."` spinner). Within 1-2s the row disappears from the list, the `quarantined_rows` count in Stage 1 / Stage 2 stat-boxes decreases by 1, and (if the group becomes empty) the group section also disappears. No error toast.
result: [pending]

#### 4.2 Edit
expected: Click "Bearbeiten" (`CutoverRowBtnEdit`) on a row → an Edit modal opens centered on screen. The modal renders ONLY two inputs: `Betrag (h)` (`CutoverEditAmountLabel`) and `Datum` (`CutoverEditDateLabel`) — D-04 forbids `Kategorie` / `Beschreibung`. Change Amount (e.g. `13.33` → `16.0`); click "Änderungen speichern". Modal closes; the drift list reflects the new amount on the same row OR the row vanishes if the new amount no longer triggers a quarantine reason.
result: [pending]

#### 4.3 Delete
expected: Click "Löschen" (`CutoverRowBtnDelete`) on a row → row disappears immediately (no confirm dialog per D-12, no `window.confirm`). `quarantined_rows` count decreases by 1. Auto-Re-Run triggers a refreshed drift report.
result: [pending]

#### 4.4 Skip
expected: Click "Eintrag überspringen" (`CutoverRowBtnSkip`) → row disappears immediately from the local UI list. CRITICAL: `quarantined_rows` count is **UNCHANGED** (Skip is frontend-only per D-05; no backend call). Reload the page (F5) → the skipped row reappears at the same position.
result: [pending]

## 5. Bulk-Convert (D-13)

#### 5.1 Bulk-Convert on a clean group (>= 2 rows, all heuristic-friendly)
expected: Pick a group with ≥ 2 rows where all rows would convert successfully. Click "Alle in Gruppe konvertieren" (`CutoverBtnBulkConvert`). All rows in the group convert atomically in a single Tx. Group section disappears entirely; `quarantined_rows` count decreases by exactly the group size. No error toast.
result: [pending]

#### 5.2 Bulk-Convert with at least one row that fails the heuristic
expected: Pick (or seed) a group containing a fractional row plus a heuristic-matching row. Click "Alle in Gruppe konvertieren". Backend returns `422` with strict-atomic semantics: NO rows are converted, an error toast renders the validation error inline at the top of the stage panel (`CutoverErrorApiFailure` or backend reason), and ALL rows of the group remain in the list unchanged.
result: [pending]
