## Decisions Made

1. **AbsenceCategoryTO serialization is PascalCase.** Plan 08.1-PATTERNS.md hinted at `#[serde(rename_all = "snake_case")]` but the live `AbsenceCategoryTO` enum (lines 1567–1572) carries no such attribute and serializes its variants as `"Vacation"`, `"SickLeave"`, `"UnpaidLeave"`. The triple-only deserialization test was adjusted accordingly with an inline comment so future readers don't reintroduce the wrong literal.
2. **Five tests instead of three.** Plan body listed three behaviors; I added two more bulk-request tests (round-trip with explicit subset + triple-only default) because they validate the dual-mode semantics from D-02 (subset OR full-triple). Cheap, additive, no scope creep.
3. **`From`-impls deferred to Plans 02/03.** Per plan instruction, the `From<&service::cutover::ConvertQuarantineEntryOutcome>` impls are NOT in this plan — they live alongside the service Outcome types created in the next two plans.

**2. [Rule 1 — Bug] Corrected `AbsenceCategoryTO` serde-tag in test JSON literal from `"vacation"` to `"Vacation"`**
- **Found during:** Initial test run after Task 1 implementation.
- **Issue:** First test iteration used snake_case `"vacation"` based on the pattern hint in 08.1-PATTERNS.md (line 286: `#[serde(rename_all = "snake_case")]`). Real `AbsenceCategoryTO` (lib.rs:1567) has no rename attribute and serializes `Vacation` literally — test failed with `unknown variant 'vacation'`.
- **Fix:** Changed JSON literal in `bulk_request_defaults_extra_hours_ids` to `"Vacation"`; added an inline comment naming the actual serde behavior so future readers do not reintroduce the wrong literal.
- **Files modified:** `rest-types/src/lib.rs`.
- **Verification:** All 5 tests pass.
- **Committed in:** `29ebffb8` (Task 1 commit, fix is part of the same change).
