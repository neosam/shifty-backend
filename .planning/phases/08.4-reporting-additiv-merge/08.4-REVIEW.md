# Phase 08.4: Code Review Report

**Reviewed:** 2026-06-09
**Depth:** standard
**Files Reviewed:** 11 (2 listed files were *deleted* this phase — `reporting_flag_off_bit_identity.rs`, `reporting_flag_on_integration.rs` — so 11 of 13 exist)
**Status:** issues_found

## Summary

The phase delivers the additive merge of `absence_period`-derived hours into the
three report methods, the symmetric `expected_hours`/`balance_hours` reduction
(Gap 2 / WR-01), and the snapshot schema bump 5 → 6. The additive merge itself
is correct in `get_report_for_employee_range` (display + balance via
`hours_per_week`), and the year-bounds clamping in `get_reports_for_all_employees`
is sound and test-covered. The version bump 5 → 6 is correctly justified: balance
and expected value_types now move, which invalidates v5 re-validation. The new
additive-merge test suite passes (9/9).

However, the **balance-symmetry guarantee is not uniform across the three report
methods**. The Gap-2 reduction was implemented by three separate, copy-pasted code
paths, and `get_reports_for_all_employees` subtracts a *year-lump* of derived
absence from an aggregate that already zeroed extra-hours absence for
dynamic / zero-expected weeks. This produces a genuine asymmetry between an
`extra_hours` vacation and an equivalent `absence_period` vacation for dynamic
employees — exactly the parity the phase set out to guarantee. The parity test
covers only the non-dynamic `get_report_for_employee_range` path, so this slips
through CI green. Details below (CR-01).

Two secondary asymmetries (vacation_days, ungated `get_week` reduction) plus a
stale doc comment round out the findings.

## Critical Issues

### CR-01: Balance symmetry breaks for dynamic / zero-expected weeks in `get_reports_for_all_employees`

**File:** `service_impl/src/reporting.rs:403-408` (interacts with the dynamic branch `248-273`)
**Issue:**
The Gap-2 reduction in `get_reports_for_all_employees` subtracts a single
year-total `absence_derived_total` from the aggregated `planned_hours`:

```rust
let absence_derived_total =
    absence_derived_vacation_hours + absence_derived_sick_leave_hours + absence_derived_unpaid_leave_hours;
let expected_hours = weekly_hours.planned_hours - weekly_hours.absense_hours - absence_derived_total;
```

`weekly_hours.absense_hours` (extra-hours absence) is **forced to `0.0` for any
week where `expected_hours <= 0.0`** (the dynamic / no-contract branch, lines 263
and 248-273), and in that branch `planned_hours` is set to `overall_hours`
(shiftplan + extra_work). But `absence_derived_total` is **not** gated the same
way — `derive_hours_for_range` returns positive hours for a dynamic contract
(`hours_per_day() = expected_hours / workdays_per_week`, no `is_dynamic` check;
see `service/src/employee_work_details.rs:106` and `absence.rs:490`).

Consequence: for a dynamic employee (or any week with `expected_hours <= 0`) who
has an `absence_period` vacation, `expected_hours` is pushed below `overall_hours`,
so `balance_hours = overall_hours - expected_hours + carryover` becomes **positive**
instead of the dynamic-invariant `~0`. The same vacation booked as a live
`extra_hours` row is zeroed by the dynamic branch and does NOT move balance. Two
semantically identical absences therefore yield different balances depending on
source — the precise non-parity WR-01 was meant to eliminate.

This is the third, hand-rolled copy of the reduction logic; the per-week,
properly-gated version lives in `hours_per_week` (`reporting.rs:988-997`, gated on
`working_hours_for_week > 0.0`) and is correct. The aggregate path does not reuse
it.

The balance-parity test (`reporting_additive_merge.rs:676`) exercises only
`get_report_for_employee_range` with a **non-dynamic** fixture
(`fixture_work_details_8h_mon_fri`, `is_dynamic: false`), so neither the dynamic
case nor the `get_reports_for_all_employees`/`get_week` paths are covered. The bug
is invisible to the current suite.

**Fix:** Gate the derived reduction per-week and on contract presence, mirroring
`hours_per_week`. Split the derived map by ISO-week and, inside the existing
per-week fold, subtract a week's derived sum only in the `expected_hours > 0.0`
branch (contribute `0.0` in the `<= 0.0` branch, exactly like `absense_hours`).
Then add a parity test using a **dynamic** fixture across all three methods
(`get_report_for_employee_range`, `get_reports_for_all_employees`, `get_week`)
asserting `balance_hours` is source-independent (and ~0 for dynamic).

## Warnings

### WR-01: `get_week` derived reduction is ungated and can drive `expected_hours` negative

**File:** `service_impl/src/reporting.rs:818, 822-824`
**Issue:**
```rust
let expected_hours = planned_hours - abense_hours - absence_derived_total;
let dynamic_hours = dynamic_hours - abense_hours - absence_derived_total;
let balance_hours = overall_hours - expected_hours;
```
For a dynamic employee `planned_hours` from `weight_for_week` is `0.0`, so
`expected_hours = -absence_derived_total` (negative) and `balance_hours` is
inflated by the full derived amount. Unlike the range path, `get_week` never
applies the `working_hours_for_week > 0` gate. `abense_hours` (extra-hours
absence) is also ungated here, so derived is symmetric with extra_hours *within
`get_week`* — but `get_week` is then asymmetric with the range method, and a
negative `expected_hours` for a dynamic week is wrong on its face. Same defect
class as CR-01; fix together.
**Fix:** Apply the contract-presence gate from `hours_per_week`: when the
week's contract hours are `<= 0`, contribute `0` derived reduction (and treat
`expected_hours` as `overall_hours` for the dynamic case) so derived and
extra_hours both follow the dynamic rule.

### WR-02: `vacation_days` / `sick_leave_days` / `absence_days` ignore absence_period-derived hours

**File:** `service_impl/src/reporting.rs:560-570, 1068-1087` and
`billing_period_report.rs:210-218`
**Issue:**
`EmployeeReport.vacation_hours` now includes `absence_derived_vacation_hours`
(line 629), but `EmployeeReport.vacation_days` is summed from `by_week`, whose
per-week `vacation_hours` (line 1068) is computed **only** from
`filtered_extra_hours_list` — derived absence is never added to the per-week
day-bearing fields (it only reduces `expected_hours`/`dynamic_hours` at
1056-1057, and `*_days()` derive from hours per `service/src/reporting.rs:118-143`).
Result: a pure `absence_period` vacation contributes to `vacation_hours` but
contributes `0` to `vacation_days` / `absence_days`. This flows directly into the
persisted snapshot: `VacationHours` moves while `VacationDays` does not, producing
internally inconsistent write-once billing-period rows (e.g. 8 vacation_hours, 0
vacation_days). The 08.4-04-SUMMARY explicitly declares this an intentional
out-of-scope deferral, but it is a real data inconsistency, not merely a scope
note, and arguably affects snapshot semantics once fixed.
**Fix:** Feed per-week derived absence into the per-week
`vacation_hours`/`sick_leave_hours`/`unpaid_leave_hours` so `*_days()` derive
correctly, or compute days from the merged hours at report level. If the deferral
stands, add a test pinning the documented (inconsistent) behavior so the gap is
not later mistaken for correct.

### WR-03: Three divergent copies of the merge/reduction logic invite drift

**File:** `service_impl/src/reporting.rs:390-408, 519-528, 804-822, 988-997`
**Issue:** The "iterate `derived.values()`, bucket by `AbsenceCategory`, sum, then
reduce expected/balance" block is duplicated across all three methods with subtly
different gating (range path gates via `hours_per_week`;
`get_reports_for_all_employees` uses a year-lump; `get_week` is ungated). This
duplication is the direct cause of CR-01/WR-01 — the reduction rule was never
centralized, so the three paths drifted.
**Fix:** Extract a shared bucketing helper (e.g.
`fn bucket_derived(&BTreeMap<Date, ResolvedAbsence>) -> (vacation, sick, unpaid)`)
plus a shared per-week reduction helper, and call them from all three methods so
the dynamic/zero-expected gate is applied uniformly.
