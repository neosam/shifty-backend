# Phase 8.4: Code Review Report

**Reviewed:** 2026-06-09
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Phase 8.4 converts the reporting computation for Vacation/SickLeave/UnpaidLeave from a
feature-flag-gated source switch to a permanent additive coexistence: living `extra_hours`
(deleted IS NULL) plus `derive_hours_for_range` absence-period hours are summed. The
`CURRENT_SNAPSHOT_SCHEMA_VERSION` bump (4→5) is correct and well-documented, the `extra_hours`
write-gate removal correctly preserves the surrounding RBAC permission check
(`HR_PRIVILEGE ∨ verify_user_is_sales_person`), per-sales-person scoping is preserved in the
range path, and the per-row double-count protection (DAO `deleted IS NULL` filter, verified in
`dao_impl_sqlite/src/extra_hours.rs`) holds.

However, the additive merge was applied to **only one of the three sibling reporting methods**.
`get_report_for_employee_range` merges absence-period hours; `get_reports_for_all_employees`
(the all-employees overview, REST `GET /report`) and `get_week` (weekly summary, consumed by
`BookingInformationService`) were **not** updated. They still source Vacation/SickLeave/
UnpaidLeave exclusively from `extra_hours`. After cutover, where absence data lives in
`absence_period` and the corresponding `extra_hours` rows are soft-deleted, these two views will
report **zero** vacation/sick/unpaid hours — a data-correctness regression and a source of
inconsistency between the overview and the per-employee drill-down. This is the primary blocker.

Secondary concerns: an asymmetry where absence-period-derived hours feed the display fields but
not `expected_hours`/`balance_hours`, a dead `FeatureFlagService` dependency left in
`ExtraHoursServiceImpl`, a missing same-day overlap test for the double-count scenario, and
several stale comments still describing the removed flag behavior.

## Critical Issues

### CR-01: Additive merge missing in `get_reports_for_all_employees` and `get_week`

**File:** `service_impl/src/reporting.rs:116-393` (`get_reports_for_all_employees`) and
`service_impl/src/reporting.rs:629-786` (`get_week`)
**Issue:** Only `get_report_for_employee_range` (line 477-496) calls
`absence_service.derive_hours_for_range` and adds the derived hours. The other two public
`ReportingService` methods compute `vacation_hours`/`sick_leave_hours`/`unpaid_leave_hours`
exclusively from `extra_hours_array` / `employee_extra_hours` (e.g. lines 290-325 and 684-723).
Neither method invokes `derive_hours_for_range`. After cutover the living `extra_hours` for
these categories are soft-deleted (Phase 8.2) and the data lives only in `absence_period`, so:

- `get_reports_for_all_employees` (REST `GET /report` overview, `rest/src/report.rs:58`) will
  show **0** vacation/sick/unpaid for every employee whose absences were converted.
- `get_week` (weekly summary, `service_impl/src/booking_information.rs:135,254` →
  `get_weekly_summary`) will likewise show 0.

This contradicts the per-employee drill-down (`get_report_for_employee` →
`get_report_for_employee_range`), producing inconsistent numbers across views for the same
person and the same period, and silently dropping legitimate absence hours. The phase goal —
"permanent additive coexistence" — is only half-implemented.
**Fix:** Apply the same additive merge in both methods. In `get_reports_for_all_employees`,
inside the per-`paid_employee` loop, call `derive_hours_for_range` for the employee's full
`[first_day_in_year(year) .. until_week Sunday]` range and add the per-category sums to the
`ShortEmployeeReport` fields, mirroring lines 487-496 and 596-619:
```rust
let derived = self.absence_service
    .derive_hours_for_range(
        ShiftyDate::first_day_in_year(year).to_date(),
        ShiftyWeek::new(year, until_week).as_date(DayOfWeek::Sunday).to_date(),
        paid_employee.id,
        Authentication::Full, // or `context.clone()` per the permission model
        tx.clone(),
    ).await?;
let mut dv = 0.0_f32; let mut ds = 0.0_f32; let mut du = 0.0_f32;
for r in derived.values() {
    match r.category {
        AbsenceCategory::Vacation => dv += r.hours,
        AbsenceCategory::SickLeave => ds += r.hours,
        AbsenceCategory::UnpaidLeave => du += r.hours,
    }
}
// ...then: vacation_hours: weekly_hours.vacation_hours + dv, etc.
```
Apply the analogous change in `get_week` (per `sales_person_id` in the loop). Add a test in
`reporting_additive_merge.rs` covering both methods, not just the range method. If excluding
these methods is intentional, document that decision explicitly and reconcile it with the
"permanent coexistence" goal — but as written it is a regression.

## Warnings

### WR-01: Absence-derived hours feed display fields but not `expected_hours`/`balance_hours`

**File:** `service_impl/src/reporting.rs:487-496, 518-524, 575-624`
**Issue:** In `get_report_for_employee_range`, `absence_derived_vacation_hours` /
`_sick_leave_hours` / `_unpaid_leave_hours` are added only to the displayed `vacation_hours`,
`sick_leave_hours`, `unpaid_leave_hours` fields. `expected_hours`, `dynamic_hours`,
`balance_hours`, `vacation_days`, and `absence_days` are computed from `by_week` /
`hours_per_week`, which receives only `extra_hours` (line 518-524) and never sees the
absence-period data. By contrast, `extra_hours` of `ReportType::AbsenceHours` *do* reduce
`expected_hours` (`reporting.rs:982`). Result: for an absence that exists only as an
`absence_period` (post-cutover), `vacation_hours` is shown but the employee's `balance_hours`
and `expected_hours` are not adjusted as they would be for the equivalent `extra_hours` entry —
two different numbers for economically identical absences, and a balance that no longer
accounts for the absence.
**Fix:** Confirm against the spec whether absence-period vacation/sick is meant to reduce
expected hours (it almost certainly is, to keep balance neutral). If so, feed the derived
per-day hours into the expected-hours / absence-days reduction path (e.g. pass them into
`hours_per_week` or subtract them from `planned_hours` / add to `absence_days`), not just the
display fields. Add a balance-neutrality test for the absence-period-only case mirroring
`test_unpaid_leave_reduces_expected_hours`.

### WR-02: No test for same-day overlap between living `extra_hours` and derived absence

**File:** `service_impl/src/test/reporting_additive_merge.rs:270-310` (Szenario 3)
**Issue:** The phase context explicitly calls out the double-count risk. The test suite covers
"only extra_hours", "only absence_period", "both sources on **distinct** days", and "converted
(soft-deleted) rows". It never tests a living (not-deleted) `extra_hours` Vacation entry on the
**same day** as a derived Vacation entry. That is exactly the coexistence case the additive
model permits (an admin can legitimately create an `extra_hours` Vacation while an
`absence_period` also covers that day — the write-gate removal in D-03 now allows it). The code
adds both unconditionally (no per-day dedup), which is the intended behavior, but it is
unverified and the most likely place for a future regression / accidental dedup to go unnoticed.
**Fix:** Add a test: derived `{2024-06-03: Vacation 8h}` AND a living `extra_hours` Vacation 8h
on `2024-06-03`, asserting `vacation_hours == 16.0`, pinning the intended additive (non-dedup)
semantics for same-day coexistence.

### WR-03: Dead `FeatureFlagService` dependency left in `ExtraHoursServiceImpl`

**File:** `service_impl/src/extra_hours.rs:14, 30` (and `shifty_bin/src/main.rs:866-875,
288-295`)
**Issue:** The D-03 write-gate removal deleted the only consumer of `feature_flag_service` /
`absence_range_source_active` inside `extra_hours.rs`, but the dependency remains: the
`FeatureFlagService` import (line 14), the `gen_service_impl!` dep declaration (line 30), and
the DI wiring in `main.rs:871`. The `gen_service_impl!` macro materializes it as a struct
field, so the compiler will not flag it as unused — it is silent dead weight that adds a
spurious construction-order constraint (main.rs:853-865 still forces `feature_flag_service`
before `extra_hours_service` "for the flag-gate" that no longer exists) and misleads future
readers into thinking the gate is still live.
**Fix:** Remove `FeatureFlagService` from the `ExtraHoursServiceDeps` macro block
(`extra_hours.rs:30`), drop the unused import (line 14), remove the
`type FeatureFlagService = FeatureFlagService;` line from `ExtraHoursServiceDependencies`
(`main.rs:292`) and the `feature_flag_service: feature_flag_service.clone(),` field from the
`ExtraHoursServiceImpl` constructor (`main.rs:871`). Verify the existing `extra_hours` unit
tests are updated to stop wiring the feature-flag mock. Re-run `cargo check --workspace`.

## Info

**Fix:** Update to: "Ab Phase 8.4 additiv: lebende `extra_hours`(UnpaidLeave) PLUS

### IN-02: Stale flag-gate comment in DI wiring

**File:** `shifty_bin/src/main.rs:288-295, 853-858`
**Issue:** Comments still describe the "service-layer flag-gate on POST `/extra-hours`" and that
"ExtraHoursServiceImpl now reads `absence_range_source_active` before each `create()` call" — the
gate was removed in D-03 (`extra_hours.rs:197-203`). The stated construction-order rationale
("FeatureFlagService MUST be constructed BEFORE ExtraHoursService so ... flag-gate") no longer
applies.
**Fix:** Remove or rewrite these comments once WR-03 drops the dead dependency, so the wiring
and its justification match the post-8.4 reality.

### IN-03: `get_reports_for_all_employees` redundant `additional_weeks` past year-end

**File:** `service_impl/src/reporting.rs:181-218`
**Issue:** Not introduced by this phase, but adjacent to the changed aggregation: when
`until_week >= weeks_in_year`, the loop range `0..=until_week + additional_weeks` plus the
`week > weeks_in_year(year)` remap can iterate into the following year's week 1. This is
pre-existing behavior and only noted because CR-01's fix will add absence derivation inside the
same loop; ensure the derived range you pass matches the actual weeks iterated (use the same
`first_day_in_year(year) .. until_week Sunday` bounds, not the carryover week 0 / overflow week)
to avoid pulling cross-year absence into the current-year report.
**Fix:** When implementing CR-01, scope the `derive_hours_for_range` call to the report's real
`[from, to]` year bounds and add a regression assertion that week-0 (previous year) and the
year-end overflow week do not leak absence hours into the wrong report year.
