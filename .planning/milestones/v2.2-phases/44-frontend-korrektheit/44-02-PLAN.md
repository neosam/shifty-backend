<objective>
BUG-02 (v2.2) beheben: die Frontend-API `list_user_invitations` in
`shifty-dioxus/src/api.rs` fängt aktuell einen `serde_json`-Parse-Fehler in
einem `match` ab und retourniert `Ok(Rc::new([]))` — ein silent-empty-Fallback
(inkl. `TODO`-Kommentar). Ein Backend-Response, dessen JSON nicht mehr zum
Frontend-`InvitationResponse`-Schema passt (Feld-Umbenennung, neue Enum-
Variante, etc.), erscheint dadurch im UI als "keine Einladungen vorhanden" —
identisch zum echten leeren Zustand. Kein Trace, keine sichtbare Meldung.

Der Fix propagiert den Parse-Fehler als eigene `ShiftyError::InvitationParse`-
Variante nach oben, so dass (a) `ERROR_STORE` ihn wie jeden anderen Fehler
rendert und (b) `user_details.rs` einen persistierenden Inline-Banner unter
dem `UserInvitations`-Titel zeigt, der leeren Zustand von Parse-Fehler
unterscheidet.

Purpose: Silent-Failure-Bug schließen; leerer Zustand ≠ Fehler-Zustand.
Output: neue ShiftyError-Variante, Fehler-Pfad in api/loader/service/UI, i18n de/en/cs, Regressionstest.
</objective>

<threat_model>
Security-relevant threats: FE-only Änderung, keine Auth-Änderung. Fehler-Text
enthält 200 Zeichen des Response-Body im `InvitationParse(String)` — akzeptable
Ops-Info, da Invitation-JSON nur `id`/`username`/`status`/`redeemed_at`
transportiert (keine Secrets, keine Tokens; `invitation_link` wird bereits im
`fix_invitation_link`-Loader vom Backend re-writeten Anteil transformiert und
kann nur die Backend-URL preisgeben, die im FE ohnehin bekannt ist).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Backend → Frontend | Fehler-Message inkl. Response-Head-Snippet wird im WASM-Client im ErrorStore + Inline-Banner angezeigt. Kein Cross-Origin-Leak (bleibt in Session). |
