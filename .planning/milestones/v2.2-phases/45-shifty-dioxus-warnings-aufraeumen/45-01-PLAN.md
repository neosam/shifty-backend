<task type="auto">
  <name>Task 2: Manuelle Kategorie-Fixes gebündelt</name>
  <files>shifty-dioxus/src/**/*.rs</files>
  <action>
Aus **Backend-Root-Shell** (`nix develop`). Reste aus Task 1 nach Kategorie abarbeiten, pro Kategorie ein Commit.

Vorgehen pro Kategorie:
1. Rest-Warnings filtern: `cd shifty-dioxus && cargo clippy --workspace --tests --message-format=short 2>&1 | grep 'warning:' | grep -oE '\[[a-z_:]+\]' | sort | uniq -c | sort -rn`
2. Größte Kategorie ansehen, alle Callsites via `grep -rn` finden.
3. Fixen — rein syntaktisch, keine Verhaltens-Änderung. Beispiele:
   - `unused_imports` → `use`-Zeile entfernen.
   - `dead_code` auf produktivem Code → prüfen ob tatsächlich tot. Wenn ja: löschen. Wenn nur „vom aktuellen Feature-Set unbenutzt aber vom Feature-Design gedacht": `#[allow(dead_code)] // reason: hooks für zukünftige Feature X gemäß …` — mit knapper Begründung.
   - `dead_code` auf Test-Helpers/Fixtures → `#[allow(dead_code)]` mit `// test fixture, referenced only conditionally` o.ä.
   - `clippy::needless_borrow` → `&x` → `x` (nur wenn Compiler beweist dass Move ok ist).
   - `clippy::redundant_clone` → `.clone()` entfernen (nur wenn Compiler beweist dass Move ok ist; bei Signals/Copy-Types trotzdem `.clone()` behalten wenn Semantik das erfordert).
   - `clippy::let_and_return` → `let x = expr; x` → `expr`.
   - `clippy::unused_async` → wenn tatsächlich kein `.await` im Body: `async` entfernen und Callsites anpassen. Wenn Callsites nicht angepasst werden können (Trait-Signatur): `#[allow(clippy::unused_async)] // signature required by trait X`.
   - `clippy::result_large_err` → wenn `Result<T, ShiftyError>` überall benutzt wird und ShiftyError groß ist, ist Box-Wrap ein API-Bruch — hier `#[allow(clippy::result_large_err)]` auf Modul-Ebene mit Begründung, statt jedem Callsite.
   - `deprecated` → aktuelle API verwenden. Wenn Ersatz breaking oder komplex: als eigenen Todo vermerken und `#[allow(deprecated)]` mit Todo-Referenz.
4. Nach jeder Kategorie:
   - `cargo build -p shifty-dioxus` — 0 Warnings der Kategorie mehr.
   - `cargo build --target wasm32-unknown-unknown` — WASM bleibt grün.
   - Kategorie-Commit: `refactor(45): fix clippy::<kategorie> warnings in shifty-dioxus`.

**Pragmatik-Regel (CONTEXT.md „Specific Ideas"):** Wenn eine Lint-Klasse >50 Callsites betrifft und rein syntaktisch/RSX-getrieben ist (z.B. `clippy::needless_return` in RSX-Macros die false-positive triggern), stattdessen einmalig auf Modul-Root: `#![allow(clippy::…)] // reason: false positive in dioxus RSX macro expansion` — anstelle von 50+ pro-Callsite `#[allow]`.

**Verbotene Änderungen:**
- Kein Anfassen von `Cargo.toml` (keine Dep-Bumps, kein `dioxus`-Version-Change — `wasm-bindgen = "=0.2.121"` MUSS gepinnt bleiben, das ist eine harte Ceiling wegen `dx serve`/`dx build`).
- Kein Anfassen von `../rest-types` (das ist der Backend-Workspace).
- Kein Anfassen von REST-Handlern, DAOs, Migrations.
- Kein Ändern von i18n-Keys / Übersetzungen.
- Kein Refactoring bestehender Logik über den Lint hinaus — nur Lint-Fix.

**Wenn ein Fix Logik ändern würde** (kein rein syntaktischer Fix, z.B. `clippy::unwrap_used` → müsste Error-Path bauen): NICHT fixen, stattdessen `#[allow(clippy::unwrap_used)] // reason: pre-existing, refactor to Result would change semantics — deferred` und Todo dafür anlegen unter `.planning/todos/pending/2026-07-02-<kurz>.md`.

Iterieren bis Rest = 0 Warnings.
