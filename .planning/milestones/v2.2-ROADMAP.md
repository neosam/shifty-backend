### Phase 43: Special-Days-Feintuning (FE + BE)

**Goal**: Der Special-Days-Loader in den Einstellungen zeigt die Feiertage kalenderjahr-basiert (behebt 1.1.-Anzeige-Bug), der „already exists"-Hinweis passt zum Replace-Verhalten und der Feiertag↔Kurzer-Tag-Umschalter im Schichtplan wirft keine Fehlermeldung mehr.
**Depends on**: Nichts Neues.
**Requirements**: SDF-03, SDF-04, SDF-05
**Plans:** 2/2 plans complete

Plans:

- [x] 43-01-PLAN.md — SDF-03 sd_year Kalenderjahr-Fix + SDF-04 Duplikat-Hinweis-Copy replace-tauglich (FE, i18n)
- [x] 43-02-PLAN.md — SDF-05 Wochenraster-Dropdown Feiertag↔Kurzer-Tag Roundtrip-Verifikation (Backend-Integrationstest + FE-pure-fn)

**Success Criteria** (what must be TRUE):

  1. `settings.rs` lädt `special_days` nach `date.year()` — 1.1.-Test grün (Datumsstich am Jahreswechsel liefert die richtige Feiertagsliste).
  2. i18n-Copy des Duplikat-Hinweises in de/en/cs entspricht dem in-place-Replace-Verhalten (kein „blockiert"-Text mehr).
  3. Umstellung Feiertag↔Kurzer-Tag im Wochenraster-Dropdown greift atomar (analog SDF-01) und produziert weder 422-Fehler noch UI-Fehlermeldung.

### Phase 44: Frontend-Korrektheit (FE)

**Goal**: Drei pre-existing Frontend-Bugs abarbeiten: `save_slot_edit` hält keinen `SLOT_EDIT_STORE`-Write-Borrow mehr über `.await`, `list_user_invitations` meldet Parse-Fehler sichtbar, und noch nicht migrierte Modals nutzen die drag-safe `BackdropPress`-Logik.
**Depends on**: Nichts Neues.
**Requirements**: BUG-01, BUG-02, BUG-03
**Plans:** 3/3 plans complete

Plans:

- [x] 44-01-PLAN.md — BUG-01 save_slot_edit ohne write-Borrow über .await (pure-fn Snapshot + Apply)
- [x] 44-02-PLAN.md — BUG-02 list_user_invitations Parse-Fehler propagieren (ShiftyError::InvitationParse + Inline-Banner + i18n)
- [x] 44-03-PLAN.md — BUG-03 Backdrop-Press-Audit + durable Grep-Invariant-Test

**Success Criteria** (what must be TRUE):

  1. `save_slot_edit` liest alle nötigen Store-Felder VOR jedem `.await` in lokale Werte — Regressionstest, der eine borrow-across-await-Panic simuliert, ist grün.
  2. `list_user_invitations`-Loader unterscheidet „leere Liste" von „Parse-Fehler" — der Fehler-Zweig rendert eine sichtbare Fehler-Meldung, kein silent-empty.
  3. Alle Dialoge/Modals unterhalb `component/`, die den Backdrop-Close-Pfad nutzen, gehen über `BackdropPress` (verifiziert per Grep); ein Panel-Drag mit mouseup-außerhalb schließt kein Modal mehr.

### Phase 48: Nextcloud-PDF-Export via WebDAV (BE + Admin-UI)

**Goal**: Ein regelmäßiger Backend-Task rendert die Wochen-Schichtpläne als PDF und lädt sie per WebDAV nach Nextcloud hoch; die Aktivierung und Zielkonfiguration sind admin-gated und via Admin-UI/DB konfigurierbar.
**Depends on**: Neue Deps (PDF-Renderer + WebDAV-Client + Cron-Scheduler) — festgelegt in discuss-phase 48: `printpdf`, `reqwest_dav`, `tokio-cron-scheduler`.
**Requirements**: EXP-01, EXP-02, EXP-03
**Plans:** 5/5 plans complete

Plans:

- [x] 48-01-PLAN.md — EXP-02/EXP-03 Persistenz-Fundament: Migration `pdf_export_config` + DAO + Basic-Service `PdfExportConfigService` (admin-gated) + REST GET/PUT + rest-types DTO + DI-Wiring + ROADMAP-Anpassung
- [x] 48-02-PLAN.md — EXP-01 PDF-Renderer (pure): `service_impl/src/pdf_render.rs` mit `printpdf`, Landscape A4, deterministisches Layout „Schichtplan KW {NN} ({JJJJ})", TDD
- [x] 48-03-PLAN.md — EXP-01 WebDAV-Client (pure): `service_impl/src/webdav_client.rs` mit `reqwest_dav`, MKCOL+PUT, In-Run-Retry 2s/4s/8s, wiremock-Tests, TDD
- [x] 48-04-PLAN.md — EXP-01/EXP-03 Scheduler (Business-Logic): `PdfExportScheduler` mit `tokio-cron-scheduler`, Config-Reload, Retry-Persistenz, POST /trigger, Boot-Wiring, E2E-Test
- [x] 48-05-PLAN.md — EXP-02/EXP-03 Admin-UI-Card in `shifty-dioxus/src/page/settings.rs`, i18n de/en/cs, „Jetzt exportieren"-Button, Status-Anzeige

**Success Criteria** (what must be TRUE):

  1. Backend-Task rendert für den konfigurierten Wochen-Horizont (Default 8 Folgewochen) ein PDF pro Woche (deterministisches Layout, alle aktiven Sales-Persons enthalten, Dateiname `schichtplan-{JJJJ}-KW{NN}.pdf`, overwrite) und legt es unter dem konfigurierten WebDAV-Pfad ab; Integrationstest gegen einen In-Memory- oder Mock-WebDAV-Server grün.
  2. Konfiguration liegt in der Datenbank (neue Tabelle `pdf_export_config`, Single-Row): URL, User, App-Token (Klartext — bewusste Ops-Entscheidung, DB-File filesystem-geschützt), Zielordner, Wochen-Horizont (Default 8), Cron-Schedule (Default „0 6 * * 1"), Enabled-Toggle. Bearbeitung ausschließlich über die admin-gated Settings-Card. Keine Env-Variablen für dieses Feature.
  3. Admin-Panel (Settings-Card, admin-gated) zeigt Konfig-Formular + Enabled-Toggle + Status („letzter Erfolg" / „letzter Fehler"); ein transienter WebDAV-Fehler unterbricht den Server nicht, wird 3× mit Exponential-Backoff (2s/4s/8s) re-versucht und danach persistiert (`last_error_at` + `last_error_message`), der nächste Cron-Slot versucht erneut.
