# Milestone v1.8: Freiwilligen-Auswahl & Urlaubsanspruch-Korrektur (HR-UX)

**Status:** ✅ SHIPPED 2026-06-29 (beide Phasen VERIFIED inkl. Live-HR-Browser-Smokes)
**Phases:** 27–28
**Total Plans:** 5

## Overview

HR-UX rund um Abwesenheiten/Urlaub: Freiwillige (`sales_person.is_paid == false`) sind
in den Abwesenheits-Selektoren auswählbar (gruppiert Angestellte/Freiwillige in
AbsenceModal **und** AbsenceFilterBar), und HR kann den berechneten Jahres-
Urlaubsanspruch einer Person per signed **Korrektur-Offset** (Delta, kein absoluter
Override) anpassen — pro Person+Jahr persistiert, HR-gekennzeichnet+editierbar, für
normale User unsichtbar (API-level Hiding). Mitgeliefert: Off-by-one-Proration-Fix
(`vacation_days_for_year`) + Snapshot-Schema-Version-Bump 11 → 12. Audit `passed`
(2/2 Requirements, 100% Integration, 2/2 Flows).

## Phases

### Phase 27: Freiwillige in Abwesenheitsliste auswählbar (FE)

**Goal**: Auf der Abwesenheitsseite lassen sich auch Freiwillige auswählen — beim Anlegen einer Abwesenheit (AbsenceModal) und im HR-Personenfilter (AbsenceFilterBar). Angestellte und Freiwillige sind im selben Dropdown sichtbar getrennt (gruppierter Selector).
**Depends on**: Phase 26 (Backend-VFA für Freiwilligen-Abwesenheiten muss stabil sein)
**Requirements**: VOL-SEL-01
**Plans**: 1/1 plan complete

Plans:

- [x] 27-01: Pure `grouped_selectable` + `PersonGroup` (Employees | Volunteers) + RSX-Helfer `grouped_person_options` (zwei native `<optgroup>`s); beide Call-Sites (AbsenceModal + AbsenceFilterBar) auf den **gemeinsamen** Helfer umgestellt (kein Copy-Paste); 2 i18n-Keys (`AbsenceGroupEmployees`/`AbsenceGroupVolunteers`) de/en/cs; 5 neue Pure-Function-Tests (D-27-01..06). Reines Frontend.

**Success Criteria** (alle erfüllt):

1. Im AbsenceModal-Dropdown erscheinen aktive Freiwillige in eigener Gruppe „Freiwillige" 🎯 unterhalb „Angestellte" — Abwesenheit für sie anlegbar.
2. Im AbsenceFilterBar (HR) sind Freiwillige genauso gruppiert auswählbar; „Alle"-Option bleibt erhalten.
3. Inaktive Personen bleiben in beiden Selektoren ausgeblendet (egal Angestellte/Freiwillige).
4. Beide Gruppen-Beschriftungen in de/en/cs vorhanden (neue i18n-Keys).
5. Leere Gruppen werden nicht gerendert (kein leeres `optgroup`).

**Konzept (entschieden 2026-06-29):** gruppierter Dropdown via native `optgroup`
(`SelectInput` unverändert); gemeinsamer Helfer für Modal + Filter; `is_selectable_employee`
NICHT gelockert (D-27-02 — HR-Urlaubsübersicht bleibt paid-only), die Gruppierung nutzt
ein eigenes `!inactive`-Predicate; kein Backend-Change.

### Phase 28: Urlaubsanspruch-Korrektur via Offset (HR, BE+FE)

**Goal**: HR kann den berechneten Jahres-Urlaubsanspruch einer Person um einen signed **Offset (Korrektur-Delta)** anheben/senken, um Rundungs-/Proration-Differenzen auszugleichen. Der Offset wandert bei Vertragsänderungen mit (Delta, kein absoluter Override). In der HR-Ansicht gekennzeichnet+editierbar; für normale User unsichtbar (nur die finale Zahl).
**Depends on**: Phase 26 (vacation_balance / EmployeeWorkDetails stabil); unabhängig von Phase 27
**Requirements**: VAC-OFFSET-01
**Plans**: 4/4 plans complete

Plans:

- [x] 28-01: Backend data layer [Wave 1] — additive Migration + Tabelle `vacation_entitlement_offset` (id PK, sales_person_id, year, `offset_days` signed, soft-delete-Spalten, partial unique index `WHERE deleted IS NULL`) + DAO trait/sqlite-impl + Basic HR-gated `VacationEntitlementOffsetService` (gen_service_impl!, HR_PRIVILEGE-Gate auf jeder Methode) + CRUD/HR-gate-Tests (D-28-01, D-28-06, D-28-06b).
- [x] 28-02: VacationBalance integration [Wave 2] — Offset **nach** `.round()` addiert (Integer-Day-Korrektur), API-level Hiding (HR-only breakdown: `offset_days`/`computed_entitled_days` nur `Some` wenn `is_hr`), `VacationBalanceTO` + Domain-Felder, HR-gated REST CRUD-Endpoint (`POST` upsert + `DELETE` + ApiDoc), DI-Wiring (BL→Basic, kein Cycle), Offset/Delta/Hiding-Tests (D-28-02, D-28-03, D-28-06b, D-28-09).
- [x] 28-03: Off-by-one proration fix [Wave 1] — `vacation_days_for_year` year-START-Branch korrigiert (1.1.-Start subtrahiert jetzt 0 statt ~1/365) + Snapshot-Bump 11→12 (`BillingPeriodValueType::VacationEntitlement`) + Guard-/Regressions-Tests (D-28-04, D-28-05).
- [x] 28-04: Frontend inline signed offset editor [Wave 3] — HR-Detail-StatBox „berechnet {n} + Offset [x]" (immer sichtbar, signed, on-blur/Enter, year-scoped); User-Seite effective-only (computed/offset == None → kein breakdown, kein Input); `SaveOffset`-Action (POST→reload); i18n `VacationOffsetComputedLabel`/`VacationOffsetLabel` de/en/cs (D-28-07, D-28-03, D-28-08, D-28-09).

**Success Criteria** (alle erfüllt):

1. Mit gesetztem Offset gilt `entitled_effective = round(berechneter Anspruch) + offset` 🎯; wirkt automatisch auf `remaining_days` durch.
2. Offset ist signed (positiv/negativ) und pro Person+Jahr persistiert; überlebt Seitenreload.
3. Bei späterer Vertragsänderung bleibt der Offset bestehen und wird auf den neuen Anspruch angewandt (Delta-Verhalten).
4. HR-Detailansicht: „Vertragsanspruch"-StatBox zeigt Effektivwert + immer sichtbares signed Inline-Offset-Feld „berechnet {n} + Offset [x]"; HR-gated Speichern.
5. User-Eigenansicht: dieselbe StatBox zeigt **nur** den Effektivwert (kein berechnet/Offset, kein Feld).
6. Offset-Setzen/-Ändern ist HR-gated (`HR_PRIVILEGE`); neue Texte de/en/cs.

**Konzept (entschieden 2026-06-29):** Mechanismus = Offset/Delta (überlebt
Vertragsänderungen); neue Tabelle `vacation_entitlement_offset`; Offset nach `.round()`
addiert; Platzierung in der Personen-Detailansicht an der „Vertragsanspruch"-StatBox;
„für User unsichtbar" als **API-level** Hiding (sauberer als nur UI-Ausblenden);
Off-by-one-Begleit-Fix mitgenommen; Snapshot-Bump 11→12 (durch geänderte
VacationEntitlement-Computation).

## Milestone Summary

**Key Decisions:**

- **D-27-02: `is_selectable_employee` NICHT gelockert** — die neue Gruppierung nutzt ein eigenes `!inactive`-Predicate; die HR-`VacationPerPersonList` bleibt paid-only. (Bewusst gegen die ursprüngliche „Predicate lockern"-Idee aus der ROADMAP entschieden.)
- **D-27-01/03: ein gemeinsamer RSX-Helfer** (`grouped_person_options`) für Modal + Filter, Employees-vor-Volunteers, leere Gruppen ausgelassen — kein Copy-Paste, kein leeres `optgroup`.
- **D-28-01: Mechanismus = signed Offset/Delta** (kein absoluter Override) — überlebt Vertragsänderungen; eigene Tabelle mit partial unique index `(sales_person_id, year) WHERE deleted IS NULL` (RESEARCH-Form A1).
- **D-28-02: Offset nach `.round()`** addiert (Integer-Day-Korrektur, nicht in die f32-Summe) → fließt in `remaining_days` durch.
- **D-28-03: API-level Hiding** — `entitled_days` immer effektiv; `offset_days`/`computed_entitled_days` nur `Some` wenn HR, sonst `None`; Self-View re-derived nichts client-seitig.
- **D-28-06b: dedizierter HR-gated REST CRUD**; HR_PRIVILEGE im Basic-Service erzwungen. Service-Tiering: Business-Logic `VacationBalanceService` konsumiert den Basic-Offset-Service (kein Cycle), DI-Order Basic-vor-Business.
- **D-28-04/05: Off-by-one nur im year-START-Branch** gefixt (year-END war korrekt) + Pflicht-Snapshot-Bump 11 → 12 (`BillingPeriodValueType::VacationEntitlement`-Computation geändert).

**Issues Resolved:**

- Freiwillige konnten in den Abwesenheits-Selektoren nicht ausgewählt werden → jetzt gruppiert auswählbar (Modal + Filter).
- HR konnte Rundungs-/Proration-Differenzen im Urlaubsanspruch nicht korrigieren → signed Offset-Mechanismus.
- Off-by-one in der Jahres-Proration (1.1.-Start zog ~1/365 ab) → korrigiert.
