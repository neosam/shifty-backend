## Overview

Slots erhalten ein optionales Capacity-Limit `max_paid_employees: Option<u8>`. Wenn der Live-Count an aktiven Bookings im Slot mit `sales_person.is_paid = true` das konfigurierte Limit übersteigt, emittiert das Backend nicht-blockierende Warnings auf zwei Surface-Pfaden:

1. **Write-Path** — `BookingCreateResult.warnings` aus `ShiftplanEditService::book_slot_with_conflict_check` (Pattern aus v1.0 Phase 3 wiederverwendet)
2. **Read-Path** — `current_paid_count: u8` neben `max_paid_employees` im `ShiftplanSlot`-DTO der Week-View

Buchen bleibt erlaubt — die Warning ist informativ, nicht blockierend (D-07: kein Tx-Rollback). `NULL`-Konfiguration bedeutet „kein Limit" (kein Check, keine Warning, keine Read-Side-Guard). Der Check lebt im Business-Logic-Tier (`ShiftplanEditService`), nicht im Basic-Tier `BookingService` — explizite Beibehaltung des v1.0-Phase-3-Regression-Locks (D-Phase3-18). Frontend-Workstream (shifty-dioxus) ist out of scope.

Backend-additiv über alle Layer: SQLite-Migration (nullable, kein Backfill), DAO-Read/Write, Service-Tier (Slot + Warning enum), Business-Logic-Tier (Edit-Service-Emission), REST-Wire (SlotTO + WarningTO + ShiftplanSlotTO).

## Milestone Summary

**Key Decisions (16 D-NN, alle in 05-VERIFICATION.md verifiziert):**

| ID | Decision | Rationale |
|----|----------|-----------|
| D-01 | Nullable `slot.max_paid_employees INTEGER` ohne DEFAULT/NOT NULL | Kein Backfill nötig — bestehende Slots = "kein Limit" semantisch korrekt |
| D-02 | `Option<u8>` über alle Tiers (DAO, Service, REST) | Wert-Range realistisch ≤ 255; `Option` propagiert NULL-Semantik typsicher |
| D-03 | Limit pro `(year, week, slot)`-Triple | Slot ist nur Template; tatsächliche Bookings sind weekly |
| D-04 | Count = aktive (`deleted IS NULL`) Bookings ∧ `is_paid=true` (joined SP) | "Bezahlt" ist die Capacity-Dimension; gelöschte/unbezahlte zählen nicht |
| D-05 | Absence-Status der gebuchten Person ist IRRELEVANT | Capacity ist Booking-Slot-Property, nicht Person-Verfügbarkeits-Property; orthogonal zu v1.0 Phase 3 Absence-Warnings |
| D-06 | Strict `current > max` (NICHT `>=`) | "Equal" ist ausgebucht, nicht overbooked; Warning nur bei echtem Overflow |
| D-07 | Kein Tx-Rollback bei Warning — Booking persistiert | Warning ist informativ, nicht blockierend; mirrors v1.0 Phase 3 BookingOnAbsenceDay-Pattern |
| D-08 | Strukturierte Variant mit 6 Feldern (slot_id, booking_id, year, week, current_paid_count, max_paid_employees) — keine Strings | Frontend-i18n-tauglich; Filterbarkeit; konsistent mit existierenden 4 Warning-Varianten |
| D-09 | Read-Side `current_paid_count: u8` per Slot in Week-View | Frontend braucht Anzeige-Wert auch ohne Booking-Trigger |
| D-10 | REST `SlotTO.max_paid_employees: Option<u8>` mit `#[serde(default)]` | Backward-Compat: alte Clients ohne Feld → deserialisiert zu `None` |
| D-11 | Update-Permission via `SHIFTPLANNER_PRIVILEGE`; Feld in-place mutable | Konsistent mit anderen Slot-Modifikationen; nicht in `ModificationNotAllowed`-Liste |
| D-12 | Limit-Check lebt im `ShiftplanEditService` (Business-Logic-Tier), NICHT im `BookingService` (Basic-Tier) | CLAUDE.md Service-Tier-Konvention + v1.0 D-Phase3-18 Regression-Lock (BookingService 0-Diff) |
| D-13 | Warning-Pattern baut auf v1.0 Phase 3 `BookingCreateResult { booking, warnings }` auf — kein neuer Wrapper | Konsistente API-Surface; existierender Akkumulator ergänzt |
| D-14 | KEIN Bump von `CURRENT_SNAPSHOT_SCHEMA_VERSION` | Keine `BillingPeriodValueType`-Änderung, keine Reporting-Computation berührt |

**Patterns Established (carry forward):**

- **Wave-Coupling-Pattern** — wenn additiver Variant zu Domain-Enum (`service::warning::Warning`) ein exhaustive downstream `match` ohne Wildcard bricht (`From<&Warning> for WarningTO`), schedule Producer-Plan + Consumer-Plan in der GLEICHEN Wave; Standalone-Akzeptanz reduziert sich auf `cargo build -p {producer-crate}`.
- **Wire-Tier-Mirror-Pattern** — additive Service-Tier-Field/Variant landet wire-tier in `rest-types/src/lib.rs` durch 3 Mechanismen: (1) Struct-Feld auf `*TO` + beide `From`-Impls mit `#[serde(default)]`-Backward-Compat; (2) Enum-Variant am Ende mit `#[serde(rename_all = "snake_case")]`-Auto-Tag + matching `From`-Arm (rustc enforced Exhaustivität); (3) Cascade-DTOs erben automatisch via `Vec<*TO>`-Embedding.
- **Read-Aggregation-Pattern** — derive read-side Counts inline in View-Builder aus bereits resolvten Sub-Entities (`slot_bookings.iter().filter(…).count().min(u8::MAX as usize) as u8`), statt separater DAO-Query. Kosten: eine Filter-Iteration über schon geladene Daten.
- **Forward-Compat-Shim-Pattern (Rule 3)** — wenn DAO-Feld eine Phase vor seinem Service-Layer-Mirror landet, hardcode `None` in `From<&Service::Slot> for SlotEntity` und im zentralen Test-Fixture mit Inline-Kommentar auf Folge-Plan.
- **Sequential-Wave-2-Friction-Mitigation** — wenn parallel-geplante Wave-Plans sequenziell ausgeführt werden (z. B. weil ein Plan andere Crates importieren muss um zu kompilieren), Rule-3-Shims in OUT-OF-SCOPE-Sites mit Folge-Plan-Kommentar einsetzen statt Wave-Reorder.
- **Warning-Emission-Heart-Pattern** (Phase-5-Plan-06) — Soft-Warning-Emission im Business-Logic-Tier-Service: Insert die Limit-Check-Logik zwischen die existierende Cross-Source-Warning-Emission und das finale `transaction_dao.commit(tx)`. Persistierte Entity ist in-hand, warnings-Akkumulator ist in-hand. Kein Rollback (D-07). Helper als private Methode auf einem zweiten `impl<Deps>`-Block; Helper-Signatur: `tx: Deps::Transaction` by-value (kein `Option<>` — Helper wird nur aus Caller heraus aufgerufen, der bereits `use_transaction`'d hat).
