# Requirements: Shifty — v1.4 Committed Voluntary Capacity

**Defined:** 2026-06-22
**Core Value:** Im Voraus zugesagte freiwillige Stunden-Kapazität wird pro Mitarbeiter erfasst und in der Jahresansicht ohne Doppelzählung als separat ausgewiesene Kapazität ausgewertet.

> Research: `.planning/research/SUMMARY.md` (HIGH confidence, line-verifiziert). Design-Grundlage: Todo `2026-06-22-committed-voluntary-capacity-jahresansicht.md` (D-01 / Variante B). Schlüssel-Erkenntnis: Jahresansicht hängt an `booking_information.rs::get_weekly_summary` (Achse B), **nicht** an `reporting.rs`.

## v1.4 Requirements

### Datenmodell (CVC)

- [x] **CVC-01**: Mitarbeiter-Vertrag (`EmployeeWorkDetails`) trägt ein zeit-versioniertes Feld `committed_voluntary: f32` (nur die freiwillige Zusage obendrauf, entkoppelt von `expected_hours`) — durchgezogen über SQLite-Migration (`REAL NOT NULL DEFAULT 0`), DAO (Entity + 4 SELECT + INSERT + UPDATE + `TryFrom`), Service (Struct + beide Konversionen), `rest-types` (`EmployeeWorkDetailsTO` mit `#[serde(default)]`); `.sqlx`-Offline-Cache regeneriert; Bestandsdaten driftfrei (Default 0).
- [x] **CVC-02**: Beim Rotieren einer Vertrags-Version (neue Zeit-Periode) wird `committed_voluntary` korrekt mitgeführt und nicht still auf Default zurückgesetzt.
- [x] **CVC-03**: Aggregation bei zwei überlappenden aktiven Vertrags-Rows in derselben ISO-Woche ist explizit definiert (sum/max/first) und durch einen Test gepinnt (D-OVERLAP-AGG — Boolean-`.any()`-Pattern des Cap-Flags generalisiert nicht auf einen numerischen Wert).

### Reporting / Berechnung (CVC)

- [x] **CVC-04**: Die Jahresansicht-Verfügbarkeit rechnet die Zusage **ohne Doppelzählung** ein — als **separater** `committed_voluntary_hours`-Term in `booking_information.rs::get_weekly_summary` (Achse B, NICHT `reporting.rs`), per ISO-Woche via `counted_volunteer = max(committed_voluntary, actual_volunteer)`, summiert über das Jahr (nie `max(Σ,Σ)`); gegated auf `cap_planned_hours_to_expected = true`.
- [x] **CVC-05** (revidiert per D-01 — KEIN Bump): `CURRENT_SNAPSHOT_SCHEMA_VERSION` bleibt **7**. Die Zwei-Band-Dekomposition (D-05: `committed_voluntary_hours` + reduzierter `volunteer_hours`) ist Achse-B-only (Jahresansicht-Verfügbarkeit) und speist keinen persistierten `BillingPeriodValueType`; `WeeklySummary` wird von `billing_period_report.rs` nie konsumiert → die CLAUDE.md-Bump-Regel ist NICHT ausgelöst (no-bump justification, Audit-Trail). Ein Regressionstest pinnt die unveränderte Version 7.
- [x] **CVC-06**: Für nicht-gedeckelte Personen (`cap_planned_hours_to_expected = false`) trägt die Zusage `0.0` zu allen Berechnungen bei und wird in der Anzeige nicht als Kapazität ausgewiesen (blank/Strich, nicht `0`).

### Jahresansicht-Anzeige (CVC)

- [x] **CVC-07**: `weekly_overview` zeigt die committed-Kapazität **separat** (eigener Token, z. B. „zugesagt") — nicht mit `paid`/`volunteer` vermischt; Überschuss wird sichtbar ausgewiesen (Beispiel committed=5, actual=7 → `🎯5.00 | 🤝2.00`); committed=5, actual=3 → gedeckte `🎯5.00 | 🤝0.00`. **Revidiert per Phase-16-CONTEXT D-03:** bei committed=0 wird `🎯0.00` gezeigt (konsistent mit paid/volunteer), KEINE blank/Strich-Sonderlogik in der aggregierten Jahresansicht (blank/Strich gehört, falls überhaupt, in Phase 17 / Mitarbeiteransicht).
- [x] **CVC-08**: Alle neuen benutzersichtbaren Strings sind in De / En / Cs vollständig gepflegt (kein `Locale::En`-statt-`Locale::De`-Bug; Per-Locale-Reference-Matcher analog v1.3).

### Editor + unbezahlte Freiwillige (CVC)

- [x] **CVC-09**: `committed_voluntary` ist im Vertrags-Editor (`contract_modal.rs`, neben dem Cap-Toggle) als numerisches Feld editierbar; Open→Save-unverändert-Round-Trip bewahrt den Backend-Wert.
- [x] **CVC-10**: Mitarbeiteransicht bekommt einen einblendbaren „alle"-Filter; rein unbezahlte Freiwillige können einen `EmployeeWorkDetails`-Record halten (`is_paid=false`, `expected_hours=0`) und werden sichtbar/auswählbar. Jede work-details-iterierende paid-only-Site ist explizit auf `sales_person.is_paid` gegated (nicht auf Record-Präsenz) — kein Leak in `paid_hours`/Billing/Year-Summary; `get_week`-Seiteneffekt durch Integrationstest abgesichert.

## Out of Scope

Explizit ausgeschlossen, um Scope-Creep zu verhindern.

| Feature | Grund |
|---------|-------|
| Variante A: `committed_total` inkl. Vertrag | Bräuchte Invariante `committed >= expected` + impliziten Subtraktionsschritt beim Lesen; D-01 wählte Variante B. |
| Zusage an Absence/Urlaub/Feiertag koppeln | Flache Wochen-Kapazität, kein Cross-Entity-Coupling (D-ABSENCE-DISPLAY). |
| Pro-rating der Zusage nach Teilwoche | Würde eine „5h-Zusage" auf ersten/letzten Teilwochen still verkleinern (D-PARTIAL-WEEK → flat pro aktiver Woche). |
| Einrechnung in `reporting.rs` (Achse A) für die Jahresansicht | Dieser Pfad erreicht die Jahresansicht nie → Doppelzählung gegen Achse B (D-FORMULA-PATH). |
| OpenAPI/utoipa-Erweiterung für EmployeeWorkDetails | Diese Endpoint-Familie hat heute bewusst keine `#[utoipa::path]`/`ToSchema`; serde-transparent → kein Phantom-Task. |
| Genehmigungs-Workflow für Zusagen | Backend kennt keinen Approval-Schritt (PROJECT.md). |
| Min-Paid-Capacity / Skill-Matching (SC-01, SC-02) | Eigene künftige Backend-Themen. |
