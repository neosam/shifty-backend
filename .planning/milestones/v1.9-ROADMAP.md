### Phase 29: Urlaubs-Balken-Konsistenz (FE)

**Goal**: Der Pro-Person-Urlaubsbalken auf der Abwesenheiten-Seite stimmt mit der Resturlaub-Zahl daneben überein.
**Depends on**: Nothing (first v1.9 phase)
**Requirements**: VAC-01
**Frontend-Pfad**: `shifty-dioxus/src/page/absences.rs` (formula fix, line ~866)
**Backend-Pfad**: none
**API change note**: Kein neuer Endpoint, kein neues DTO-Feld. Alle Felder (`used_days`, `planned_days`, `remaining_days`) sind bereits in `VacationBalanceTO` vorhanden. Reiner FE-Compute-Fix.
**Success Criteria** (what must be TRUE):

  1. Der Urlaubsbalken zeigt `(used + planned) / total` als Füllstand — Balken und Resturlaub-Zahl daneben messen exakt dieselbe Größe und stimmen überein.
  2. Bei Überschreitung des Urlaubsanspruchs (`used + planned > total`) wird der Überzug sichtbar signalisiert: voller Balken in Warnfarbe (amber) + negative Resturlaub-Zahl (Farb-Signal, kein physischer Überlauf — D-29-02 aus 29-CONTEXT).
  3. Wenn `remaining_days < 0`, werden Balken und Resturlaub-Zahl in Warnfarbe (rot/amber) dargestellt.

**Plans**: 1 plan

- [x] 29-01-PLAN.md — Pure `compute_vacation_bar` helper ((used+planned)/total, clamp, low-flag) + unit tests, verdrahtet in `PersonVacationCard` (VAC-01, D-29-01..03)

**UI hint**: yes

### Phase 30: Stale-Daten-Race Guard (FE)

**Goal**: Die Summary-Karten unter dem Schichtplan zeigen beim schnellen Wochenwechsel immer nur die Daten der aktuell gewählten Woche — kein gemischter Zustand aus verspäteten Loads.
**Depends on**: Phase 29
**Requirements**: SHP-02
**Frontend-Pfad**: `shifty-dioxus/src/service/weekly_summary.rs`, `shifty-dioxus/src/service/booking_conflict.rs`, `shifty-dioxus/src/page/shiftplan.rs` (`reload_unavailable_days`-Closure + Render-Guard)
**Backend-Pfad**: none
**API change note**: Kein neuer Endpoint. Reines FE-Concurrency-Fix; bestehende API-Calls bleiben unverändert.
**Success Criteria** (what must be TRUE):

  1. Schnelles Durchklicken durch mehrere Wochen erzeugt niemals einen Zustand, in dem die Summary-Karten Daten einer anderen Woche zeigen als die aktuell angezeigte.
  2. Alle drei betroffenen Loader (`WEEKLY_SUMMARY_STORE`, `BOOKING_CONFLICTS_STORE`, `reload_unavailable_days`) sind mit demselben `(year, week)`-Guard atomar abgesichert — kein Partial-Fix-Antipattern, bei dem ein einzelner Loader noch racy ist.
  3. Ein Loader-Ergebnis, das nach einem Wochenwechsel eintrifft, wird still verworfen; die angezeigten Daten bleiben konsistent mit der aktuell gewählten Woche.

**Plans**: 1 plan

- [x] 30-01-PLAN.md — Shared SELECTED_WEEK guard + pure is_current_selection predicate (tested); wire all 3 loaders + summary-card render-guard

**UI hint**: yes

### Phase 31: Abwesenheit → Nicht-Verfügbar-Markierung im Schichtplan (FE)

**Goal**: Tage mit eigenen Abwesenheits-Zeiträumen erscheinen im Schichtplan-Grid proaktiv als „Nicht Verfügbar" (discourage), bevor eine Buchung versucht wird — nicht erst als nachträgliche Warnung beim Buchen.
**Depends on**: Phase 30 (Race-Guard muss aktiv sein, bevor ein weiterer Async-Load-Pfad hinzukommt)
**Requirements**: SHP-01
**Frontend-Pfad**: `shifty-dioxus/src/page/shiftplan.rs` (neuer `reload_absence_days`-Closure, `person_absences`-Signal, pure Helfer-Funktion `absence_periods_to_discourage_days`, Merge in `discourage_weekdays` bei ~Zeile 1120)
**Backend-Pfad**: none — bestehender `/absence-period`-Endpoint; kein neues API
**API change note**: Kein neuer Endpoint. Nutzt bestehende Absence-Period-API; reiner FE-Join.
**Research flag**: NEEDS DISCUSS — Decision `D-NN` (Kategorie-Whitelist: welche `AbsenceCategory`-Varianten lösen die Markierung aus) muss im CONTEXT vor der Planung festgehalten sein. Empfehlung: alle drei (Vacation + SickLeave + UnpaidLeave), analog `BookingOnAbsenceDay`-Verhalten.
**Success Criteria** (what must be TRUE):

  1. Tage, an denen der aktuell angemeldete User einen Abwesenheits-Zeitraum hat, erscheinen im Schichtplan-Grid visuell als „Nicht Verfügbar" (discourage), bevor ein Buchungsversuch unternommen wird.
  2. Welche Absence-Kategorien die Markierung auslösen, ist durch eine dokumentierte Entscheidung (`D-NN`) festgelegt — kein stiller Kategorie-Drift zwischen Discourage-Markierung und Post-Booking-Warnung.
  3. Der neue `reload_absence_days`-Async-Pfad erzeugt keine Stale-Data-Race (ist durch den `(year, week)`-Guard aus Phase 30 geschützt); beim schnellen Wochenwechsel bleiben alle vier Loader konsistent.

**Plans**: 1 plan

- [x] 31-01-PLAN.md — Pure helper `absence_periods_to_discourage_days` (tested) + guarded `reload_absence_days`/`person_absences` + union-merge into `discourage_weekdays`

**UI hint**: yes

### Phase 32: Admin-Impersonation Frontend + Audit-Schicht (FE+BE)

**Goal**: Admins können temporär als anderer User agieren (lesen und schreiben), mit persistentem Banner auf jeder Seite, strukturiertem Audit der echten Admin-Identität in den Logs und sauberem Store-Teardown beim Beenden der Impersonation.
**Depends on**: Phase 31
**Requirements**: IMP-01, IMP-02, IMP-03, IMP-04
**Frontend-Pfad**: `shifty-dioxus/src/service/impersonate.rs` (neu), `shifty-dioxus/src/component/impersonation_banner.rs` (neu), `shifty-dioxus/src/app.rs` (Banner-Mount + Init-Call), `shifty-dioxus/src/api.rs` (3 neue API-Calls für Start/Stop/Status), i18n de/en/cs
**Backend-Pfad**: `rest/src/session.rs` (`context_extractor` — `Extension<RealUser>` einfügen wenn `impersonate_user_id.is_some()`), `rest/src/impersonate.rs` (`tracing::info!` Audit-Events bei Start/Stop)
**API change note**: Kein neuer Endpoint (alle drei `/admin/impersonate`-Endpoints bereits vorhanden). Backend-Änderung ist auf `context_extractor` + `tracing`-Calls beschränkt; keine Signatur-Änderung an `Authentication<Context>`.
**Research flag**: NEEDS DISCUSS — zwei `D-NN`-Decisions müssen im CONTEXT vor jedem Write-Path-Code festgehalten sein: (1) Audit-Design (`RealUser`-Extension-Ansatz vs. Context-Struct-Umbau — Empfehlung: Extension, kein Blast-Radius); (2) Admin-Gate-Two-Path-Vertrag (neue Handler lesen echte Session-`user_id` direkt, nicht den effektiven Context — muss in `rest/src/lib.rs` dokumentiert sein).
**Known limitation**: Admin ist während der Impersonation eines Nicht-Admins von admin-only Endpoints ausgesperrt (P10 — korrekte Sicherheitseigenschaft, im Banner-Text und DISCUSS als bekannte Einschränkung dokumentieren).
**Success Criteria** (what must be TRUE):

  1. Ein Admin kann die Impersonation eines anderen Users aus der HR-/Admin-Personenliste starten und sieht sofort auf jeder Seite einen persistenten, nicht-schließbaren Amber-Banner mit dem Namen des impersonierten Users und einem One-Click-Stop-Button.
  2. Nach einem harten Page-Reload während einer aktiven Impersonation-Session erscheint der Banner automatisch wieder — der Impersonation-Status wird via `GET /admin/impersonate` als erster Init-Call beim App-Mount geladen (nicht nur aus dem Start-Callback).
  3. Alle schreibenden Aktionen unter Impersonation erscheinen in den strukturierten Logs (`tracing::info!`) mit sowohl der echten Admin-Identität als auch dem impersonierten User — keine Schreib-Aktion bleibt unattribuiert.
  4. Nach dem Beenden der Impersonation werden user-bezogene Frontend-Stores (z. B. `current_sales_person`) für den echten Admin neu geladen; kein Stale-State des impersonierten Users bleibt in der UI.
  5. Ein Nicht-Admin-User kann keine Impersonation starten oder ausnutzen — der Admin-Gate prüft die echte Caller-Identität (rohe Session-`user_id`), nicht den effektiven (impersonierten) Context.

**Plans**: 3 plans

- [x] 32-01-PLAN.md — Backend audit schicht: RealUser extension + central audit middleware + start/stop tracing + two-path/P10 doc + integration tests (IMP-03, IMP-01; SC3, SC5) [Wave 1]
- [x] 32-02-PLAN.md — FE runtime: 3 api.rs calls + service/impersonate.rs (status load, start/stop, full-reload teardown) (IMP-02, IMP-04, IMP-01; SC4) [Wave 1]
- [x] 32-03-PLAN.md — FE UI: i18n de/en/cs + non-closable amber banner mounted on every page + first-init GET-status + Users-tab "Act as this person" entry (IMP-01, IMP-02; SC1, SC2) [Wave 2, depends on 32-02]

**UI hint**: yes
