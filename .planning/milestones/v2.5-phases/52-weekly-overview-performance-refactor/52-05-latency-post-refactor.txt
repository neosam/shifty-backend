# 1. **Load-once für `sales_person_service.get_all` im `assemble_weeks`-
#    Helper** (RESEARCH Q2 Perf-Anmerkung + A2). Aktuell wird pro
#    Person pro Woche `sales_person_service.get(sp_id, ...)` aufgerufen —
#    das sind ~N_persons × 55 Roundtrips. `get_all` existiert bereits
#    und wird in `booking_information.get_weekly_summary` schon
#    verwendet. Der `assemble_weeks`-Helper könnte einen HashMap-Index
#    `HashMap<Uuid, SalesPerson>` als zusätzlichen Slice-Input bekommen.
#    Byte-identisch. Geschätzter Impact: 30-40% zusätzlicher Speedup.

# 2. **Batching für `absence_service.derive_hours_for_range` und
#    `reporting.build_derived_holiday_map`** (RESEARCH R3). Beide werden
#    aktuell pro Person pro Woche gerufen. Ein Jahres-Bulk-Load-Muster
#    (analog `find_by_year`) würde die Anzahl der DAO-Roundtrips
#    weiter reduzieren. Höherer Refactor-Aufwand — separater Task.

#    `extra_hours(date_time)`, `working_hours(from_year, to_year)`

#    `find_working_hours_for_calendar_week` läuft im

#    neuer DAO-Call, nur In-Memory-Umbau. Byte-identisch.
