Implement NAV-01: bidirectional per-employee deep-links between the absence view and the
employee report, plus a bookmarkable `/absences/:employee_id` route that pre-fills the
existing person selector (D-26-05). Cover both report entry points (D-26-06): Sales
("My time account" ↔ own absences) and HR (EmployeeDetails(:id) ↔ Absences(:id)). All
labels localized de/en/cs.

</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser user → /absences/:employee_id route param | untrusted String parsed to Uuid in AbsencesFor |
| AbsencesPage UI selector → absence data fetch | fetch is permission-gated server-side (AbsenceService HR ∨ self) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-26-03 | Information Disclosure (IDOR) | /absences/:employee_id deep-link as a non-HR user | medium | mitigate | The param only seeds a UI selector; the non-HR self-view loads ONLY the user's own absences (LoadForSalesPerson). Backend AbsenceService enforces HR ∨ self on every fetch — the deep-link cannot read another person's data. |
| T-26-04 | Tampering | malformed UUID in route param | low | mitigate | AbsencesFor parses with Uuid::parse_str; on failure renders an inline error and does NOT seed the selector or fetch. |
| T-26-SC | Tampering | npm/registry installs | n/a | accept | No new npm deps, no new registry blocks (per 26-UI-SPEC Registry Safety). |
</threat_model>
