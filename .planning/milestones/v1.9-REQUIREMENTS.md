# Requirements: Shifty — v1.9 Schichtplan-/Urlaubs-UX-Korrekturen & Admin-Impersonation

> **Versions-Hinweis:** `v1.9` ist das interne GSD-Planungs-Label. Die reale
> Release-Version vergibt der User datumsbasiert via `./cli-update-version`.

**Core Value:** Drei betrieblich aufgefallene UX-/Korrektheits-Lücken im Schichtplan-/
Urlaubs-Bereich schließen und Admins eine vollwertige (lesend **und** schreibend)
Impersonation anderer Benutzer geben — mit korrektem Audit der echten Admin-Identität
und ohne Privilege-Eskalation.

**Research:** `.planning/research/SUMMARY.md` (HIGH confidence, direkt code-verifiziert).
Headline: alle vier Themen frontend-zentriert, **keine neuen Dependencies**, das
**Impersonation-Backend (Mechanismus + admin-Gating) existiert bereits** — verbleibend
sind die Audit-Schicht (`Extension<RealUser>`) und das gesamte Impersonation-Frontend.

**i18n (querschnittlich):** Alle neuen benutzersichtbaren Texte in de/en/cs — gilt für
IMP-01..04 und SHP-01, wo neue Strings entstehen.

## v1.9 Requirements

### Schichtplan-UX (SHP)

- [x] **SHP-01**: Eigene Abwesenheits-Zeiträume markieren die betroffenen Tage im
  Schichtplan-Grid **proaktiv** als „Nicht Verfügbar" (discourage), nicht nur als
  Buchungs-Warnung beim Buchen. Reiner Frontend-Join: zusätzlicher
  `reload_absence_days`-Pfad (datumsbasiert, analog `reload_unavailable_days`) +
  `person_absences`-Signal + reine Helfer-Funktion `absence_periods_to_discourage_days`,
  gemergt in `discourage_weekdays` (`shifty-dioxus/src/page/shiftplan.rs:1120`).
  *Offene Decision (discuss-phase, D-NN):* Welche `AbsenceCategory`-Varianten lösen die
  Markierung aus (Empfehlung: alle drei — Vacation + SickLeave + UnpaidLeave, analog
  bestehendem `BookingOnAbsenceDay`); betrifft nur eigene Person (Editier-Sicht) oder
  alle Personen mit Urlaub in der Woche.

- [x] **SHP-02**: Die Wochen-Summary-Karten unter dem Schichtplan zeigen beim schnellen
  Wochenwechsel **immer nur die Daten der aktuell gewählten Woche** — kein kurzzeitiges
  Stale-State aus verspäteten Loads. Fix über `(year, week)`-Guard/Generation-Token,
  **atomar über alle drei betroffenen Loader** (`WEEKLY_SUMMARY_STORE`,
  `BOOKING_CONFLICTS_STORE`, `reload_unavailable_days`); Store-Write nach `await` nur,
  wenn das Ergebnis noch zur aktuell gewählten Woche passt + Render-Guard. Reines Frontend.

### Urlaubs-Anzeige (VAC)

- [x] **VAC-01**: Der Pro-Person-Urlaubsbalken auf der Abwesenheiten-Seite stimmt mit der
  Resturlaub-Zahl daneben überein: Füllstand `(used + planned) / total` statt nur
  `used / total`, **kein 100%-Clamp** (Überzug sichtbar), Warnfarbe bei `remaining < 0`.
  Reiner Frontend-Fix (`shifty-dioxus/src/page/absences.rs:866`); alle Felder
  (`used_days`/`planned_days`/`remaining_days`) liegen bereits in `VacationBalance`.

### Admin-Impersonation (IMP)

- [x] **IMP-01**: Ein Admin kann die Impersonation eines anderen Users **starten und
  beenden** (read **und** write), strikt `admin`-gated (Gate gegen die ECHTE
  Caller-Identität, kein Privilege-Leak). Einstieg aus der HR-/Admin-Personenliste.
  Backend-Mechanismus (`/admin/impersonate` POST/DELETE/GET, `Session.impersonate_user_id`,
  `context_extractor`) existiert bereits — Requirement umfasst die FE-Anbindung daran.

- [x] **IMP-02**: Während aktiver Impersonation zeigt das Frontend auf **jeder Seite**
  einen persistenten, **nicht-schließbaren** Banner („Du agierst als <Name> —
  Impersonation beenden") mit One-Click-Beenden. Der Banner ist auch **nach einem
  Page-Reload** sichtbar — der Impersonation-Status wird beim App-Mount via
  `GET /admin/impersonate` als erster Init-Call geladen (nicht nur aus dem Start-Callback).

- [x] **IMP-03**: **Schreibende Aktionen unter Impersonation werden mit der echten
  Admin-Identität auditiert.** Da `Context = Option<Arc<str>>` nur einen Identitäts-Slot
  hat, trägt ein zweites Axum-`Extension<RealUser>` (gesetzt im `context_extractor`, wenn
  `impersonate_user_id.is_some()`) die echte Identität; Schreib-Stellen loggen
  `tracing::info!(real_user, impersonating, …)`. **Keine Änderung an
  `Authentication<Context>`-Signaturen** (zu großer Blast-Radius). Keine
  Privilege-Eskalation: der impersonierte User bekommt nie mehr Rechte als er selbst hat.
  *Offene Decision (discuss-phase, D-NN):* Audit log-only (`tracing`) vs. zusätzlich
  DB-persistiert — Empfehlung log-only für v1.9.

- [x] **IMP-04**: Beim **Beenden** der Impersonation werden die user-bezogenen
  Frontend-Stores (z. B. `current_sales_person`) für den echten User neu geladen, damit
  keine Aktion nach dem Stop fälschlich dem impersonierten User zugeschrieben wird
  (kein stale Store-State).

## Out of Scope (bestätigt)

| Feature | Grund |
|---------|-------|
| Impersonation eines **anderen Admins** | v1.9-Limitierung — als bekannte Einschränkung dokumentieren (Pitfall P10), nicht im Scope. |
| Impersonation-**Session-Auto-Timeout** (`impersonate_expires_at`) | Manuelles Beenden genügt für v1.9; Security-Differentiator für v2+. |
| DB-persistierte **Audit-Log-UI** (für andere Admins sichtbar) | v1.9 nutzt strukturierte Logs (`tracing`); persistente UI später. |
| Absence-Approval-Workflow (pending vs. approved) | Off-theme, nicht Teil der UX-Fixes. |
| Änderung an `Authentication<Context>`-Signaturen | Bewusst vermieden — Audit über `Extension<RealUser>` (Blast-Radius). |
