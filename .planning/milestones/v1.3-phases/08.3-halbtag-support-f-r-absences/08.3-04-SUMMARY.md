## Task 1 — Reporting-Halbierung + Snapshot-Bump + 3 Reporting-Tests

### Schritt A — `derive_hours_for_range` Halbtag-Match-Arm

`service_impl/src/absence.rs:506-520`. Direkt vor dem `result.insert(...)` wird `effective_hours` berechnet:

```rust
let effective_hours = match dominant.day_fraction {
    dao::absence::DayFractionEntity::Half => hours * 0.5,
    dao::absence::DayFractionEntity::Full => hours,
};
result.insert(
    day,
    ResolvedAbsence {
        category: (&dominant.category).into(),
        hours: effective_hours,
    },
);
```

Begleitender Code-Kommentar referenziert D-08.3-04 und verweist auf die unveränderte Aufruferseite plus den Snapshot-Version-Bump.

### Schritt B — `CURRENT_SNAPSHOT_SCHEMA_VERSION` 3 → 4

`service_impl/src/billing_period_report.rs:37-57`. Doc-Kommentar wurde um eine **History**-Sektion erweitert (v1/v2 baseline, v3 Phase-2-AbsencePeriod-derived, v4 Phase-8.3 day_fraction-Half). Wortlaut des neuen v4-Eintrags:

> "Phase 8.3 — `AbsencePeriod.day_fraction == Half` halbiert die effektive Soll-Stundenzahl pro Tag in `derive_hours_for_range`; betrifft die Vacation/SickLeave/UnpaidLeave-derived value_types (hours + days). Alte Snapshots (v3) haben kein `day_fraction`-Feld in den Quell-Rows (Default `'full'` nach Migration), aber die Computation würde bei Re-Validation andere Werte liefern, sobald ein Half-Eintrag existiert. ## Decisions Made

1. **Atomare Commits pro Schritt:** Task 1 wurde in drei jj-Commits zerlegt (Halbtag-Multiplikator → Snapshot-Bump → Reporting-Tests), Task 2 in zwei (Filter → Regression-Test). Pro-Schritt-Commits machen den Schema-Bump-Kontext im Log nachvollziehbar — der Bump steht unmittelbar nach der Computation-Änderung, wie es CLAUDE.md § Snapshot Schema Versioning fordert.
2. **History-Sektion im Doc-Kommentar:** Der Plan-Text empfahl optional eine History-Sektion ("Falls die existing Doc-Kommentar-Struktur anders ist ... füge nur einen Phase-8.3-Bump-Hinweis hinzu"). Ich habe die History-Sektion strukturell ergänzt (v1/v2 baseline + v3 Phase-2 + v4 Phase-8.3), weil sie spätere Bumps systematisch dokumentiert.
