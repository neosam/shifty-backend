result: issue
reported: |
  Live test (2026-05-24, headless Chrome via CDP):
  Setup: INSERT extra_hours row for Anna Müller, 4.0h Vacation, 2026-12-24
  (Thursday, Heiligabend pattern). Anna's contract = 8h/day Mon-Fri.

  UI behavior — all correct:
    (a) Drift entry rendered with reason "amount_below_contract_hours"
    (b) Suggestion hint rendered VERBATIM: "Hours suggest half-day (4.00h ≈
        ½ × contract day)." — matches expected spec language ✓
    (c) Per-row select with options "Full day"/"Half day" works at UI level,
        selectable and dispatches change events ✓

  Submit behavior — backend rejects per-row Convert with Halbtag:
    (d) Click "Convert entry" with day_fraction=Half on a 4h Thursday row:
        Backend returns ValidationError: `[InvalidValue("Row does not match
        weekly-lump-sum heuristic; manual edit required")]`.

  Root cause — heuristic unchanged in Phase 8.3:
    service_impl/src/cutover.rs:511-525 — the per-row convert path still
    runs `detect_weekly_lump_sum` (Phase 8.1 heuristic) BEFORE applying the
    operator-supplied `day_fraction`. The heuristic only accepts true
    weekly-lump-sum patterns (e.g., 20h on Friday for a 3-day contract). A
    single 4h row on Thursday does not match the lump-sum pattern, so the
    backend rejects with "manual edit required" regardless of the
    day_fraction value. Code comment at cutover.rs:544 confirms intent:
    "Backend bleibt passiv — kein Auto-Vorschlag aus der
    detect_weekly_lump_sum-Heuristik."

  Workaround confirmed: Edit → inline manual-range form → Day fraction =
  Half → Create. This path skips the heuristic, uses the manual_range
  branch (Phase 8.2), and writes day_fraction='half' correctly.

  Verification of downstream behavior — all correct after Manual-Convert:
    (e) absence_period row created: from_date=2026-12-24, to_date=
        2026-12-24, category=Vacation, day_fraction=half ✓
    (f) Original extra_hours row soft-deleted (deleted=NOT NULL) ✓
    (g) Reporting `/report/{anna}/?year=2026&until_week=52` returns
        `vacation_hours: 12.0` — exactly (8h Mo full + 4h Thu half) ✓
        proving derive_hours_for_range Halbtag-multiplication (Plan 08.3-04)
        is wired through.

  Interpretation: Test 3 spec language ("Per-Row-Select auf Halbtag stellen,
  Convert drücken") doesn't match implementation. Phase 8.3 added the
  Halbtag plumbing but did not extend Phase 8.1's heuristic to recognize
  single-day half-day patterns; user must use the Manual-Convert path. The
  UI suggestion-hint correctly indicates a half-day amount but the per-row
  Convert button is not wired to accept it.

  Severity: minor — the half-day feature works end-to-end (just via a
  different button than the test spec describes). Possible follow-ups:
   (1) extend heuristic to accept 4h ≈ ½ × contract as a valid single-day
       half-day pattern; or
   (2) UX: disable per-row Convert when heuristic doesn't match + route
       user to Edit/Manual-Convert via the suggestion hint; or
   (3) update Test 3 spec to use a heuristic-matching row (e.g., 12h
       Friday for a 3-day contract) and verify Halbtag halves it.
severity: minor
