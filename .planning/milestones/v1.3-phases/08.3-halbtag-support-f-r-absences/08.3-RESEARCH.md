# Phase 8.3 — Research

**Researched:** 2026-05-16
**Source phase:** 08.3-halbtag-support-f-r-absences

---

## Summary

Phase 8.3 ergaenzt das `AbsencePeriod`-Datenmodell um ein `day_fraction: DayFraction { Full | Half }` -Feld, das von DB-Migration ueber alle Schichten (DAO, Entity, Domain, DTO, REST, Cutover, Frontend) durchgezogen werden muss. Der groesste inhaltliche Eingriff liegt in `service_impl/src/absence.rs:486ff.` (`derive_hours_for_range`): hier muss `dominant.day_fraction` bei `Half` den angelieferten Stundenwert halbieren. Da die Vacation-Aggregation in `reporting.rs` direkt auf den Stunden-Summen aus `derive_hours_for_range` basiert (Zeile 528ff., `absence_derived_vacation_hours`), propagiert die Halbtag-Logik automatisch in `VacationHours`, `VacationDays` und alle Snapshot-`value_type`-Felder -- daher ist der Snapshot-Schema-Version-Bump 3 -> 4 Pflicht. Das Frontend benoetigt Aenderungen an zwei Seiten (AbsenceModal + CutoverAdminPage/ManualConvertModal) und drei i18n-Dateien; die groesste Unbekannte ist die Cutover-Auto-Heuristik (s. Abschnitt 11).

**Primaerrisiken:** (1) `vacation_days` = `vacation_hours / hours_per_day` -- bei Halbtag liefert `derive_hours_for_range` halbierte Stunden, die Tages-Berechnung funktioniert korrekt ohne weiteren Eingriff. (2) Bestandssnapshots mit Version 3 duerfen nach dem Bump nicht mit Version-4-Logik neu bewertet werden -- das sichert `billing_period_snapshot_locking.rs`. (3) Alle sechs SELECT-Queries in `dao_impl_sqlite/src/absence.rs` muessen um die neue Spalte erweitert werden.


## 3. Reporting & Snapshot-Versioning

### 3.1 `vacation_hours`-Aggregation in `billing_period_report.rs`

Datei: `service_impl/src/billing_period_report.rs`

`CURRENT_SNAPSHOT_SCHEMA_VERSION` an Zeile 37: aktuell `3`, muss auf `4`.

Die Aggregation (Zeile 147-153) liest `report_delta.vacation_hours` aus `ReportingService::get_report_for_employee_range`. Diese Methode ruft intern `absence_service.derive_hours_for_range` auf (Zeile 520). Da `derive_hours_for_range` bei `Half` halbierte Stunden zurueckliefert, propagiert die Halbtag-Logik automatisch in die Billing-Period-Values -- kein zusaetzlicher Eingriff in `billing_period_report.rs` noetig ausser dem Versionsbump.

### 3.2 `vacation_days()` in `service/src/reporting.rs`

`GroupedReportHours::vacation_days()` (Zeile 118-122):
```rust
pub fn vacation_days(&self) -> f32 {
    if self.hours_per_day() == 0.0 { return 0.0; }
    self.vacation_hours / self.hours_per_day()
}
```
Bei 8h/Tag-Vertrag und 4h Vacation (Half-Eintrag): `4.0 / 8.0 = 0.5 Tage`. Automatisch korrekt.

`by_week`-Aggregation (Zeile 572-580 in `service_impl/src/reporting.rs`): Summiert `week.vacation_days()` -- korrekt fuer Halbtag.

### 3.3 `CURRENT_SNAPSHOT_SCHEMA_VERSION`-Locking-Test

Datei: `service_impl/src/test/billing_period_snapshot_locking.rs`

- `test_snapshot_schema_version_pinned` (Zeile 25-33): Behauptet `== 3`. Muss nach Bump auf `== 4` geaendert werden. Kommentar und Fehlermeldungstext muessen ebenfalls aktualisiert werden.
- `test_billing_period_value_type_surface_locked` (Zeile 41-64): Compile-only exhaustive match. Kein neuer `BillingPeriodValueType`-Variant fuer Halbtag noetig -- bestehende `VacationHours` und `VacationDays` bleiben unveraendert.

**Konsequenz des Bumps fuer Bestandssnapshots:** Bestehende Snapshots tragen `snapshot_schema_version = 3`. Der Validator (`compute_gate` in `cutover.rs`) vergleicht Snapshot-Version mit `CURRENT_SNAPSHOT_SCHEMA_VERSION`. Da sich die Vacation-Computation aendert, wuerden Snapshots mit Version 3 bei erneuter Berechnung andere Werte liefern koennen -- genau das ist der Grund fuer den Bump. Der Validator muss Snapshots mit aelterer Version als "older schema" markieren (kein falscher Drift-Alert).

### OQ-1: Datenmodell-Form

**RESOLVED:** Rust-Enum `DayFraction { Full, Half }` (type-safe, idiomatisch) + DB-Spalte `TEXT NOT NULL DEFAULT 'full'` mit `CHECK (day_fraction IN ('full', 'half'))` — siehe Plan 08.3-01 Decision **D-08.3-01**.

**Verifikation bestehender Patterns:**
- `AbsenceCategoryEntity` (DAO): `enum AbsenceCategoryEntity { Vacation, SickLeave, UnpaidLeave }` -- keine serde-Tags, kein DB-Mapping-Enum-Ableitung.
- DB-Mapping: `TEXT`-Spalte mit `match str { "Vacation" => ..., "SickLeave" => ... }` in `TryFrom`. Check-Constraint im Schema ist optional (SQLite prueft ihn zur Insert-Zeit).
- Empfehlung aus CONTEXT.md D-01: `TEXT NOT NULL DEFAULT 'full'` mit `CHECK (day_fraction IN ('full', 'half'))`. [ASSUMED: Check-Constraints in SQLite werden korrekt durchgesetzt -- nicht via Tool verifiziert.]

**INTEGER vs. TEXT:** `INTEGER` (0=Full, 1=Half) moeglich, aber `TEXT` ('full'/'half') konsistenter mit `category`-Pattern und self-dokumentierend in DB-Dumps.

### OQ-2: Cutover-Auto-Vorschlag-Heuristik

**RESOLVED:** Backend bleibt passiv — `detect_weekly_lump_sum` unangetastet, Default `Full`; Frontend rendert einen visuellen Hint (`CutoverDriftHalfDaySuggestion`) wenn `(amount - contract_hours_per_day * 0.5).abs() < 0.5`, Operator entscheidet manuell. Siehe Plan 08.3-05 Decision **D-08.3-06** (Backend-Verhalten) und Plan 08.3-06 Decision **D-08.3-FE-02** (Frontend-Hint-Render-Pfad inkl. SSR-Snapshot-Tests).

`detect_weekly_lump_sum` (Zeile 1518-1579): vergleicht `row.amount` mit Wochen-Vertragsstunden-Summe. Ankerpunkt fuer Halbtag-Heuristik: zusaetzliche Pruefung `(row.amount - target_sum * 0.5).abs() < CONTRACT_HOURS_EPSILON`. Rueckgabetyp muesste von `Option<(Date, Date)>` auf `Option<(Date, Date, DayFraction)>` erweitert werden. Alle drei Call-Sites (Zeile 513, 676, 864) betreffend.

**Kein bestehender Multi-Day-Halbtag-Ansatz gefunden** -- verifiziert durch Grep auf `day_fraction`, `half_day`, `half` in Service-Code (negatives Ergebnis). Das Modell kennt keine Halbtage.

**Alternative (gewählt):** Frontend zeigt immer "Ganztag" als Default + visueller Hint bei Halbtag-Verdacht. Keine Backend-Heuristik-Erweiterung. Einfacher Ansatz, geringere Risikooberflaeche.
