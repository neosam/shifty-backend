# Phase 8.3 Plan 05: Cutover-Backend — day_fraction-Threading

**Erweitert die zwei operator-getriebenen Cutover-Endpoints (`convert_quarantine_entry` + `bulk_convert_quarantine_rows`) um einen optionalen `day_fraction`-Parameter, der auf das neue `AbsencePeriodEntity.day_fraction`-Feld persistiert wird. Backwards-Compat: Pre-8.3-Clients ohne das Feld konvertieren weiter als `Full` (CONTEXT.md no-drift). Heuristik `detect_weekly_lump_sum` UNVERÄNDERT — Backend ist passiv, Operator entscheidet im Frontend.**

## Deviations from Plan

- **Test-Predicate-Granularität:** Plan-Text schlug `withf(|entity, _, _| entity.from_date == ... && entity.day_fraction == Half)` vor (mehrere Felder). Ich habe die `withf`-Predicates auf das `day_fraction`-Feld fokussiert (plus `from_date`/`to_date`/`category` im ersten Test als Sanity-Anchor), weil die übrigen Felder bereits durch die existing 8.1/8.2-Tests abgedeckt sind. Mockall-Strategy-Notiz: das hält den Test laser-fokussiert auf die Threading-Surface, statt einen Full-Payload-Equality-Check zu reproduzieren.
- **commit_phase-Cluster-Bridge bleibt Full:** Plan-Text fokussiert auf `convert_quarantine_entry` + `bulk_convert_quarantine_rows` und erwähnt commit_phase nicht. Die Plan-01-SUMMARY listete `commit_phase` als dritten Bridge-Punkt für Plan 05. Ich habe die Bridge bewusst NICHT entfernt: commit_phase ist der Legacy-Bulk-Migration-Pfad ohne Operator-Input — semantisch korrekt für historische Voll-Tag-Einträge (Migration ist privileged + automatisch, nicht UI-getrieben). Frontend (Plan 06) zeigt die UI für die zwei operator-getriebenen Pfade.
- **Roundtrip-Test-Anpassungen:** Plan-Text listete 2 neue rest-types-Tests. Ich habe zusätzlich 2 Bulk-spezifische Tests ergänzt (symmetrisch zum Single-Variant) und die 3 existing Tests (`convert_request_roundtrips`, `convert_request_with_manual_range_roundtrips`, `bulk_request_roundtrips_with_subset`) um das Pflichtfeld erweitert — ohne diese Anpassung hätten die existing Tests den compile gebrochen.

## Issues Encountered

- **Tippfehler `0x..._HALF_0001_u128`:** Ich hatte zuerst ein nicht-existierendes hex-Literal `HALF` verwendet. Korrigiert zu `0x..._BAFF_0001`. Test compile + grün.
- **`ManualRangeTO` ohne `PartialEq`:** Mein initialer `assert_eq!(parsed.manual_range, None)`-Vergleich type-checked nicht (`ManualRangeTO` derived kein `PartialEq`). Ersetzt durch `assert!(parsed.manual_range.is_none())`.
