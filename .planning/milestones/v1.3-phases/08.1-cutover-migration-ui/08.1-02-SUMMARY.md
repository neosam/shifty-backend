The `convert_quarantine_entry` body follows the locked 10-step flow:

1. `permission_service.check_permission(CUTOVER_ADMIN_PRIVILEGE, ...)` — short-circuits before any Tx work if the caller lacks the privilege.
2. `transaction_dao.use_transaction(tx)` — opens the cutover Tx.
3. `cutover_dao.find_legacy_extra_hours_not_yet_migrated` returns the full not-yet-migrated set; we look up the requested id by `id == extra_hours_id`. The DAO already filters `deleted IS NULL`, so a re-invocation for an already-soft-deleted row falls through to the `EntityNotFoundGeneric` branch — idempotent per RESEARCH P-02.
4. `employee_work_details_service.find_by_sales_person_id` for the sales person — replicates the migration-loop pre-fetch shape.
5. `detect_weekly_lump_sum(&row, all_legacy.as_ref(), |d| lookup_active_contract(...))` (file-private fns from Plan 08-09 — reachable in the same module per RESEARCH P-01). Mismatch → `ServiceError::ValidationError`.
6. `absence_dao.create(&entity, CUTOVER_MIGRATION_PROCESS, tx.clone())` — direct DAO insert mirroring `commit_phase` Z. 877-908. Bypasses `AbsenceService::create`'s forward-warning loop because cutover is privileged.
7. `cutover_dao.upsert_migration_source` with a synthetic `Uuid::new_v4()` `cutover_run_id` (RESEARCH P-02 — ad-hoc Convert is not part of any orchestrated cutover-run; the run_id is for audit-trail only).
8. `extra_hours_service.soft_delete_bulk(Arc::from([extra_hours_id]), CUTOVER_MIGRATION_PROCESS, Authentication::Full, Some(tx.clone()))`.
9. **Inline refreshed drift-report** (D-08, RESEARCH P-03 option a): replay `migrate_legacy_extra_hours_to_clusters` (rebuilds the bucket-map from the Tx-current state) then call `compute_gate_diagnostic` with the new bucket-map. 10. `transaction_dao.commit(tx)` — atomic flip.

### Refactor: `compute_gate` → 3 functions

Plan body locked the strategy as **extract-helper** (over call-through-with-flag): `compute_gate_inner` produces `(drift_rows, scope_set)` purely — no IO. `compute_gate` wraps that with `persist_gate_diff_report` (the audit JSON writer) and constructs `GateResult`. `compute_gate_diagnostic` ignores `scope_set` and returns a typed `CutoverGateDriftReport` directly.

The on-disk JSON behaviour for the existing `run`/`commit`/`gate-dry-run` paths is preserved bit-for-bit — the existing `gate_tolerance_pass_below_threshold` and `gate_tolerance_fail_above_threshold` integration tests still pass, including the file-content `assert!`s on `passed`, `total_drift_rows`, `sales_person_name`, and `category` JSON keys.

## Decisions Made

1. **Service-Tier-Klassifikation: Business-Logic auf existing CutoverServiceImpl** (D-03 + RESEARCH P-01). Ein Sub-Service hätte den `pub(crate)`-Bump für `detect_weekly_lump_sum` / `iso_week_range` / `lookup_active_contract` / `extra_hours_category_to_absence` erfordert. Direkte Erweiterung bleibt bei file-private Helpers — kein Visibility-Bump nötig.
2. **Refactor-Strategie: extract-helper (compute_gate_inner)** statt call-through-mit-flag oder Code-Duplication. Beide `compute_gate` und `compute_gate_diagnostic` teilen 100 % der drift-row-Berechnungslogik via `compute_gate_inner`; das einzige asymmetrische Stück ist die `persist_gate_diff_report`-Aufruf, der jetzt explizit pro Call-Site (run/commit) gemacht wird.
3. **Synthetic Uuid::new_v4() für `cutover_run_id`** in `upsert_migration_source` (RESEARCH P-02). Ad-hoc Convert ist Teil keines orchestrierten Cutover-Runs — der run_id existiert nur für den Audit-Trail. Alternative wäre eine separate Spalte `synthetic_run_origin` gewesen; das wurde explizit verworfen.
4. **Replay-Pfad statt empty-bucket-map für `refreshed_drift_report`** (Plan-`<action>` Z. 489-498). Das macht den inline drift-report fidelitätsgleich mit dem normalen `run`-Flow. Falls Profiling zeigt das ist zu teuer, kann eine spätere Phase das auf `compute_gate_diagnostic(empty_bucket_map)` umstellen — der Wire-Contract bleibt gleich, weil `refreshed_drift_report` `Option<CutoverGateDriftReportTO>` ist.
5. **Replay-Failure ist non-fatal**: wenn `migrate_legacy_extra_hours_to_clusters` oder `compute_gate_diagnostic` während Step 9 fehlschlagen, returnen wir `refreshed_drift_report: None` und `commit`en trotzdem. Der convert selbst ist fertig — den User durch ein Diagnose-Helper-Failure sein commit verlieren zu lassen wäre wesentlich schlechter UX.
6. **From-Impl-Bridge über service::absence::AbsencePeriod** (statt einen direkten `From<&AbsencePeriodEntity> for AbsencePeriodTO` neu in rest-types einzuführen). Der existing `From<&AbsencePeriodEntity> for service::absence::AbsencePeriod` aus `service/src/absence.rs:72` macht das mit einer kostenlosen Zwischen-Konvertierung möglich; rest-types bleibt frei vom dao-tier.

## Patterns Established

- **Extract-and-share pure helper before adding IO-free variant.** Wenn eine Service-Methode IO + Berechnung tut und der Plan eine "without IO"-Variante fordert, extrahiere die Berechnung in einen privaten Helper, halte den IO-Schritt als separates `pub(crate) fn` (free-standing), und lass die zwei Methoden den Helper teilen. Vermeidet Code-Duplication und hält die existing on-disk-Behaviour bit-for-bit identisch (testbar via existing snapshot/file-content-Assertions).
- **Mockall Sequence für sequenzielle DAO-Returns mit unterschiedlicher Antwort.** Wenn ein Mock vom Test-Subjekt mehrfach mit unterschiedlichen erwarteten Returns aufgerufen wird (z.B. "vor + nach soft_delete liefert die DAO unterschiedliche Werte"), nutze `let mut seq = Sequence::new(); .expect_X().times(1).in_sequence(&mut seq).returning(...)`. Funktioniert ohne extern Counter / `AtomicUsize`.
- **Synthetic-Uuid für Audit-Trail-fields ohne semantische Run-Zugehörigkeit.** `cutover_run_id: Uuid::new_v4()` für ad-hoc-Operationen ist ein additive non-breaking Erweiterung — die Spalte existiert bereits, das Schema erfordert NOT NULL, und für Audit-Logs ist eine fresh synthetic UUID semantisch valide ("dieser Migration-Source-Eintrag entstand außerhalb eines orchestrierten Runs").
