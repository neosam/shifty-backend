## Decisions Made

1. **Arc-newtype-wrappers (StateRef / DriftRowRef / EntryRef)** ŌĆö see key-decisions[0]. The `#[component]` macro auto-derives `PartialEq` on the props struct, but the wire DTOs (`CutoverWizardState` / `CutoverGateDriftRowTO` / `CutoverQuarantineEntryTO`) don't implement `PartialEq` (their nested fields carry float/arc/date payloads). Wrapping in `Arc<...>` with a hand-written `PartialEq` that compares by `Arc::ptr_eq` lets the `#[component]` derive succeed AND gives correct re-render-skip semantics (same allocation = no re-render needed). Mirrors `absences.rs` `WarningsList` pattern from Plan 08-05. Three wrappers were needed because three distinct types appear as component props.

2. **`try_consume_context::<Coroutine<CutoverAction>>()` instead of `use_coroutine_handle::<CutoverAction>()`** ŌĆö the latter PANICS in unit tests where no coroutine is registered. The former returns `Option<Coroutine<...>>` which is `None` in tests and `Some(...)` in production (Plan 08 registers the live coroutine in `app.rs` before the page is mounted). All `cutover_handle.send(...)` call-sites are wrapped in `if let Some(h) = cutover_handle.as_ref() { h.send(...) }`. Mirrors `extra_hours_modal.rs` L150-154 pattern.

3. **`commit_form_inner` as a free fn** ŌĆö see key-decisions[2]. Tests 6 + 7 want to render the gating logic directly with externally-controlled bools; manipulating use_signal state from inside `rebuild_in_place` tests is complex in dioxus 0.6. Splitting the production `CommitStage` into (a) signal-owning component + (b) free helper fn that takes the gating bools by value gives us both: production reactive UI + test-friendly entry point.

4. **Profile-Stage stat aggregation derived from buckets** ŌĆö see key-decisions[3]. The plan-body sketched `profile.total_legacy_rows` etc. but those fields don't exist on `CutoverProfileTO`. Instead, the wire DTO carries `buckets: Vec<CutoverProfileBucketTO>` where each bucket has `row_count` + `sum_hours` + `fractional_count` + `weekend_on_workday_only_count` + `iso_53_indicator`. We aggregate at render time:
   - `total_rows = sum(buckets.row_count)` ŌĆö the "rows to migrate" stat.
   - `affected_persons = distinct(buckets.sales_person_id)` ŌĆö the "betroffene Mitarbeiter" stat.
   - `quarantine_signal_buckets = count(buckets where fractional_count > 0 OR weekend_on_workday_only_count > 0)` ŌĆö the "quarantine entries" stat (counts buckets with quarantine signals, not the actual quarantined-row count which lives on the Drift report).
   - `carryover_signal_buckets = count(buckets where iso_53_indicator)` ŌĆö the "carryover diff" signal stat.
   This gives the operator a useful per-bucket signal-summary in Stage 1 without needing a separate Profile API enrichment.

5. **Custom backdrop+stop-propagation modal for EditExtraHoursModal** ŌĆö see key-decisions[4]. The project's `Dialog` component requires `title: ImStr` + `on_close: EventHandler<()>` + a `variant` + `width` and is geared for header+body+footer modals. The Edit form is a minimal 2-input form; the simpler custom modal pattern produces less HTML in the snapshot tests and avoids the Dialog's `use_media_query` hook (which doesn't fire in `rebuild_in_place` tests, leading to flakiness).

6. **Edit-modal `on_save` closes the modal but does NOT yet construct an `ExtraHoursTO` update** ŌĆö see key-decisions[5]. Constructing `ExtraHoursTO` requires `category` + `description` fields that are NOT carried in `CutoverQuarantineEntryTO`; the coroutine would need to load the row server-side first. This is intentionally deferred ŌĆö Plans 10..12 (diagnostics + UAT) will refine this wiring. Documented as a Known Stub below.
