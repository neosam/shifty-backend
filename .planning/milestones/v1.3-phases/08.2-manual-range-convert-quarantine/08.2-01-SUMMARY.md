# Phase 8.2 Plan 01: Manual-Range-Convert Backend Summary

**Backend `convert_quarantine_entry` accepts an optional `manual_range`-Branch (D-29) that skips `detect_weekly_lump_sum` completely and writes an `absence_period` with the operator-given range. Validation in Step 5 (year-match, `DateRange::new`, `find_overlapping`) protects against cross-year, inverted, and overlapping ranges. Steps 6-10 (DAO insert, audit row, soft-delete, inline `refreshed_drift_report`, atomic commit) are byte-identical to the 8.1-02 heuristic path — `synthetic_run_id` audit cohesion holds. Karin-Pattern (gap-1a from 08.1-10-SUMMARY) is now structurally resolvable: a fresh `manual_range_resolves_karin_quarantine` test demonstrates the same fixtures (mid-week contract change Mo-Fr 40h ending Wed → Mo-Fr 30h starting Thu) that the heuristic correctly rejects can be converted via Manual-Range. 4 unit tests + 1 REST integration test + 2 rest-types roundtrip tests cover the new surface; existing 8.1-02 convert tests + Karin diagnose test remain green (no regression).**

### Service Impl (`service_impl/src/cutover.rs`)

Step 5 ersetzt — die 13-LOC Heuristik-only Stelle (Z. 459-471) wird durch eine `match`-Branch-Form ersetzt (~70 LOC):

```rust
let (from_date, to_date) = match manual_range {
    Some(r) => {
        // D-30 #1: year-match (vor DateRange::new für präzise Fehler)
        let row_year = row.date_time.date().year();
        if r.start_date.year() != row_year || r.end_date.year() != row_year {
            return Err(ServiceError::ValidationError(...InvalidValue("calendar year")));
        }
        // D-30 #2: start <= end via DateRange::new → DateOrderWrong
        let range = DateRange::new(r.start_date, r.end_date)
            .map_err(|_| ServiceError::DateOrderWrong(r.start_date, r.end_date))?;
        // D-30 #3: kein Overlap mit existing (sales_person, category)
        let dao_category = extra_hours_category_to_absence(&row.category);
        let conflicts = self.absence_dao
            .find_overlapping(row.sales_person_id, dao_category, range, None, tx.clone())
            .await?;
        if !conflicts.is_empty() {
            return Err(ServiceError::ValidationError(...OverlappingPeriod(conflicts[0].logical_id)));
        }
        (r.start_date, r.end_date)
    }
    None => detect_weekly_lump_sum(...).ok_or_else(...)?, // byte-identisch zu 8.1-02
};
```

### REST Handler (`rest/src/cutover.rs`)

`cutover_convert_quarantine_entry_handler` erweitert um TO→Domain-Mapping VOR dem Service-Call:

```rust
let manual_range: Option<service::cutover::ManualRange> = req
    .manual_range
    .as_ref()
    .map(|to_range| {
        let fmt = time::macros::format_description!("[year]-[month]-[day]");
        let start = time::Date::parse(&to_range.start_date, fmt).map_err(|_| {
            ServiceError::ValidationError(...InvalidValue("manual_range.start_date is not a valid ISO-8601 date"))
        })?;
        let end = time::Date::parse(&to_range.end_date, fmt).map_err(|_| {
            ServiceError::ValidationError(...InvalidValue("manual_range.end_date is not a valid ISO-8601 date"))
        })?;
        Ok::<service::cutover::ManualRange, ServiceError>(service::cutover::ManualRange {
            start_date: start,
            end_date: end,
        })
    })
    .transpose()?;
let outcome = rest_state.cutover_service()
    .convert_quarantine_entry(req.extra_hours_id, manual_range, context.into(), None)
    .await?;
```

Parse-Errors werden als `ServiceError::ValidationError` (HTTP 422) gesurft — kein panic, Pitfall P-5 mitigiert.
