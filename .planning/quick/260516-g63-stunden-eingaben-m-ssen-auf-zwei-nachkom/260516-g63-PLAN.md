# Quick Task 260516-g63: Hour inputs with two decimal places

## Goal

User reports: hour values like `7.25` or `8.50` cannot be entered in the
frontend. Cause: HTML `<input type="number">` defaults to `step=1`, which
blocks fractional input on most browsers. Fix all hour-input form fields to
accept two decimal places (`step=0.01`).

Display formatting via `format_hours(_, 2)` is already correct in most pages;
this task only touches **input** controls, not display.

## must_haves

- All hour-amount input fields in shifty-dioxus carry `step="0.01"` HTML attribute
- `TextInput` component supports an optional `step` prop (passed through to the underlying `<input>`)
- No regression: existing tests pass; new unit/SSR tests cover the four affected inputs
- `cargo build` + `cargo test` green for both root workspace and shifty-dioxus

## Out of scope

- Display formatting (already `format_hours(_, 2)` in weekly_overview, billing_period_details etc. — `my_shifts.rs` uses `_, 1` but is read-only display, separate concern)
- Backend persistence (f32 already accepts decimals — DB schema unchanged)
- The integer inputs: `workdays_per_week`, `vacation_days`, `min_persons`, `expiration_hours` (these stay step=1)

## Scout: confirmed input locations

Already identified before planning. Four hour-input touchpoints:

| # | File | Line(s) | Current state | Fix |
|---|------|---------|---------------|-----|
| 1 | `shifty-dioxus/src/component/employee_work_details_form.rs` | 240 | `step: 1.0` on `FloatInput` for `expected_hours_per_week` | `step: 0.01` |

| 4 | `shifty-dioxus/src/component/add_extra_hours_form.rs` | 254-267 | Raw `<input>` with `"step": "0.001"` | `"step": "0.01"` |
