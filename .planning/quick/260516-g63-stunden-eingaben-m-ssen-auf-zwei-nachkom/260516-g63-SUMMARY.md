# Quick Task 260516-g63: Hour inputs with two decimal places — Summary

All four hour-input touchpoints in shifty-dioxus now carry `step="0.01"`, allowing
users to enter fractional hour values (e.g. 7.25, 8.50) in every browser.

## What changed

### Task 1 — TextInput `step` prop (`form/inputs.rs`)

Added `#[props(!optional, default = None)] pub step: Option<ImStr>` to `TextInputProps`.
The rendered `<input>` passes `step: step_attr` where `step_attr` is `Option<String>`;
Dioxus omits the attribute when the value is `None`, so all existing callers without
`step` are unaffected.

Two new tests added to the existing `#[cfg(test)] mod tests` block:
- `text_input_step_propagates_when_provided` — SSR assertion that `step="0.01"` appears
- `text_input_step_absent_when_none` — asserts the attribute is absent by default

### Task 2 — Wire `step="0.01"` through every hour-input

| File | Change |
|------|--------|
| `employee_work_details_form.rs` | `FloatInput { step: 1.0 }` → `step: 0.01` |
| `contract_modal.rs` | Added `step: Some(ImStr::from("0.01")),` to the `expected_hours` TextInput only |
| `extra_hours_modal.rs` | Added `step: Some(ImStr::from("0.01")),` to the `amount` TextInput |
| `add_extra_hours_form.rs` | Raw `<input>` attribute `"step": "0.001"` → `"step": "0.01"` |

Integer inputs (`workdays_per_week`, `vacation_days`) were left without `step` (default=1
is correct for integers, per plan constraint).

### Task 3 — SSR/regression tests for step attribute

Five new tests total across four files:

| Test | File | Approach |
|------|------|----------|
| `text_input_step_propagates_when_provided` | `form/inputs.rs` | Full SSR of `TextInput` |
| `text_input_step_absent_when_none` | `form/inputs.rs` | Full SSR of `TextInput` |
| `expected_hours_float_input_step_is_0_01` | `employee_work_details_form.rs` | SSR of `FloatInput` in isolation |
| `expected_hours_text_input_carries_step_0_01` | `contract_modal.rs` | Minimal `TextInput` SSR (see deviation note) |
| `amount_input_carries_step_0_01` | `extra_hours_modal.rs` | Minimal `TextInput` SSR (see deviation note) |

For `add_extra_hours_form.rs` (raw `<input>`, WASM-only JS dependencies):
- `amount_input_source_has_step_0_01` — `include_str!` source assertion
- `raw_input_step_attribute_renders_in_html` — atomic Dioxus `<input>` SSR assertion

## Deviations from plan

### Deviation 1: SSR tests use minimal TextInput renders, not full-modal renders

**Affected:** `contract_modal.rs`, `extra_hours_modal.rs`

**Reason:** `ContractModalBody` and `ExtraHoursModal` require `use_coroutine_handle` /
`try_consume_context` for `EmployeeWorkDetailsAction` / `EmployeeAction`, plus global
signal stores (`EMPLOYEE_WORK_DETAILS_STORE`, `CONFIG`, `I18N`). Wiring all those in
a test would take > 20 lines per test and require test infra that doesn't exist.

**Alternative applied:** Each file's test renders the minimal `TextInput { step: Some(ImStr::from("0.01")), input_type: "number" }` element via `dioxus_ssr::render` — this is identical to what the `ContractModalBody` / `ExtraHoursModal` RSX expands to for the affected field. This matches the plan's fallback clause: "acceptable to assert against a freshly-constructed `<TextInput step=… />` element".

### Deviation 2: `add_extra_hours_form.rs` gets two tests instead of one SSR test

**Reason:** `AddExtraHoursForm` uses `js::current_datetime()` which panics outside WASM
(there is a `#[cfg(not(target_arch = "wasm32"))]` fallback in `extra_hours_modal.rs` but
not in `add_extra_hours_form.rs`). Rather than risk a panic in test mode, two tests are
provided: a source-level `include_str!` assertion (guarantees the literal is in the file)
and an atomic Dioxus raw `<input>` SSR test (verifies Dioxus renders the attribute correctly).

### WASM build — pre-existing lld linker failure

`cargo build --target wasm32-unknown-unknown` fails with `error: linker 'lld' not found`
in this NixOS shell. This is pre-existing (not introduced by this task) and does not affect
code correctness; all logic-level compilation errors would surface via `cargo check` and
`cargo test`, both of which pass cleanly.
