# Quick Task 260516-gv8: Lösch-Button für Arbeitsverträge

## Goal

User reports: in production they need to delete employment contracts
(`employee_work_details`) but no UI affords it. Backend + a partial set of
frontend props already exist — the wire-up is incomplete:

- ✓ Backend `DELETE /employee-work-details/{id}` works (soft-delete via `deleted` column)
- ✓ Frontend API call `api::delete_employee_work_details(...)` exists at `shifty-dioxus/src/api.rs:924`
- ✓ Frontend service helper `delete_employee_work_details(uuid)` exists (private) at `shifty-dioxus/src/service/employee_work_details.rs:73`
- ✓ `EmployeeViewProps` carries `show_delete_employee_work_details: bool` + `on_delete_employee_work_details_clicked: Option<EventHandler<Uuid>>`
- ✗ **`EmployeeView` renders no actual button for delete-working-hours**
- ✗ **`employee_details.rs:177` handler triggers only `Update`, not a real delete**
- ✗ **No `EmployeeAction::DeleteEmployeeWorkDetails(Uuid)` variant in service coroutine**
- ✗ **No confirmation dialog**

This task wires the existing pieces together + adds a confirmation modal,
following the established `ShiftplanDeleteConfirm` pattern from
`shifty-dioxus/src/component/shiftplan_tab_bar.rs:50,201`.

## must_haves

- An admin viewing an employee's details (`employee_details.rs` page, NOT `my_employee_details.rs`) sees a "Delete contract" button next to each `employee_work_details` row
- Clicking the button opens a confirmation modal ("Wirklich löschen? Daten gehen verloren.")
- On confirm → backend DELETE call runs, view reloads → contract is gone (soft-deleted)
- On cancel → modal closes, no API call
- Regular employees (`my_employee_details.rs`) **do not** see the button (admin-only)
- New `EmployeeAction::DeleteEmployeeWorkDetails(Uuid)` variant in coroutine handler
- i18n keys for button + confirm title/body/btn in **all three locales** (de/en/cs)
- 2-3 SSR tests covering: (a) button renders when `show_delete_employee_work_details=true`, (b) button is absent when `false`, (c) handler is wired to the action
- `cargo build` + `cargo test` green (both root + shifty-dioxus)

## Out of scope

- Hard-delete (we keep soft-delete via `deleted` column — backend handles this already)
- Undo-delete UI (separate feature, deferred)
- Cascade-delete to dependent bookings (backend already handles via soft-delete + `deleted IS NULL` filters; no frontend concern)
- The `format_hours(_, 1)` display in `my_shifts.rs` (separate concern)

### Pattern B — Confirmation modal (from `shifty-dioxus/src/component/shiftplan_tab_bar.rs:50,201`)

```rust
let mut delete_confirm_id: Signal<Option<Uuid>> = use_signal(|| None);

// button click:

// elsewhere in rsx!:
if let Some(delete_id) = *delete_confirm_id.read() {
    Modal {
        title: i18n.t(Key::EmployeeWorkDetailsDeleteConfirmTitle).into(),
        body: i18n.t(Key::EmployeeWorkDetailsDeleteConfirmBody).into(),
        on_confirm: move |_| {
            employee_service.send(EmployeeAction::DeleteEmployeeWorkDetails(delete_id));

        on_cancel: move |_| delete_confirm_id.set(None),

}
```
