# Quick Task 260614-a26: Urlaubsanspruch als gerundete ganze Zahl

## Problem

Die `VacationEntitlementCard` / „Urlaubsanspruch"-Anzeige zeigt den
Vertragsanspruch mit einer Nachkommastelle (z. B. `27.4` statt `27`).

**Ursache:** `VacationBalanceServiceImpl::compute_balance`
(`service_impl/src/vacation_balance.rs`) summiert
`vacation_days_for_year(year)` über alle aktiven Verträge **ohne Rundung**.
Bei unterjährigem Vertragsstart/-ende ist dieser Wert anteilig (aliquot) und
damit fraktional.

Der parallele Pfad in `service_impl/src/reporting.rs:602` rundet die identische
Summe bereits mit `.round()` — der Vacation-Balance-Pfad blieb inkonsistent
ungerundet.

## Lösung

`entitled_days` in `compute_balance` analog zum Reporting-Pfad mit `.round()`
auf die Summe runden. `remaining_days` (= `entitled + carryover − used − planned`)
profitiert automatisch.

## Tasks

### Task 1 — Rundung von entitled_days
- **files:** `service_impl/src/vacation_balance.rs`
- **action:** `.sum::<f32>().round()` auf die `entitled_days`-Aggregation
  anwenden (Muster aus `reporting.rs:598-602`).
- **verify:** `cargo build -p service_impl`
- **done:** `entitled_days` ist immer ganzzahlig.

### Task 2 — Regression-Test
- **files:** `service_impl/src/test/vacation_balance.rs`
- **action:** Test mit unterjährigem Vertrag (Start KW27/2026, 30 Urlaubstage)
  → roher Anspruch ≈ 15.21, erwartet exakt `15.0`. Assert: Wert ist
  ganzzahlig und == 15.0.
- **verify:** `cargo test -p service_impl vacation_balance`
- **done:** Test schlägt ohne `.round()` fehl, mit `.round()` grün.
