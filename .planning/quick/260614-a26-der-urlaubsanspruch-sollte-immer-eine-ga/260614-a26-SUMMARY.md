# Quick Task 260614-a26 — Summary

## Was geändert wurde

Der „Urlaubsanspruch" (Vertragsanspruch, `entitled_days`) wird jetzt im
Vacation-Balance-Pfad auf eine gerundete ganze Zahl gerundet — vorher war er
bei unterjährigem Vertragsstart/-ende anteilig fraktional (Nachkommastelle).

### Diagnose

Zwei Pfade berechnen den Anspruch aus derselben Quelle
(`EmployeeWorkDetails::vacation_days_for_year`, anteilig/aliquot als `f32`):

- `service_impl/src/reporting.rs:602` → rundete bereits (`.sum::<f32>().round()`).
- `service_impl/src/vacation_balance.rs` (hinter `VacationEntitlementCard` /
  „Urlaubsanspruch"-Anzeige) → rundete **nicht** → seltsame Nachkommastelle.

### Fix

`service_impl/src/vacation_balance.rs` — `compute_balance`: `.sum::<f32>().round()`
auf die `entitled_days`-Aggregation, konsistent mit dem Reporting-Pfad.
`remaining_days` (= entitled + carryover − used − planned) profitiert automatisch.

### Test

`service_impl/src/test/vacation_balance.rs` — neuer Test
`get_rounds_aliquot_entitlement_to_whole_number`: Vertrag startet KW27/2026
(2026-06-29), 30 Urlaubstage → roher Anspruch ≈ 15.21 → erwartet exakt `15.0`.
Schlägt ohne `.round()` fehl, mit `.round()` grün.

## Verifikation

- `cargo build --workspace` → OK
- `cargo test -p service_impl` → 410 passed, 0 failed
- Bestehende Ganzjahres-Tests (25.0/24.0/20.0/0.0) unverändert grün.

## Scope-Hinweise

- `used_days`/`planned_days` bleiben bewusst fraktional (legitime Halbtage).
  Nur der Anspruch (`entitled_days`) wird gerundet, wie vom User gefordert.
- Kein Snapshot-Schema-Bump nötig: der Reporting-/Billing-Pfad rundete bereits;
  nur der separate Vacation-Balance-Read-Aggregat-Pfad wurde angeglichen.
